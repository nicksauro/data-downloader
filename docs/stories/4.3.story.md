# Story 4.3 — Public API estável V1.0 release

## Contexto

Story criada por mini-council Aria + Sol + Felix em **COUNCIL-13** (Epic 4
prep). Após Story 4.1 (multi-symbol broker) e Story 4.2 (multi-asset
support), a superfície funcional do `data_downloader.public_api` está
**completa para V1**: `download`, `read`, `read_continuous`,
`vigent_contract`, `DownloadHandle/Progress/Result`, hierarquia de exceções.

Hoje `__api_version__ = "0.3.0"` (Story 1.7b — minor aditivo). O squad
precisa fechar V1.0.0 antes do release final (Story 4.4) porque:

- **Auto-updater (Story 4.4)** assume API estável — usuários atualizam
  desktop sem pensar em breaking changes da lib.
- **Backtest engine** (próximo projeto do squad) é o primeiro consumidor
  externo. Se importa `data_downloader.public_api.download(...)`, esse
  contrato precisa ser SemVer-governed a partir do release.
- **Constitution Article IV (No Invention)** + ADR-007a (DownloadHandle)
  já estabelecem rigor de fronteira; falta apenas ritual de release.

Esta story é leve em código (1.5d) e pesada em **disciplina de
documentação + ritual de versionamento**. Não introduz novas funções —
documenta, anota tipos, escreve política, e bumpa a versão.

## User Story

**As a** Aria (custodian da fronteira pública) preparando o data-downloader
para servir como library para backtest engine, signal generator e risk
monitor (próximos projetos do squad)

**I want** o `data_downloader.public_api` declarado V1.0.0 com docstrings
completos, política de deprecação documentada, exemplos de consumo, e
regression tests de SemVer (v0.3.0 → v1.0.0 sem quebrar nenhum import
existente)

**So that** projetos downstream possam pin `data-downloader>=1.0,<2.0` com
confiança SemVer estrita, e qualquer mudança breaking futura exija major
bump explícito + deprecation cycle (não acidente).

## Acceptance Criteria

1. **AC1 — Docstrings completos em toda função/classe pública.**
   - `download`, `read`, `read_continuous`, `vigent_contract` têm
     docstring no formato Google style com seções: descrição,
     `Args`, `Returns`, `Raises`, `Examples`.
   - `DownloadHandle`, `DownloadProgress`, `DownloadResult`,
     `DownloadStatus` têm docstring no nível classe + cada método/property
     público.
   - Hierarquia de exceções (`DataDownloaderError`, `DownloadError`,
     `IntegrityError`, `InvalidContract`, `DiskFull`, `DLLInitError`)
     documenta quando é levantada.
   - Linter docstring (interrogate ou pydocstyle) configurado em
     `pyproject.toml`; CI falha se cobertura docstring `public_api/` < 100%.

2. **AC2 — Type stubs / inline annotations completas.**
   - Toda assinatura em `public_api/` tem type hints (validar com
     `mypy --strict src/data_downloader/public_api/`).
   - Aria decide entre (a) `.pyi` separados ou (b) inline annotations
     com `py.typed` marker — recomendação preliminar é (b) por simplicidade,
     mas decisão final no comeco da execução.
   - `py.typed` marker file em `src/data_downloader/` se decisão = (b).

3. **AC3 — `docs/public_api/USAGE.md` com exemplos de consumidor.**
   Documento criado com 3 exemplos completos copy-paste:
   - **Backtest:** loop por dia, `read("WDOJ26", start, end)` → DataFrame
     → strategy. Mostra import, error handling, schema esperado.
   - **Signal generator:** `read_continuous("WIN", start, end)`,
     processamento incremental, computação de indicador.
   - **Risk monitor:** `download` agendado + `vigent_contract` para
     resolver rollover, exposição em real-time.
   Cada exemplo é testado via `doctest` (AC7) ou `pytest --markdown`.

4. **AC4 — Política de deprecação documentada.**
   `docs/public_api/DEPRECATION_POLICY.md` cria com:
   - Decorador `@deprecated(since="X.Y.Z", removed_in="A.B.C", reason="...")`
     implementado em `data_downloader._internal/deprecation.py` (interno —
     não exportado), emitindo `DeprecationWarning` no primeiro call.
   - Regra: deprecação anunciada em release N → removida no mais cedo em
     release N+major+1 (mínimo 6 meses entre anúncio e remoção).
   - Changelog seção "Deprecated" lista todas deprecadas em vigência.
   - Exemplos: como deprecar arg, como deprecar função inteira, como
     deprecar classe.

5. **AC5 — `__api_version__` bump 0.3.0 → 1.0.0.**
   Diff exatamente uma linha em `src/data_downloader/public_api/__init__.py`:
   ```
   - __api_version__ = "0.3.0"
   + __api_version__ = "1.0.0"
   ```
   `__all__` revisado para garantir que tudo listado é doc'd e estável.

6. **AC6 — `CHANGELOG.md` seção "API v1.0.0".**
   Seção criada listando explicitamente:
   - Garantias V1: SemVer estrito, política de deprecação, ABI estável
     para imports listados em `__all__`.
   - O que muda vs 0.3.0: nada quebra; apenas formalização.
   - O que NÃO é coberto por SemVer: módulos `_internal/`, módulos
     fora de `public_api/`, comportamento exato de exceções não
     documentadas, performance (governada por bench BASELINES, não SemVer).
   - Roadmap V1.x (minor): novas funções aditivas (ex: `download_batch`
     se decidido — pode ser Story futura).
   - Roadmap V2.0 (major): nada planejado — declarado intencionalmente.

7. **AC7 — Regression tests de SemVer backwards-compat.**
   `tests/api/test_semver_v1_backwards_compat.py`:
   - Para cada símbolo em `__all__` da v0.3.0 (lista hardcoded), valida
     que ainda existe em v1.0.0 com mesma assinatura (inspect
     comparativo).
   - Para cada exemplo em `USAGE.md`, executa via `doctest` ou
     `pytest --doctest-modules` e verifica que roda sem erro
     (mockado quando precisa DLL).
   - Test `test_api_version_bumped.py` simples: `assert
     data_downloader.public_api.__api_version__ == "1.0.0"`.
   - Test `test_no_undocumented_exports`: para cada símbolo em
     `__all__`, verifica docstring não vazio.

  versionamento. Zero linha de código muda em `download`, `read`, etc.

  Story 4.4 que distribui o `.exe` com `__api_version__=1.0.0` embutido.
