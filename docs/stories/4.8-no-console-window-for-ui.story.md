# Story 4.8 — UI launch sem console window

## Contexto

**Bug release-blocker descoberto por Pichau** v1.0.2: ao dar
double-click no `data_downloader.exe`, **uma janela preta de console
abre junto com a UI** (e fecha quando o app fecha). Isso quebra a
percepção de "produto desktop polido" e parece com um malware/script
amador para usuário leigo.

### Causa raiz (build/data_downloader.spec:178)

```python
console=True,  # CLI-first: stdout disponível para `data_downloader.exe download ...`
# Trade-off: console window pisca brevemente quando user clica para abrir UI;
# aceitável para v1.0.1, refinar em v1.1 com .exe duplo (console + windowed).
```

Felix introduziu `console=True` em v1.0.1 para que `data_downloader.exe
download --symbol ...` (CLI promised em INSTALL.md) tivesse stdout. O
trade-off de "pisca brevemente" virou "console permanece aberta toda
sessão" porque PyInstaller --onedir + console=True faz a janela ficar
viva enquanto o processo Python existe (e a UI Qt é o processo).

## User Story

**As a** usuário final do data-downloader.exe (não-dev) que double-clica
o ícone para abrir o produto

**I want** apenas a janela da UI Qt aparecer — sem console preto, sem
flash, sem "isso parece um vírus"

**So that** o produto pareça profissional e eu mantenha confiança em
clicar "Sim" no SmartScreen warning na primeira abertura.

## Acceptance Criteria

1. **AC1 — Dois `.exe` no zip distribuído.**
   - `data_downloader.exe` — `console=False` (windowed). Default
     para double-click. Ícone do Qt; nenhum console aparece.
   - `data_downloader-cli.exe` — `console=True` (console). Para
     uso terminal: `data_downloader-cli.exe download --symbol WDOFUT`.
   - Ambos compartilham mesmo `_internal/` (build PyInstaller --onedir
     com 2 EXE() targets na spec).

2. **AC2 — INSTALL.md atualizado.**
   - Seção "Como abrir":
     - "Double-click `data_downloader.exe` → UI abre."
     - "Para usar via terminal/PowerShell, rode `data_downloader-cli.exe
       download --symbol WDOFUT --period 30d`."
   - Removido qualquer mention de "console pisca brevemente é normal".

3. **AC3 — Atalho do Menu Iniciar (V2 polish — opcional V1.0.3).**
   - Se Story 4.X "installer NSIS" estiver Done, atalho aponta para
     `data_downloader.exe` (windowed). [Dependência futura.]

4. **AC4 — `data_downloader.exe` (windowed) ainda dispatcha CLI
   quando recebe args.**
   - Given `data_downloader.exe download --symbol WDOFUT` em
     PowerShell;
   - When user roda;
   - Then funciona MAS sem console output capturável (output vai
     para `~/.data_downloader/logs/cli.log`).
   - Recomendação na microcopy: "Para output em terminal, use
     `data_downloader-cli.exe`".

5. **AC5 — Smoke local Pichau (Windows 10).**
   - Build local com `python scripts/build_release.py`.
   - Double-click `dist/data_downloader/data_downloader.exe`.
   - **Apenas janela Qt aparece.** Zero console.
   - `dist/data_downloader/data_downloader-cli.exe download --help`
     em PowerShell mostra Typer help.

6. **AC6 — Manifest/zip atualizados.**
   - `build-manifest-v1.0.3.json` lista os 2 .exe com SHA256.
   - `data-downloader-v1.0.3-win64.zip` contém ambos.

## Out of Scope

- Code signing (Story 4.4-followup mantém WAIVER ativo até EV cert).
- Tray icon / minimize-to-tray (V1.1 polish, opcional).

## Risks

| Risco | Mitigação |
|-------|-----------|
| 2 exe = 2x storage | --onedir compartilha `_internal/` — overhead ~200KB stub adicional. |
| User confunde qual abrir | INSTALL.md + microcopy claros; atalho default = windowed. |
| Scripts existentes que invocam `data_downloader.exe download ...` em CI quebram | AC4 mantém compat — windowed dispatcha CLI; só perde stdout interativo. |
