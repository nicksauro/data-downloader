# Story 2.8 — Storage perf tuning

## User Story

**As a** Sol (storage authority) responsável por longevidade e leitura performante do dataset Parquet/SQLite + Pyro (perf authority) responsável por baselines absolutos
**I want** finalizar a **matriz de tuning** de storage: (a) `row_group_size` ótimo para queries DuckDB típicas (filtro por timestamp + projeção subset de colunas); (b) escolha **final** entre Snappy / ZSTD-1 / ZSTD-3 com base em pareto throughput-write × tamanho-arquivo × throughput-read; (c) ajuste das `PRAGMAs` SQLite (`mmap_size`, `cache_size`) para perfil de máquina modesto vs alto-end (finding **M6** Plan Review)
**So that** queries DuckDB típicas (`SELECT side, price FROM parquet WHERE ts BETWEEN ... AND ...`) tenham latência mínima, tamanho em disco fique no melhor pareto-ótimo, e SQLite catalog não estoure RAM em laptops modestos (16GB) — fechando os findings **H5** (Snappy escolhido sem matriz) + **M6** (PRAGMAs estouram RAM) + **H6** (append+dedup re-escreve arquivo a cada chunk = O(n²)) do Plan Review original.

**NOTA:** Esta story é **complementar** a Story 2.2 (que vectoriza writer). 2.2 fecha gap de **escrita**. 2.8 fecha gap de **layout** (row_group, compression) e **leitura** (cache PRAGMAs). Sem 2.8, dataset cresce gigante e queries DuckDB ficam IO-bound. Workload representativo: trades de WDOJ26 5 dias × 1M trades/dia = 5M trades, query típica filtra 1d × 1M trades.

## Acceptance Criteria

1. **AC1 — Matriz compression bench documentada:** Existe `docs/perf/COMPRESSION_MATRIX_2026-MM-DD.md` com tabela:
   - Linhas: Snappy, ZSTD-1, ZSTD-3, ZSTD-5 (LZ4 e GZIP avaliados se cabe).
   - Colunas: write_throughput (trades/s), file_size (MB), read_throughput (trades/s scan full), cpu_decode (% in-query), pareto_dominated (bool).
   - Workload: 5M trades sintéticos representativos (gerados via mock fixture Story 1.4.5 strategy).
   - Recomendação **final** com justificativa (ex: ZSTD-1 pareto-dominante; muda default Sol em `parquet_writer.py`).

2. **AC2 — `row_group_size` tuning:** Testar pelo menos 4 valores: 50_000, 100_000, 200_000, 500_000 (configurações ParquetWriter). Para cada, medir:
   - Latência query típica `SELECT * WHERE ts BETWEEN T1 AND T2` (filtro hit ~1% rows).
   - Latência query agregada `SELECT exchange, COUNT(*) GROUP BY exchange` (full scan).
   - Memory peak DuckDB durante query (proxy para readability eficiente).
   Recomendação default registrada em `docs/storage/SCHEMA.md` §"Layout Parquet" (Sol authority — atualizar doc canônico).

3. **AC3 — SQLite PRAGMAs perfil-adaptativo:** Atual hardcoded `mmap_size=256MB + cache_size=200MB` causa OOM em laptops 16GB com outros apps abertos (M6). Implementar 3 perfis:
   - `low` — mmap_size=64MB, cache_size=50MB (laptop/dev).
   - `medium` (default) — mmap_size=128MB, cache_size=100MB.
   - `high` — mmap_size=512MB, cache_size=400MB (workstation Pyro).
   Selecionável via env var `DATA_DOWNLOADER_SQLITE_PROFILE=low|medium|high`. Detection automática heurística no boot (RAM total / 32 → bucket). Override manual sempre vence.

4. **AC4 — Threshold rewrite (finding H6) re-medido:** Story 1.4 implementou threshold (rewrite vs new-file) baseado em estimativa. Pyro mede empiricamente:
   - Para diferentes tamanhos de partição (10MB, 50MB, 100MB, 500MB, 1GB): qual threshold de novos trades minimiza latência total?
   - Resultado: tabela de threshold otimizado por bucket de partition_size em `docs/perf/REWRITE_THRESHOLD_2026-MM-DD.md`.
   - Aplicar default novo em `parquet_writer.py` (Sol authority).

5. **AC5 — Bench updated em BASELINES.md:** Pyro re-roda `bench_parquet_write` + `bench_parquet_read` + `bench_sqlite_query` com configs novas (compression, row_group, PRAGMAs). Atualiza `docs/perf/BASELINES.md` v1.1.0-mock (ou v1.0.0-real se followup já PASS) com:
   - Improvement absoluto vs config antiga.
   - Pareto-dominance justificada.
   - Regression budget mantido.

6. **AC6 — Tests + property:**
   - **Unit:** Cada profile PRAGMA aplicado corretamente (test inicializa Catalog com profile=X, verifica via `PRAGMA mmap_size`).
   - **Integration:** ParquetWriter com row_group_size=N gera arquivo com N rows/group (validado via `pq.read_metadata`).
   - **Property test (Hypothesis):** `read(write(table, compression=X)) == table` para X em [snappy, zstd-1, zstd-3] (round-trip preservation).
   - **Bench:** `bench_compression_matrix` (NEW), parametrized.

7. **AC7 — Sol audit + Pyro sign-off:** Como esta story muda defaults Sol (compression, row_group, PRAGMAs, threshold rewrite), Sol audit completo obrigatório (`docs/qa/AUDIT_REPORTS/2.8-storage-{date}.md`) — sem APPROVED, story NÃO vai a InReview. Pyro sign-off no bench results (mismatch entre Sol decision e Pyro number → mini-council).

8. **AC8 — Documentação canônica atualizada:** `docs/storage/SCHEMA.md` §"Layout Parquet" + `docs/storage/QUERIES.md` §"Performance" atualizados com defaults novos. README atualizado com nota sobre profile selection. Migration NÃO necessária (mudança aditiva — arquivos antigos com Snappy continuam legíveis; novos arquivos usam novo default).

### Constraints CRÍTICAS
- **R4 (schema contrato perpétuo):** mudança de compression é AC8-aditivo — arquivos antigos legíveis sem migração. Mudança de row_group_size só afeta arquivos novos. Mudança de PRAGMA é runtime-only (não toca schema).
- **R5 (idempotência):** dedup property test continua válida (compression não muda dedup logic).
- **Round-trip preservation:** `read(write(table)) == table` para qualquer compression suportada (property test obrigatório).
- **Pareto formal:** Sol NÃO escolhe default por gosto. Decisão registrada em `COMPRESSION_MATRIX_*.md` com 4+ candidatos medidos.
- **Reproducibilidade bench:** seed fixo + workload determinístico (Story 1.4.5 strategy reusada).
