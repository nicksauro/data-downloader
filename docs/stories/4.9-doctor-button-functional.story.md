# Story 4.9 — Botão Diagnóstico Completo funcional

## Contexto

**Bug descoberto por Pichau** v1.0.2: na SettingsScreen, botão
"Diagnóstico Completo (doctor)" no bottom-bar **NÃO faz nada**.

### Causa raiz (settings_screen.py:163-165)

```python
self._doctor_btn = QPushButton(format_msg("BTN_DOCTOR_FULL"), self)
self._doctor_btn.setObjectName("doctorBtn")
bottom_bar.addWidget(self._doctor_btn)
# ⚠️ ZERO .clicked.connect(...) — botão é puramente visual.
```

Adicionalmente, **6 menções a `data-downloader doctor` na microcopy**
(microcopy_loader.py linhas 67, 115, 145, 193, 247, 257, 271) sugerem
ao usuário rodar esse comando — mas `doctor` **NÃO existe** em
`src/data_downloader/cli.py` nem em qualquer subcomando Typer
(`grep -rn "def doctor" src/` retorna ZERO matches; o subcommand
existe na lista `_CLI_SUBCOMMANDS` em `ui/app.py:106` mas nunca foi
implementado no CLI Typer).

Owners Council B5 (2026-05-04) já levantou esse débito; esta story
fecha definitivamente.

## User Story

**As a** usuário com problema (DLL não conecta, partição corrupta,
volume baixo)

**I want** clicar "Diagnóstico Completo" e ver um relatório claro
(modal ou painel) com checks rodados e sugestões de fix

**So that** consigo auto-resolver problemas comuns sem abrir issue
no GitHub.

## Acceptance Criteria

1. **AC1 — Implementar `data-downloader doctor` (Typer subcommand).**
   - Em `src/data_downloader/cli.py`, adicionar `@app.command()
     def doctor(reconcile: bool = False) -> int:`.
   - Checks executados:
     a. **DLL load** — `from data_downloader.dll.session import open_session`
        + tenta abrir + reporta versão ou erro técnico.
     b. **Companion files** — invoca `verify-dll-companions.py`
        bundled (já existe em scripts/).
     c. **Catalog integrity** — `from data_downloader.storage.catalog
        import validate_schema` (se existir) + count partições.
     d. **Disk free space** — `shutil.disk_usage(data_dir)` +
        warn se <5GB.
     e. **Env vars** — verifica PROFITDLL_KEY/USER/PASS/PATH presentes.
   - Output JSON em stdout (`--format=json`) ou tabela human (default).
   - Exit code: 0 se todos PASS, 1 se algum WARN, 2 se algum FAIL.

2. **AC2 — Botão "Diagnóstico Completo" abre modal com resultado.**
   - `_on_doctor_clicked` (handler novo) executa `doctor()` em
     QThread (bloqueia 5-15s; UI continua responsiva).
   - Modal `QDialog` com `QTextEdit` read-only mostra resultado
     formatado (ícones ✓ / ⚠ / ✗ + microcopy por check).
   - Botão "Copiar para clipboard" e "Fechar".
   - Se `--reconcile` opcional, oferece checkbox "Reconciliar
     catálogo após diagnóstico".

3. **AC3 — Microcopy review.**
   - Todas as 6 menções a `data-downloader doctor` em
     `microcopy_loader.py` continuam válidas (apontam para comando
     real agora).
   - Adicionar `BTN_DOCTOR_COPY_REPORT`, `LBL_DOCTOR_RUNNING`,
     `LBL_DOCTOR_RESULT_PASS/WARN/FAIL`.

4. **AC4 — Pytest cobre o comando CLI.**
   - `tests/unit/test_cli_doctor.py` mocka cada check, valida
     exit codes 0/1/2 + output JSON shape.

5. **AC5 — Smoke local Pichau.**
   - Pichau abre Settings → clica "Diagnóstico Completo".
   - Modal aparece com 5 checks listed; cada um com status
     real (DLL configurada → ✓; sem catálogo → ⚠).

6. **AC6 — `_CLI_SUBCOMMANDS` em `ui/app.py` mantém `"doctor"` como
   válido (já está) — agora com handler real correspondente.**

## Out of Scope

- Auto-fix automático (V1.1 polish — Story 4.X-followup).
- Diagnóstico remoto/upload (privacy concern — V2).
- Histórico de diagnósticos (catálogo `~/.data-downloader/doctor.log`)
  — V1.1.

## Risks

| Risco | Mitigação |
|-------|-----------|
| Doctor demora >15s e parece travado | Spinner + timeout por check (5s cada). |
| Check de DLL trava porque sem credenciais | Detecta env-vars antes de tentar load; pula gracefully com WARN "credenciais ausentes". |
| Modal vira blob de texto ilegível | Microcopy estruturado (1 linha por check + sub-detalhe indented). |
