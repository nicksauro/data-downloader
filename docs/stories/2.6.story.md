# Story 2.6 — Retry inteligente + circuit breaker

## User Story

**As a** orchestrator (Story 1.7a) que precisa atravessar quirks DLL (Q02-E 99% reconnect, Q11-99, NL_* códigos diversos, timeouts esporádicos)
**I want** uma política formal de **retry categorizado por tipo de erro NL_*** (transient vs permanent vs ambiguous) acoplada a **circuit breaker** que aborta loops infinitos quando timeouts repetem, integrada ao `with_retry` existente em `orchestrator/retry.py` (Story 1.7a) sem alterar contrato externo
**So that** download longo (30+ dias) sobrevive a falhas transitórias sem mascarar bugs lógicos (R7 — falha rápido em erro lógico) nem cascatear loops infinitos quando DLL fica em estado degradado, fechando o gap operacional identificado em Q02-E (99% reconnect indefinido) e dando observabilidade real (counter `dll_drops_total{reason}` AC3 Story 2.4) à pressão de retries.

**NOTA:** Esta story finaliza o item "Retry inteligente" do EPIC-2 escopo IN. Constrói sobre o `with_retry` mínimo de 1.7a (max_attempts=3 fixo, sem categorização) — agora extende com taxonomia NL_* + circuit breaker stateful por símbolo. Desbloqueia close de Quirk Q02-E na QUIRKS.md (workaround formalizado em policy).

## Acceptance Criteria

1. **AC1 — Taxonomia NL_* documentada em código:** Existe `src/data_downloader/dll/error_taxonomy.py` exportando enum `ErrorCategory` (`TRANSIENT | PERMANENT | AMBIGUOUS | UNKNOWN`) + função `classify(nl_code: int, context: str) -> ErrorCategory`. Mínimo 12 códigos NL_* mapeados (NL_OK=0, NL_INTERNAL_ERROR=80, NL_NOT_INITIALIZED, NL_INVALID_ARGS, NL_WAITING_SERVER, NL_NO_LOGIN, NL_NO_LICENSE, NL_OUT_OF_RANGE, NL_LICENSE_NOT_ALLOWED, NL_TOO_MANY_TRADES, NL_DISCONNECT, NL_LOST_AUTH). Cada classificação justificada via comment + referência ao manual ProfitDLL §X.Y ou QUIRKS.md.

2. **AC2 — Retry policy por categoria:** `orchestrator/retry.py` ganha parâmetro `policy: RetryPolicy` (dataclass) que define `max_attempts` + `base_delay` + `backoff` + `jitter` + `categories_to_retry: set[ErrorCategory]`. Default policy: TRANSIENT retry (max=5, base=30s, exp backoff até cap 600s, jitter ±20%); AMBIGUOUS retry com cap menor (max=3, base=60s); PERMANENT/UNKNOWN raise imediato (fail fast — R7). Backwards compat: se chamador não passa policy, usa default policy NÃO o comportamento antigo (mudança documentada).

3. **AC3 — Circuit breaker stateful por símbolo:** `orchestrator/circuit_breaker.py` implementa `CircuitBreaker(symbol)` com 3 estados: `CLOSED` (normal), `HALF_OPEN` (testando recovery), `OPEN` (bloqueando). Trigger OPEN: ≥ N retries categoria TRANSIENT em janela W (default N=10, W=300s). Em OPEN, próximo `request()` raise `CircuitOpenError` sem invocar DLL. Após cooldown (default 600s), transita HALF_OPEN; primeira call testa; sucesso → CLOSED, falha → OPEN novamente com janela ampliada (×2). Métrica `circuit_breaker_state{symbol}` (gauge 0=CLOSED, 1=HALF_OPEN, 2=OPEN) emitida via `ProgressEmitter` Protocol (Story 2.4 hook).

4. **AC4 — Integração com Q02-E (99% reconnect):** Workaround `Q02-E` em `QUIRKS.md` é formalizado como `RetryPolicy.q02e_progress_aware` — circuit breaker **NÃO conta** "progress=99%" como evento de retry; apenas conta NL_* error codes reais. Timeout duro 1800s sem progresso real continua sendo SLA de Story 1.3, mas agora circuit breaker registra essa transição como `forced_open_q02e` (label da métrica). Test reproduz cenário sintético (mock DLL emite progress=99% por 100 callbacks, depois 100% + último trade) e confirma circuit breaker permanece CLOSED.

5. **AC5 — Integração orchestrator não-quebradora:** `DownloadOrchestrator` (Story 1.7a) instancia 1 `CircuitBreaker` por `(symbol, exchange)` em `__init__` ou lazy. `with_retry` é chamado dentro do circuit breaker context. Mudanças cirúrgicas em `orchestrator.py` (sem refactor amplo). Sinal externo: comportamento de `run()` para chamadores existentes (Story 1.7b CLI, public_api download_handle) é idêntico em golden path; em erros transientes acumulados, raise `CircuitOpenError` (subclasse `OrchestratorError` per ADR-011) em vez de loop infinito.

6. **AC6 — Logging estruturado dos eventos:** Eventos emitidos via structlog (ADR-010): `retry.attempt` (existente), `retry.exhausted` (existente), `retry.skipped_permanent` (NEW, fail fast), `circuit_breaker.transition` (NEW, fields: symbol, from, to, reason), `circuit_breaker.rejected` (NEW, request bloqueado em OPEN). Todos com `correlation_id` herdado do contextvar (ADR-010). Hot path RULE R21 respeitada: log per-chunk OK, NÃO per-trade.

7. **AC7 — Suite de testes:**
   - **Unit:** `error_taxonomy.classify` para cada NL_* mapeado (table-driven test com 12+ casos).
   - **Unit:** `RetryPolicy.default` retry counts conferem cap (TRANSIENT max=5, AMBIGUOUS max=3, PERMANENT max=0 fail fast).
   - **Unit:** `CircuitBreaker` state transitions (CLOSED→OPEN após N falhas; OPEN→HALF_OPEN após cooldown; HALF_OPEN→CLOSED após sucesso; HALF_OPEN→OPEN após nova falha com janela ampliada).
   - **Property test (Hypothesis):** sequência aleatória de TRANSIENT/PERMANENT errors → `OPEN` é alcançado se contagem TRANSIENT em janela ≥ N; nunca alcançado de outra forma.
   - **Integration:** orchestrator com mock DLL que emite NL_DISCONNECT em 50% das chamadas — download completa ou falha CircuitOpenError dentro de SLA, nunca loop infinito.
   - **Smoke (gated):** opcional, deferred para 1.7b-followup ambiente real (sem AC bloqueante neste smoke).

8. **AC8 — Configuração via env + defaults documentados:** Override via env vars: `DATA_DOWNLOADER_RETRY_MAX_ATTEMPTS_TRANSIENT`, `DATA_DOWNLOADER_CB_THRESHOLD`, `DATA_DOWNLOADER_CB_WINDOW_S`, `DATA_DOWNLOADER_CB_COOLDOWN_S`. Defaults registrados em `docs/dev/RETRY_POLICY.md` (NEW) com tabela: categoria → max_attempts → base_delay → cap. Doc inclui troubleshooting (como saber se circuit está OPEN, como forçar reset em dev).

### Constraints CRÍTICAS
- **R7 — Fail fast em erro lógico:** PERMANENT category NÃO retry. Categoria UNKNOWN também — explícito por design (alucinar retry em código desconhecido mascara bug).
- **R21 — Hot path:** logging per-trade proibido. Eventos circuit/retry são per-call (não per-trade) → OK.
- **ADR-005 thread model:** CircuitBreaker thread-safe (lock interno); nenhum estado compartilhado entre `OrchestratorThread` e `IngestorThread`.
- **Idempotência (R5):** retry NÃO causa duplicate write — dedup é responsabilidade de Story 1.4 (já garantido).
- **Sem mudança de fronteira `public_api/`:** novo erro `CircuitOpenError` é tipo público (em hierarquia ADR-011), mas semântica de `download(...)` continua "raise se erro fatal".
- **Sem nova dependência:** circuit breaker é código nosso (deque + threading.Lock); NÃO usar `pybreaker` ou similar (lei R10 — minimal deps).

Property test + Nelo audit + Aria audit são as defesas. Mudança de default em `with_retry` é breaking-ish para chamadores internos — mitigado por factory `default_policy()`.
