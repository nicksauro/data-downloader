## Contexto / Problema

A revisão consolidada do squad (BIG COUNCIL Frente 5, 2026-05-16) identificou
**dois P0 arquiteturais** que travam a evolução do projeto:

### P0-A1 — `cli.py` monolito multifunção (~2.374 LOC)

`src/data_downloader/cli.py` agrega responsabilidades heterogêneas num único
arquivo:

| Bloco                                             | Linhas (aprox) | Responsabilidade                                 |
|---------------------------------------------------|----------------|--------------------------------------------------|
| Encoding setup + imports                          | 1-66           | Bootstrap antes de Rich/Typer                    |
| `dotenv` + credentials (`_get_credential`)        | 83-150         | Boot env (compartilhado com `ui/app.py`)         |
| Frozen-mode sentinels + microcopy                 | 154-231        | Map de erro pra microcopy R17                    |
| Global callback + healthcheck                     | 234-388        | Setup logging + `--healthcheck` self-check       |
| `contracts` sub-app (list/add/validate/vigent)    | 391-744        | 4 comandos de contrato                           |
| `_safe_day`                                       | 751-772        | Helper de calendário                             |
| `integrity` sub-app (check/validate-data)         | 776-963        | 2 comandos de integridade                        |
| `download` command + helpers                      | 966-1632       | Command + last_symbol cache + Rich progress      |
| `_detect_resumable_job` (auto-resume)             | 1640-1696      | Auto-resume v1.2.0                               |
| `doctor` command + 6 helpers `_check_*`           | 1700-2066      | Diagnóstico (5 default + 1 opt-in)               |
| `migrate` sub-app (plan/execute/rollback/cleanup) | 2070-2367      | 4 comandos de schema migration                   |

Consequência operacional: **adicionar um comando novo é refator no mesmo
arquivo gigante**. Story 4.22 (`catalog recover-pending`, frente 1 storage
atomicity) adiciona o 5º sub-app — sem decomposição, cada nova frente
empilha mais ~200 LOC no monolito, com merge conflicts inevitáveis
entre frentes paralelas.

### P0-A2 — `ARCHITECTURE.md §6` promete 5 Protocols, código tem 1

`ARCHITECTURE.md §6` (linhas 426-500), adicionado em v1.1.0 (H21) por Aria,
prescreve **5 Protocols** em `src/data_downloader/contracts/`:

```
contracts/
├── __init__.py
├── writer.py          # WriterProtocol
├── catalog.py         # CatalogProtocol
├── dll_client.py      # DLLClientProtocol
├── progress.py        # ProgressEmitter
└── handle.py          # DownloadHandle (Protocol; impl em public_api/download.py)
```

Estado real (2026-05-17):

```bash
$ ls src/data_downloader/contracts/
__init__.py  observability.py
```

Só `MetricsEmitter` (Story 2.4) foi implementado. Os 5 prometidos ficaram
como TODO por 4 releases (v1.1.0 → v1.3.0).

Consequência arquitetural: **boundary violations**. Refator em `storage/`
(ex.: Story 4.23 inverter Fase 1/replace, Story 4.24 advisory lock no
writer race) força mudanças em 6 outros lugares — `ui/`, `cli.py`,
`public_api/download.py`, `orchestrator/`, `tests/`. Acoplamento
estrutural impede evolução isolada.

### Isolamento de escopo (crítico)

Para evitar conflito com **Frente 4** (PR #5 aberto, toca `ui/`):

- **NÃO migrar UI** para usar Protocols nesta story. UI mantém imports
  concretos (`from data_downloader.storage.catalog import Catalog`).
- Migração UI fica em PR futuro (v1.5.0+) **após** esta story merger e a
  Frente 4 (Story 4.27) merger.
- Protocols criados aqui são **opt-in** — código existente usa classes
  concretas, novo código DEVE usar Protocols (regra explícita em ADR-030).

Outras restrições:

- **Story 4.31 AC4** já trocou `_PACKAGE_VERSION` por `importlib.metadata`
  — esta story NÃO duplica esse trabalho.
- **Story 4.22 (frente 1)** adiciona `catalog_app` no `cli.py` atual; esta
  story 4.28 absorve essa adição em `cli/catalog.py` (se já mergeada
  para `main` quando 4.28 iniciar, o submódulo recebe o código existente
  intacto; senão, fica como placeholder pronto pra receber).
- **Entry point** `pyproject.toml [project.scripts] data-downloader =
  "data_downloader.cli:app"` DEVE continuar válido. O símbolo `app`
  precisa existir em `data_downloader.cli` (pacote) — backward-compat
  obrigatória.

### Política de migração formalizada em ADR-030

A decisão "Protocols disponíveis mas opt-in, não big-bang" foi
formalizada em **ADR-030 — Protocol-First Boundary Policy** (criado
junto com este draft, status `Proposed`). Regra resumida:

| Quem                                       | Obrigação                                           |
|--------------------------------------------|-----------------------------------------------------|
| Código existente (v1.3.0)                  | Mantém imports concretos                            |
| Código novo (post-Story 4.28)              | DEVE usar Protocols ao cruzar fronteira de camada   |
| Testes novos                               | DEVEM type-hint via Protocols (mocks duck-typed)    |
| Implementações concretas                   | NÃO herdam do Protocol (structural subtyping)       |

ADR-030 transita `Proposed → Accepted` no commit final desta story
(após AC11 PASS).

### Restrição constitutional

- **Article IV (No Invention):** todo método dos Protocols rastreia para
  callsite real (auditável via grep). Sem invenção de superfície.
- **Article V (Quality First):** AC11 exige suite verde antes de Done.
- **R20 (suite estável):** zero quebra de teste existente. Adições não
  invalidam baseline (1809 passed Story 4.22 — manter ou crescer).

## Acceptance Criteria

### P0-A1 — Decomposição de `cli.py` em pacote `cli/`

- [ ] **AC1 — Pacote `src/data_downloader/cli/` substitui `cli.py`.**
  Layout final:
  ```
  src/data_downloader/cli/
  ├── __init__.py        # app: typer.Typer + bootstrap dotenv + registry de sub-apps
  ├── _helpers.py        # microcopy formatters, last_symbol cache, _format_*, _make_console
  ├── contracts.py       # sub-app contracts (list/add/validate/vigent)
  ├── integrity.py       # sub-app integrity (check/validate-data)
  ├── migrate.py         # sub-app migrate (plan/execute/rollback/cleanup)
  ├── catalog.py         # sub-app catalog (recover-pending — vindo da Story 4.22)
  ├── doctor.py          # comando doctor + 6 helpers _check_* + run_doctor_checks
  └── download.py        # comando download + _download_one_symbol + _detect_resumable_job
  ```
  - `src/data_downloader/cli.py` (arquivo) é REMOVIDO; pacote substitui.
  - Cada submódulo expõe `register(app: typer.Typer)` que chama
    `app.add_typer(...)` ou `@app.command()` em decorators no nível do módulo
    (escolha do submódulo — manter próximo do estilo atual).
  - `cli/__init__.py` define `app = typer.Typer(...)`, executa `_bootstrap_env()`
    (delegando a `data_downloader._env_loader` — não duplicar lógica),
    define `_global_callback` (logging + healthcheck) e chama os 7
    `register(app)` em ordem determinística.
  - Tamanho alvo: `__init__.py` ~120-180 LOC; `download.py` ~400-450 LOC
    (maior, contém Rich progress loop); cada outro submódulo ~150-350
    LOC. Total ~2.300-2.500 LOC (mesma soma, distribuída).

- [ ] **AC2 — Entry point intacto (backward-compat).**
  - `pyproject.toml` permanece com `data-downloader = "data_downloader.cli:app"`

  - Smoke CLI: `data-downloader --help` lista os mesmos sub-comandos

- [ ] **AC3 — Comandos e sub-apps preservados (sem regressão de UX).**

### P0-A2 — Protocols em `contracts/`

- [ ] **AC7 — Criar `src/data_downloader/contracts/_protocols.py` com 5
  Protocols `runtime_checkable`.**
  Conforme ADR-030 §2.1:
  - `WriterProtocol` — `write(trades, partition, *, catalog=None, job_id=None) -> WriteResult`; `compact_month(partition) -> WriteResult`; `close() -> None`.
  - `CatalogProtocol` — métodos críticos da fronteira real (NÃO todos os
    50+ públicos): `register_partition`, `recover_pending_commits`,
    `pending_commit` (context manager), `list_contracts_in_range`,
    `completed_days`, `maybe_compact_month`, `close`.
  - `DLLClientProtocol` — superfície mínima do wrapper:
    `initialize_market_only`, `wait_market_connected`,
    `get_history_trades`, `subscribe_ticker`, `unsubscribe_ticker`,
    `finalize`.
  - `ProgressEmitter` — `emit(event: ProgressEvent) -> None`. Decisão
    final entre "1 método unificado" vs "3 métodos (`emit_progress`/
    `emit_finished`/`emit_failed`)" segue ADR-030 Q3 (proposta: 1 método;
    confirmar com Aria no PR).
  - `DownloadHandle` — re-export simples de
    `data_downloader.public_api.handle.DownloadHandle` (ADR-030 Q2 —
    Opção (b) re-export). Senão duplicação com ADR-007a.
  - Todos com `@runtime_checkable` para suporte a `isinstance(x, Protocol)`.

- [ ] **AC8 — Exports via `contracts/__init__.py`.**
