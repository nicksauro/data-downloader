# Story 1.4 — Storage layer: writer Parquet + leitor DuckDB

## User Story

**As a** orchestrator (Story 1.7)
**I want** uma camada de storage que recebe `list[TradeRecord]` (de Story 1.3), persiste em Parquet (Snappy, row_group=100k) com metadata `schema_version`, e expõe leitor via DuckDB
**So that** os trades baixados ficam queryáveis por projetos downstream sem perder schema-versioning ou integridade.

---

## Acceptance Criteria

1. **AC1:** Schema v1.0.0 implementado conforme `docs/storage/SCHEMA.md` (entregue por Story 0.0 — Sol). Inclui campos novos: `side` (uint8), `ingestion_ts_ns` (int64 NOT NULL), `chunk_id` (string NOT NULL), `dll_version` (string NOT NULL coletado via Story 1.1 wrapper).
2. **AC2:** `src/data_downloader/storage/parquet_writer.py` expõe classe `ParquetWriter` com método `write(trades: list[TradeRecord], partition: PartitionKey) -> WriteResult`.
3. **AC3:** `PartitionKey` dataclass: `exchange: str (1 char)`, `symbol: str`, `year: int`, `month: int`. `WriteResult`: `path: Path`, `row_count: int`, `first_ts_ns`, `last_ts_ns`, `checksum_sha256: str`.
4. **AC4:** Path resolvido conforme ADR-004: `data/history/{exchange}/{symbol}/{year}/{month:02d}.parquet`. Diretório criado se não existir.
5. **AC5:** Escrita atômica: escreve em `{path}.tmp.{uuid}` → fsync(file) → calcula SHA256 → `os.replace(tmp, path)` (atômico Windows + Linux) → **fsync(parent_dir)** (finding H7 — garante diretório persistido antes de catalog avançar).
6. **AC6:** Metadata Parquet inclui chaves: `schema_version` (`"1.0.0"`), `row_count`, `first_ts_ns`, `last_ts_ns`, `dll_version` (placeholder por enquanto), `created_at` (ISO).
7. **AC7:** Configuração Parquet conforme ADR-002: `compression='snappy'`, `row_group_size=100_000`, `use_dictionary=True`, `write_statistics=True`.
8. **AC8:** Se arquivo já existe na partição, `write` faz **append + dedup** com chave canônica `(symbol, exchange, timestamp_ns, sequence_within_ns)` (conforme SCHEMA.md Story 0.0 — finding H2 reformulado). Lei R5.

   **Threshold rewrite vs new file (finding H6 — evita O(n²) append):** se arquivo existente > N MB (default 256MB, configurável), em vez de reescrever, criar `MM.{N}.parquet` (sequência incrementada). DuckDB reader (AC9) usa glob `MM.*.parquet`. Detalhes em INTEGRITY.md (Story 0.0).
9. **AC9:** `src/data_downloader/storage/duckdb_reader.py` expõe `DuckDBReader` com método `read(symbol, start_ts_ns, end_ts_ns, exchange='F') -> pa.Table`. Usa `read_parquet('data/history/{exchange}/{symbol}/**/*.parquet')` com filtro `WHERE timestamp_ns BETWEEN start AND end ORDER BY timestamp_ns ASC`.
10. **AC10:** Hypothesis property test: `dedup(L ++ L) == dedup(L)` (INV-2).

   **REMOVIDO (movido para Story 2.1):** round-trip property test (`read(write(L)) == sorted_dedup(L)`) vai para Story 2.1 (validators como código). Aqui em 1.4 ficou só dedup idempotência.

### Constraints CRÍTICAS
- **Atomicidade** — `os.replace` é atômico em ambos OS. Tmp file pattern essencial.
- **Dedup** chave é `(symbol, timestamp_ns, trade_id)` quando `trade_id` existe; senão hash canônico `(price, quantity, ?)`.
- **Append**: precisa ler existente, juntar, deduplicar, reescrever. Não tem append in-place no Parquet.
- **DuckDB conexão**: pode ser persistida (`DuckDBReader` mantém conn aberta).
