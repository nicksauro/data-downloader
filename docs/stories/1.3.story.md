# Story 1.3 — History download primitive: 1 símbolo / 1 chunk

## User Story

**As a** orchestrator (Story 1.7)
**I want** uma primitiva `download_chunk(dll, symbol, exchange, start, end) -> list[Trade]` que registra HistoryTradeCallback + ProgressCallback, chama `GetHistoryTrades`, drena callbacks via fila, aguarda fim, retorna trades
**So that** a Story 1.7 possa chunkar download grande em chunks diários e persistir cada chunk via Story 1.4.

---

## Acceptance Criteria

1. **AC1:** `src/data_downloader/dll/wrapper.py` ganha métodos `set_history_trade_callback(cb)` e `set_progress_callback(cb)` que registram `WINFUNCTYPE` apropriado (signatures via Nelo `*callback-spec HistoryTrade` e `*callback-spec Progress`). **DECISÃO EXPLÍCITA V1 vs V2 obrigatória:** preferir `SetHistoryTradeCallbackV2` (R10). Se V2 escolhido, `TranslateTrade` é chamado **fora** do callback (callback faz APENAS `put_nowait(payload_bruto)`; tradução roda em IngestorThread). Decisão documentada em comentário do código + auditada por Nelo.
2. **AC2:** `src/data_downloader/orchestrator/download_primitive.py` expõe função `download_chunk(dll, symbol, exchange, dt_start, dt_end, *, timeout: int = 1800) -> ChunkResult`.
3. **AC3:** `ChunkResult` é dataclass frozen com: `symbol`, `exchange`, `requested_start/end`, `actual_start/end` (do primeiro/último trade recebido), `trades: list[TradeRecord]`, `progress_history: list[int]`.
4. **AC4:** `TradeRecord` dataclass frozen com campos do schema v1.0.0 (vide `docs/storage/SCHEMA.md` quando criado por Sol; mínimo neste momento: `symbol`, `exchange`, `timestamp_ns`, `timestamp_str`, `price`, `quantity`, `trade_id`, `flags`).
5. **AC5:** Callback de história faz APENAS `_trade_queue.put_nowait()` — lei R3.
6. **AC6:** Função aguarda fim do download via callback de progresso (chega 100%) OU timeout. Quirk "99% reconectando" (manual silencioso, empírico Nelo): tolerar 99% por até `timeout` total; não considerar erro até timeout final. **Política unificada com Story 1.7a/1.7b:** referenciar `docs/ux/CLI_PATTERNS.md` (Story 0.3) para texto user-facing — Dex NÃO inventa microcopy aqui.
7. **AC7:** Datas convertidas para formato exato do manual: `"DD/MM/YYYY HH:mm:SS"` (manual §3.1 linha 1750 — Nelo). **Política 99% reconnect unificada com Story 1.7:** apenas referenciar — texto user-facing fica em CLI_PATTERNS.md (Uma).
8. **AC8:** Bolsa passada como letra única (`"F"` para BMF, `"B"` para Bovespa) — lei R8.
9. **AC9:** Timestamps recebidos preservados em **BRT naive** — lei R7. Conversão para `timestamp_ns` (epoch BRT naive) feita via parse do `timestamp_str` no formato `"DD/MM/YYYY HH:mm:SS.ZZZ"` (manual) ou `"DD/MM/YYYY HH:mm:SS:ZZZ"` (quirk com `:` antes ZZZ — Nelo). Aceitar ambos.
10. **AC10:** Smoke test gated por env (`PROFITDLL_KEY`): baixa 1 dia de WDOJ26 (data fixa: dia útil recente conhecido) e valida `len(trades) > 0`.

## Dev Notes

### Referências
- Manual ProfitDLL §3.1 (`GetHistoryTrades`), §3.2 (`HistoryTradeCallback`, `ProgressCallback`)
- `docs/dll/QUIRKS.md` (a criar) — Nelo documenta quirks de timestamp e 99%
- ADR-005 (thread model — filas bounded, block back-pressure)
- Lei R3 (callback = put_nowait), R7 (BRT naive), R8 (bolsa = letra), R9 (não chutar contratos)

### Constraints CRÍTICAS
- **Não chamar DLL dentro de callback** (R3) — toda lógica de drain está em thread separada.
- **Timestamp BRT naive** (R7) — não converter para UTC. `timestamp_ns` = nanos desde 1970-01-01 BRT.
- **Bolsa = `"F"` ou `"B"`** (R8). NÃO `"BMF"`.
- **Timeout >= 1800s** (Nelo quirk empírico) — default 1800.
- **V2 callback preferido** (R10) — `SetHistoryTradeCallbackV2` se disponível na versão da DLL.

### Cuidado com TradeRecord vs Sol schema
Sol vai definir o schema completo em Story 1.4. Aqui em 1.3, `TradeRecord` é dataclass mínima. Em 1.4, Sol pode pedir ajuste — Dex aplica.
