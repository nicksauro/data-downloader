# Story 4.10 — Dead buttons (Verificar Integridade / Reconciliar)

## Contexto

Audit UX da SettingsScreen v1.0.2 revelou **2 botões dead** na seção
Storage (settings_screen.py:320-326):

```python
self._integrity_btn = QPushButton(format_msg("BTN_INTEGRITY_CHECK"), section)
self._integrity_btn.setObjectName("integrityBtn")
actions.addWidget(self._integrity_btn)
# ⚠️ ZERO .clicked.connect

self._reconcile_btn = QPushButton(format_msg("BTN_RECONCILE"), section)
self._reconcile_btn.setObjectName("reconcileSettingsBtn")
actions.addWidget(self._reconcile_btn)
# ⚠️ ZERO .clicked.connect
```

Usuário clica → nada acontece → frustração silenciosa (não há nem
toast "feature em breve").

`CatalogScreen` já tem `CatalogAdapter` com `reconcile()` e
`revalidate_checksum()` implementados — basta cabear esses 2 botões
da Settings ao adapter (compartilhado ou paralelo).

## User Story

**As a** usuário que desconfia que catálogo está dessync com Parquet
files (após mover pasta, deletar manualmente, etc)

**I want** clicar "Reconciliar" / "Verificar Integridade" no Settings
e ver feedback real (count de partições reconciliadas, checksums
validados)

**So that** posso curar problemas de catálogo sem precisar abrir
terminal e rodar comando CLI.

## Acceptance Criteria

1. **AC1 — `_integrity_btn` invoca CatalogAdapter.revalidate_checksum
   em loop sobre todas as partições.**
   - Click → confirmação modal "Vai validar checksum de N partições
     (~30s para 100 partições). Continuar?".
   - Progress bar / spinner durante execução.
   - Resultado: toast "✓ N partições OK / M com drift detectado"
     + caso M>0, banner "Veja relatório completo" linkando para
     CatalogScreen com filtro `drift=true`.

2. **AC2 — `_reconcile_btn` invoca CatalogAdapter.reconcile.**
   - Click → toast "Reconciliando catálogo..." (loading state).
   - Resultado: toast "✓ Reconciliado: N partições adicionadas,
     M removidas".
   - data_dir_changed signal NÃO é emitido (data_dir não muda),
     mas CatalogScreen é notificado para refresh (signal novo
     `catalog_changed` ou cross-screen wiring).

3. **AC3 — Cross-thread safety (QT_PATTERNS §2).**
   - Adapters em QThread separada (já existem para CatalogScreen).
   - SettingsScreen não bloqueia MainThread; usa signals Queued.

4. **AC4 — Microcopy.**
   - Reuse `BTN_INTEGRITY_CHECK`, `BTN_RECONCILE` (já existem).
   - Novos: `MOD_INTEGRITY_CONFIRM_TITLE/BODY`, `TST_INTEGRITY_OK`,
     `TST_RECONCILE_OK`, `LBL_INTEGRITY_RUNNING`.

5. **AC5 — Smoke local Pichau.**
   - Pichau, com data_dir contendo ≥1 partição:
     - Clica "Reconciliar" → toast OK aparece.
     - Clica "Verificar Integridade" → modal confirm + toast OK.

## Out of Scope

- Implementação de drift detection automático (background) — V1.1.
- Reconciliação parcial (apenas range de meses) — V1.1.

## Risks

| Risco | Mitigação |
|-------|-----------|
| Validação de 1000 partições demora minutos | Confirmação modal mostra estimativa; cancellable via Esc. |
| Adapter Catalog vs Adapter próprio | Story usa o adapter existente (CatalogAdapter). MainWindow já injeta wiring; SettingsScreen recebe ref via construtor ou lookup. |

## Dev Notes (Felix)

- Reusar `CatalogAdapter` instanciado em CatalogScreen. Padrão:
  MainWindow expõe método `get_catalog_adapter()` ou injeta no
  construtor da SettingsScreen.
- Confirmação modal: `QMessageBox` com `Yes`/`No` (R17 microcopy).
