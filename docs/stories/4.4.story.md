# Story 4.4 — Auto-updater + packaging final V1 release

## Contexto

Story criada por mini-council Aria + Sol + Felix em **COUNCIL-13** (Epic 4
prep). Última story do Epic 4 e do release V1 do data-downloader. Após Story
4.3 (`__api_version__ = 1.0.0` + USAGE/DEPRECATION docs), o squad pode
publicar o instalador Windows com:

- Build determinístico (ADR-009 — `PYTHONHASHSEED`, `SOURCE_DATE_EPOCH`,
  lockfile).
- PyInstaller `--onedir` (ADR-003 amendment — facilita updates granulares
  vs `--onefile`).
- Auto-updater (ADR-017 — recomendação preliminar `tufup`, decisão final
  no início desta story após POC).
- Code signing Windows (ADR-016 — EV cert opcional V1; pode ser deferred
  para V1.1).
- GitHub Release pipeline com SHA256.

Esta story é **devops-heavy** (Gage owner, Felix implementa UI de
notificação e wrapper de installer). Aria é reviewer porque define escolha
final do tooling de updater (re-abrir ADR-017 conforme planejado).

A AC5 (code signing) é deliberadamente **flag opcional V1** — EV cert
custa $300/ano e tem 3-5 dias hábeis de emissão. Se Gage não concluir até
release window, signing fica para V1.1 e instalador ganha SmartScreen
warning aceitável (documentado em INSTALL.md).

A AC6 (smoke em VM Windows limpa) é **humano** — mesmo padrão Story 1.7b.

## User Story

**As a** Felix (UI installer integration) e Gage (devops release pipeline)
fechando o release V1 do data-downloader

**I want** instalador `.exe` Windows assinado (se cert disponível) com
auto-updater funcional (tufup ou alternativa selecionada após POC),
publicado via GitHub Release com SHA256 verificável, e validado em VM
Windows limpa por humano

**So that** o data-downloader pode ser instalado por novos usuários sem
fricção, atualizado automaticamente sem re-baixar manual, e o release V1
seja oficialmente publicado fechando os Epics 1-4.

## Acceptance Criteria

1. **AC1 — Auto-updater implementado (tufup ou alternativa pós-POC).**
   ADR-017 reabre no D-1 desta story para decisão final:
   - Aria + Gage rodam POC tufup em VM Windows (~2h).
   - Comparam atualização preliminar com Velopack (Opção F do ADR-017).
   - Decidem: tufup (recomendação preliminar) OU pivot.
   Implementação:
   - `src/data_downloader/updater/client.py` wrapper sobre tooling escolhido.
   - `src/data_downloader/updater/installer.py` lógica de apply (atomic
     replace `.tmp` + `os.replace`).
   - `src/data_downloader/updater/notifier.py` integração Felix (toast UI).
   - CLI `data-downloader self-update --check-only|--rollback`.

2. **AC2 — Build determinístico (ADR-009 conformance).**
   Pipeline build:
   - `PYTHONHASHSEED=0` setado no env de build.
   - `SOURCE_DATE_EPOCH=<commit-timestamp>` setado.
   - Lockfile `requirements.lock` (ou `uv.lock`) commitado e usado.
   - Test `tests/release/test_deterministic_build.py`: roda 2 builds
     sequenciais em CI (mesmo commit) e compara SHA256 do `.exe` —
     **devem ser idênticos** (target ADR-009).

3. **AC3 — PyInstaller `--onedir` output (ADR-003 amendment).**
   - Spec `build/data_downloader.spec` configurado para `--onedir`
     (não `--onefile`). Razão: `--onedir` permite updates granulares
     (substituir só arquivos changed via tufup), `--onefile` é monolítico.
   - Output em `dist/data_downloader/` (pasta) com `data_downloader.exe`
     no root.
   - Inclusão de companions: `ProfitDLL.dll`, `libcrypto-1_1-x64.dll`,
     `libssl-1_1-x64.dll`, `*.dat` (timezone2.dat etc), `theme.qss`,
     ícones.
   - Smoke `data_downloader.exe download --help` em CI Windows runner
     (sem DLL real — apenas verifica que carrega sem ImportError).

4. **AC4 — GitHub Release pipeline com SHA256.**
   - GitHub Action `release.yml` triggered em tag `v1.*`:
     - Build determinístico em Windows runner.
     - Empacota `dist/data_downloader/` em `data-downloader-{version}-win64.zip`.
     - Gera `SHA256SUMS.txt` com hash.
     - Upload como release asset.
     - Atualiza tufup metadata (targets/snapshot/timestamp) e publica em
       branch `gh-pages` ou release asset (decisão tooling).
   - Pipeline executável apenas por @devops (token escopado).

5. **AC5 — Code signing (ADR-016 — opcional V1, condicional).**
   - **Caminho A (cert disponível):** integrar signing no pipeline
     `release.yml` (signtool.exe via Azure Trusted Signing OU EV cert
     local). Smoke: instalador ganha selo "Verified Publisher: Squad
     data-downloader" no SmartScreen.
   - **Caminho B (cert não disponível V1):** documentar em `INSTALL.md`
     a etapa "More info → Run anyway" no SmartScreen + plano de upgrade
     V1.1 com signing. Aceitável V1 (squad uso interno + early adopters).
   - Decisão Caminho A vs B no D+1 da story (Gage informa status do cert).

6. **AC6 — Smoke run em VM Windows limpa (humano).**
   `tests/smoke/test_release_install.md` (protocolo manual humano):
   - Provisiona VM Windows 10/11 limpa (sem Python, sem ProfitDLL prévia).
   - Baixa instalador V1 do GitHub Release.
   - Verifica SHA256 contra `SHA256SUMS.txt`.
   - Executa instalador → app abre.
   - Configura ProfitDLL.dll path + credenciais (se humano tem).
   - Roda 1 download de teste (WDOJ26 1 dia).
   - Simula update: publica V1.0.1-rc1 com bump trivial → verifica que
     auto-updater detecta + propõe + aplica + restart com nova versão.
   - Evidência sanitizada salva em
     `docs/qa/SMOKE_EVIDENCE/4.4-{ISO}.md`.

7. **AC7 — `docs/release/INSTALL.md` para usuário final.**
   Documento criado em pt-BR (público alvo: usuários do squad +
   adopters externos):
   - Pré-requisitos (Windows 10+, ProfitDLL.dll obtida via Nelogica,
     credenciais Nelogica).
   - Passo-a-passo download + verificação SHA256 + execução.
   - Caso SmartScreen warning (Caminho B): instruções "More info → Run
     anyway".
   - Configuração inicial (ProfitDLL path, credenciais .env).
   - Troubleshooting (logs em %APPDATA%/data-downloader/, comando
     `doctor` para diagnóstico).
   - Auto-update: comportamento default + opt-out via Settings screen.
   - Rollback: `data-downloader self-update --rollback`.
