# Story 2.3 — Schema Migration Framework

## User Story

**As a** squad data-downloader (Sol custodian + Dex implementer + operadores em produção)
**I want** um framework executável para migrar `schema_version` de partições Parquet existentes (ex: v1.0.0 → v1.1.0 aditivo, v1.x.x → v2.0.0 quebrador) com dry-run, backup, checkpoint e rollback documentados
**So that** evolução de schema canônico (R4 — schema contrato perpétuo) não exija re-baixar dados históricos do zero, preservando dataset que pode chegar a 10+ GB e dezenas de meses de trades.

**NOTA:** Esta story implementa o esqueleto descrito em `docs/storage/MIGRATIONS.md` (Sol — v1.0.0 SCAFFOLD), transformando-o em código real + CLI. Foi finding **H16** do Plan Review original; entrou no Epic 2 como P1 porque sem ela primeiro bump de schema (provável em Epic 4 multi-asset) bloqueia release.

## Acceptance Criteria

1. **AC1 — Estrutura de pacote:** Existe `src/data_downloader/storage/migrations/__init__.py` expondo `Migration` (ABC), `MigrationRegistry`, `MigrationRunner`, `MigrationResult`. Subpastas: `parquet/` (migrations Python — ex: `v1_0_0_to_v1_1_0.py`) e `catalog/` (migrations SQL — `v{from}_to_{to}.sql`). Convenção de nome estritamente respeitada (regex `v\d+_\d+_\d+_to_v\d+_\d+_\d+\.(py|sql)`); Registry rejeita arquivos fora do padrão com mensagem clara.

2. **AC2 — Classe base `Migration`:** ABC `Migration` (em `migrations/_base.py`) com atributos obrigatórios `from_version: str`, `to_version: str`, `breaking: bool`, `description: str`, `rollback_supported: bool` e métodos abstratos `read_old(path) -> pa.Table`, `transform(table) -> pa.Table`, `write_new(table, dst, dry_run) -> MigrationResult`, `verify(src_old, dst_new) -> bool`. Default `read_old`/`write_new` em mixin `ParquetMigration` para reuso (subclasses só implementam `transform`).

3. **AC3 — CLI público:** Comando `data-downloader migrate --from vX.Y.Z --to vA.B.C [--dry-run] [--data-dir PATH] [--symbol X] [--continue-on-error] [--rollback] [--resume RUN_ID]` registrado em `cli.py` (Typer). Help inline em PT-BR conforme MICROCOPY_CATALOG (UX gate). Exit codes: `0` clean, `1` parse/erro, `2` pre-condition fail, `3` migration error, `4` rollback error.

4. **AC4 — Dry-run não escreve nada:** `--dry-run` lê catálogo, identifica partições afetadas, simula `transform` em memória (sem `pq.write_table`), reporta plano: número de partições, bytes lidos estimados, bytes escritos estimados, ETA, lista de migrations a aplicar. Test verifica `mtime` de arquivos data/ não muda + nenhum `.parquet` ou `.parquet.bak` criado.

5. **AC5 — Execução partição-a-partição com checkpoint:** Migration itera partições uma a uma; cada partição registra estado em tabela SQLite `_migration_log` (run_id UUID, partition_path, from_version, to_version, status `pending|migrated|rolled_back|failed`, started_at, completed_at, error). Comando `data-downloader migrate --resume {run_id}` retoma do último ponto `pending`/`failed` sem refazer `migrated`.

6. **AC6 — Catalog atualizado pós-migration:** Para cada partição migrada com sucesso, `catalog.partitions.schema_version` é atualizado de `from_version` → `to_version` em transação atômica junto com escrita do arquivo novo (semântica two-phase commit existente — finding H16 referência). Test integration valida: rodar `migrate v1.0.0 → v1.1.0`, depois ler catálogo, todas partições com `schema_version='1.1.0'`.

7. **AC7 — Rollback policy:** Antes de overwrite, framework cria backup `{partition}.parquet.bak` (mesmo diretório). Comando `data-downloader migrate --rollback --from vA.B.C --to vX.Y.Z` reverte: para cada partição em `_migration_log` com `status='migrated'`, restaura `.bak` (rename atômico), atualiza catálogo de volta. Migration sem `rollback_supported` recusa `--rollback` com mensagem clara (referência seção §4.2 MIGRATIONS.md).

8. **AC8 — Migration aditiva exemplo (v1.0.0 → v1.1.0):** `migrations/parquet/v1_0_0_to_v1_1_0.py` implementa Migration concreta: adiciona campo `liquidity_classification: uint8 nullable` (todos NULL no migrate; preservar campos existentes byte-a-byte). Migration de catálogo correspondente em `migrations/catalog/v1_0_0_to_v1_1_0.sql` (atualiza `_schema_meta.catalog_version`). Esta migration serve de teste real do framework + referência para futuras.

9. **AC9 — Suite de testes obrigatória:** Conforme MIGRATIONS.md §5:
   - `test_round_trip` — escreve dados v1.0.0, migra para v1.1.0, lê e compara: campos comuns iguais byte-a-byte + novo campo presente (todos NULL).
   - `test_idempotent` — rodar `migrate_file` duas vezes seguidas = segunda é no-op (detecta `schema_version` já correto e skip).
   - `test_rollback` — migrate → rollback → arquivo restaurado igual ao original (sha256 idêntico).
   - `test_dry_run_no_io` — dry-run não cria/altera arquivos (mtime preservado).
   - **Property test (Hypothesis):** `migrate(read_v1_table)` preserva todos os campos comuns (compara `to_pydict()` excluindo coluna nova — 100 examples).
   - **Pre-condition tests:** falha cedo se `_pending_commits` não vazio, lock file ativo, espaço em disco insuficiente.

10. **AC10 — Documentação atualizada:** `docs/storage/MIGRATIONS.md` §6 (Migrations registradas) recebe entrada `1.0.0 → 1.1.0` com link para PR + descrição. README do `migrations/__init__.py` em PT-BR explicando como adicionar nova migration (template + checklist). Sol audit trail registra esta story como gate de close do framework.

- Story 1.5 (Catalog two-phase commit) — semântica reusada para escrita atômica de migration.

### Constraints CRÍTICAS

- **R4 (schema contrato perpétuo):** Migration aditiva (bump minor) NÃO quebra leitores antigos — leitor de v1.0.0 lendo arquivo v1.1.0 ignora coluna nova. Migration quebradora (bump major) EXIGE ADR aceito + janela de manutenção documentada.
- **Backup OBRIGATÓRIO antes de overwrite** — `.bak` no mesmo diretório (rename atômico). Sem `--i-have-backup` flag explícita, framework recusa execução sem backup.
- **Pre-conditions duras:** `_pending_commits` vazio, lock file `data/.migrate.lock`, espaço em disco >= 2× tamanho do dataset.
- **Catálogo + arquivo em transação atômica:** se write falha, catálogo NÃO atualiza (semântica two-phase já existente em Story 1.5).
- **Property test obrigatório:** `migrate(read_v1) preserva campos comuns` — Hypothesis. Sem property test, story NÃO vai a InReview.
- **Sol é authority** — qualquer dúvida sobre semântica de schema escala para Sol via mini-council.

### Bloqueia

- Gate close Epic 2 (G-Quality-Final) — sem migration framework, primeiro bump de schema em Epic 4 bloqueia.
- Future: bump de schema canônico (provável Epic 4 multi-asset).

### NÃO bloqueia (paralelo)

- Story 2.4 (Prometheus exporter), Story 2.5 (calendar B3) — independentes.

### Foundation Impact

**ALTO.** Toca `storage/` (Sol authority). Framework é base para TODA evolução futura de schema. Bug aqui significa perda de dados em produção. Por isso Sol audit é obrigatório + Aria revisa CLI integration.

## Testing

- **Unit:**
  - ABC contract tests (Migration must implement métodos obrigatórios).
  - MigrationRegistry: discover by name regex, detect cycles, find shortest path.
  - MigrationLog: pending/migrated/failed transitions.
  - Pre-conditions: lock file, disk space, pending commits.

- **Integration:**
  - Round-trip migrate v1.0.0 → v1.1.0 em dataset sintético (10 partições).
  - Resume após crash artificial (mata processo após 3 de 10 partições; resume completa as 7 restantes).
  - Rollback completo após migrate (todos arquivos `.bak` restaurados, catálogo revertido).
  - Catalog + filesystem sincronizados pós-migrate (delegar `IntegrityChecker.check_catalog_filesystem_sync` da Story 2.1).

- **Property-based (Hypothesis):**
  - `migrate(read_v1) preserva campos comuns` — para qualquer Table v1.0.0 gerada pelo strategy de TradeRecord, após migration: todos os 17 campos canônicos preservados byte-a-byte; novo campo presente com NULL.
  - 100 examples mínimo.

- **Smoke (gated env):**
  - Após Story 1.7b smoke produzir dataset real, rodar `migrate v1.0.0 → v1.1.0 --dry-run` retorna plano sensato. Migrate real opcional (pode ser deferred — mock dataset basta para gate).
