# Story 1.7a — Orchestrator core

## User Story

**As a** Story 1.7b (CLI smoke MVP gate)
**I want** o core do orchestrator (chunker + retry + state machine + integração com Stories 1.3/1.5/1.6) implementado e testado isoladamente — sem CLI, sem public_api, sem smoke real
**So that** a Story 1.7b possa focar em CLI typer + public_api + smoke MVP, sem misturar responsabilidades (era o erro do 1.7 original — finding C10).

---

## Acceptance Criteria

1. **AC1:** `src/data_downloader/orchestrator/chunker.py`:
   - Função `chunk_range(symbol, start, end, *, chunk_days_map) -> list[ChunkRange]`
   - Mapa `CHUNK_DAYS = {'WDO': 5, 'WIN': 1, ...}` derivado do prefixo do contrato
   - Property test: união de chunks == intervalo solicitado, sem overlap, sem gap
2. **AC2:** `src/data_downloader/orchestrator/retry.py`:
   - Decorator/função `with_retry(fn, *, max_attempts=3, base_delay=30, retryable_errors=...)`
   - Backoff exponencial com jitter
   - Logging eventos `retry.attempt`, `retry.exhausted`
3. **AC3:** `src/data_downloader/orchestrator/state_machine.py`:
   - State machine `Running → DrainingDLL → DrainingWrite → Committed` (amendment ADR-005)
   - INV-11 enforced: declare "fim de chunk" só quando dll_queue vazia AND write_queue vazia AND último write commitou no SQLite (referência H11 — race no shutdown)
   - Transições logadas + observáveis (gauge `orchestrator_state`)
4. **AC4:** `src/data_downloader/orchestrator/orchestrator.py`:
   - Classe `DownloadOrchestrator(dll, catalog, writer)` com `run(symbol, exchange, start, end, *, resume_job_id=None) -> DownloadResult`
   - Resolução de contract code via Story 1.6 `vigent_contract`
   - Cache hit detection via Catalog (range fully covered → return cache_hit)
   - Loop de chunks: `chunker → for chunk: with_retry(download_chunk → writer.write → catalog.register_partition)` com state machine
   - Resume: se `resume_job_id` passado, calcula pending via Story 1.5
   - Cleanup órfãos antes de iniciar (delegado a Catalog.cleanup_orphans)
5. **AC5:** Política retry conforme finding C7/C8/H7 unificada:
   - Timeout default 1800s por chunk
   - Quirk 99%: tolerar 1800s sem progresso real após 99% (não considerar timeout até total)
   - Erro NL_*: retry com backoff (3 tentativas, base=30s)
   - Após 3 falhas: chunk marked `failed` + log + segue próximo chunk
6. **AC6:** AC novo: cache hit é range coverage REAL (não "partição existe") — verifica que range solicitado é subconjunto da união de ranges das partições registradas no catálogo (finding H8).
7. **AC7:** Logging estruturado eventos: `orchestrator.start`, `.contract_resolved`, `.chunk_start`, `.chunk_progress` (per chunk, não per trade — R21), `.chunk_complete`, `.chunk_failed`, `.cache_hit`, `.complete`. Inclui `correlation_id = job_id` (finding L2).
8. **AC8:** Integração test `tests/integration/test_orchestrator_mock.py`:
   - Mock DLL+writer
   - Fluxo completo (chunking, persistência, cache hit no 2º run)
   - State machine respeitada
9. **AC9:** Integração test `tests/integration/test_resume.py`:
   - Interrompe meio do caminho (mata processo)
   - Resume completa apenas chunks pendentes
   - Idempotência E2E (re-rodar = no-op)
10. **AC10:** Property test `tests/property/test_idempotency_e2e.py`: download(s, [a,b]) duas vezes ≡ uma vez.
