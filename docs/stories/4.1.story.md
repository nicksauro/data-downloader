# Story 4.1 — Multi-symbol broker process

## Contexto

Story criada por mini-council Aria + Sol + Felix em **COUNCIL-13** (Epic 4
prep). Implementa concretamente a decisão tomada em **ADR-015 (Multiprocess
catalog coordination — broker process)**, transformando o limite "1 DLL por
processo" (Nelógica) em paralelismo real **sem corromper o catálogo SQLite
único** (INV-6) e sem disparar `SQLITE_BUSY` (finding C9, validado por Sol).

A decisão arquitetural já está tomada (Opção A do ADR-015 — broker dedicado
no master, workers só leem catálogo R/O e enviam mutações via
`multiprocessing.Queue` com ACK). Esta story leva o ADR para código de
produção, integrando-o ao Orchestrator existente (`src/data_downloader/
orchestrator/orchestrator.py`, hoje single-symbol — Story 1.7a).

Pyro identificou em **H20** que `multiprocessing.Process` em Windows custa
2.7-10s de spawn por subprocess. Para evitar essa penalidade, esta story
exige **pool persistente** (workers aquecidos reusados entre jobs), não
spawn por job — alinhado a ADR-015 §"H20 mitigação".

Story 1.7b-followup (smoke real WDOJ26 com humano) é pré-requisito porque o
broker é multiplicador da operação base — paralelizar algo que não passou o
smoke single-symbol é prematuro. Story 1.7a fornece o orchestrator multi-chunk
que será reused pelos workers.

## User Story

**As a** Dex (executor) e Aria (responsável arquitetural) implementando
multi-asset support do data-downloader

**I want** um `CatalogBroker` rodando no processo master que serializa todas
as escritas SQLite, comunicando com workers via `multiprocessing.Queue`
(com protocolo ACK), e um pool persistente de N workers (cada um com sua
instância de ProfitDLL e seu Orchestrator)

**So that** podemos baixar N símbolos em paralelo (WDOJ26, WINH26, PETR4, ...)
sem race no catálogo SQLite, sem `SQLITE_BUSY`, sem spawn overhead repetido
em Windows, mantendo INV-6 (catalog é fonte única de verdade) e atingindo
speedup ≥ 3.2x para N=4 workers (target V1).

## Acceptance Criteria

1. **AC1 — `CatalogBroker` implementado.**
   `src/data_downloader/orchestrator/broker.py` define `CatalogBroker` (thread
   no master process — não subprocess) que:
   - Possui a única conexão R/W SQLite (`PRAGMA journal_mode=WAL`).
   - Drena `mutation_queue` (`multiprocessing.Queue`) em loop contínuo.
   - Aplica cada `CatalogMutation` em transação (commit por mutação para
     simplicidade V1; batching opcional via flag).
   - Devolve `CatalogAck` (`status='committed'|'rejected'|'error'`) via
     `ack_queue` específica do `job_id` correspondente.
   - Implementa shutdown gracioso (drena queue antes de sair).

2. **AC2 — Workers comunicam via `multiprocessing.Queue` com `CatalogClient`.**
   `src/data_downloader/orchestrator/broker_client.py` define `CatalogClient`
   stub usado pelo worker. Operações cobertas: `register_partition`,
   `mark_chunk_done`, `mark_chunk_pending`, `register_gap`, `update_job_progress`.
   Cada chamada é síncrona do ponto de vista do worker (envia mutação +
   bloqueia em ACK com timeout 10s — configurável). Usa `request_id` UUID
   para correlacionar mutação ↔ ACK.

3. **AC3 — Broker mantém SQLite write lock; serializa writes.**
   Apenas o broker abre conexão R/W SQLite (`isolation_level=None` +
   `PRAGMA journal_mode=WAL`). Workers que precisarem ler catálogo
   (idempotência check) abrem conexão **read-only** (`?mode=ro`, URI mode).
   Property test (Quinn — AC8) cobre: N=10 workers escrevendo
   concorrentemente nunca disparam `SQLITE_BUSY` no broker, e catálogo
   permanece consistente (sem partition record duplicado, sem chunk_id
   pendente órfão).

4. **AC4 — ACK protocol garante persistência percebida pelo worker.**
   `CatalogClient` só retorna controle ao caller (worker code) após ACK
   `committed` chegar. ACK `error` levanta `IntegrityError` (ADR-011).
   ACK `rejected` (regra de negócio violada — ex: range já registrado por
   outro worker) levanta `DownloadError` com mensagem humana. Timeout sem
   ACK (>10s) levanta `TimeoutError` — testado em integração simulando
   broker travado.

5. **AC5 — Pool persistente (sem spawn por job — finding H20).**
   `WorkerPool` em `broker.py` mantém N processos workers aquecidos
   (default N=4, configurável via CLI). Workers ficam em loop aguardando
   `JobRequest` na sua own input queue; processam um job de cada vez;
   sinalizam `JobResult` via output queue. Re-uso entre jobs evita o
   penalty de 2.7-10s spawn por download. Workers só morrem em shutdown
   do master (ou em crash detectado por monitor — broker reinicia se < N
   ativos).

6. **AC6 — CLI `download` aceita `--symbol` múltiplo + `--parallel`.**
   `data-downloader download --symbol WDOJ26 WINH26 PETR4 --parallel 4
   --start 2026-04-01 --end 2026-04-30` distribui os 3 símbolos no pool
   de 4 workers. Saída agrega progresso por símbolo (formato a definir
   com Uma — não bloqueante; placeholder usa formato existente
   indented-by-symbol). Ao final, agrega `JobResult` por símbolo em
   summary line. Flag `--parallel 1` força modo single-process (path
   atual de Story 1.7a — sem broker overhead).

7. **AC7 — `bench_multi_symbol` re-rodado com broker; speedup ≥ 3.2x para
   N=4 (target V1).**
   `benchmarks/bench_multi_symbol.py` (já existente — tocado em Story 1.4.5)
   é re-rodado contra implementação broker. Métricas registradas em
   `BASELINES.md` v2.0.0-broker:
   - Wall time single-process N=4 símbolos sequencial (baseline).
   - Wall time broker N=4 símbolos paralelo.
   - Speedup ≥ 3.2x (margem para overhead IPC + spawn cold-start;
     Pyro define exato após bench piloto).
   - Mutation throughput broker ≥ 5000 tx/s (consistente com ADR-015 §Performance).
   - Spawn cold-start uma vez por run (não por job — confirma pool persistente).
   - SQLITE_BUSY count == 0 (ZERO).

8. **AC8 — Testes integração + property + smoke gated.**
   - Integração (`tests/integration/test_broker.py`):
     - 4 workers concorrentes registram 100 partitions cada → todas
       persistidas, sem duplicates, sem race.
     - Worker crash simulado (`os.kill`) → broker detecta, reinicia
       worker, jobs reagendados (ou marcados como falhos com gap em
       catálogo conforme política).
     - Broker shutdown gracioso após drenar queue (workers aguardam
       ACKs pendentes).
   - Property (`tests/property/test_broker_invariants.py` via Hypothesis):
     - INV-6 preservada: para qualquer ordering de mutações, catálogo
       final é equivalente à sequência aplicada serialmente.
     - Idempotência: aplicar `register_partition(P)` duas vezes via
       workers diferentes não duplica linha (UPSERT no broker).
   - Smoke gated (`tests/smoke/test_multi_symbol_broker.py`):
     - 4 símbolos paralelos × 1 dia cada → completam sem erro.
     - Catálogo final tem 4 jobs, 4+ partitions, zero gaps inesperados.
     - Marcado `@pytest.mark.skipif(not env_has_dll())` — humano roda
       (igual padrão Story 1.7b).

## Dev Notes

- **ADR autoritativo:** ADR-015 (broker process) já em estado `accepted`.
  Sem decisão arquitetural pendente; story é implementação fiel.
- **Protocol Pattern:** `CatalogProtocol` (Aria define) garante que V1
  (single-process — `LocalCatalogClient`) e V2 (multi-process —
  `MultiprocessCatalogClient`) sejam intercambiáveis sem mudança de
  call-site no Orchestrator. Critical para minimizar diff.
- **Licença Nelógica:** ADR-015 assume "1 conexão DLL por processo".
  Confirmar com Nelo se múltiplas instâncias da mesma chave em processos
  diferentes na mesma máquina é OK. Risco operacional — escalado para
  Morgan/Nelo (ver EPIC-4 §Riscos).
- **Sem mudança de fronteira pública:** Esta story NÃO altera `public_api/`
  (download/read/etc continuam single-symbol no contrato). CLI é a única
  superfície que ganha multi-symbol em V1. Multi-symbol via public_api é
  Story 4.3+ (decisão Aria pendente — pode ser `download_batch(...)` ou
  loop client-side).
