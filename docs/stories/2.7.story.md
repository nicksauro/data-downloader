# Story 2.7 — Hot path tuning (HOT_PATH_RULES.md aplicado)

## User Story

**As a** Pyro responsável por throughput sustentado e Aria responsável por arquitetura de hot paths
**I want** auditar **todos os hot paths** (callbacks DLL `TNewTradeCallback`/`THistoryTradeCallback`, vectorized `_trades_to_table` pós-2.2, write loop, dedup loop) contra a lei nova **R21** (`MANIFEST.md`) + ADR-010 §"hot-path rules", **remover/migrar `structlog`** e qualquer formatação síncrona desses pontos, e formalizar o documento `docs/dev/HOT_PATH_RULES.md` com checklist auditável
**So that** o overhead de logging não consuma mais que 5% do tempo total de CPU em download contínuo (medido por Pyro `bench_callback_to_disk` + flame graph), fechando o finding **H22** do Plan Review (`structlog` em hot path = 50-150% de 1 core só logando) e o finding **F-Q-1** consolidado (cobertura `--cov` bloqueada por duckdb 1.x x Python 3.14 — hot path tuning também investiga workaround de cobertura formal como follow-up).

**NOTA:** Esta story é **gemêa** de Story 2.2 (vectorize ParquetWriter). 2.2 acelera o caminho de escrita; 2.7 garante que NADA no hot path emite log síncrono. Sem 2.7, o ganho de 2.2 (+400% throughput) pode ser parcialmente comido por structlog. Pyro mede ANTES (baseline pós-2.2) e DEPOIS (objective: log < 5% CPU em flame graph).

## Acceptance Criteria

1. **AC1 — Documento HOT_PATH_RULES.md formalizado:** Existe `docs/dev/HOT_PATH_RULES.md` com:
   - Definição operacional de "hot path": código executado **per-trade** ou **per-callback** em frequência ≥ 1000 Hz sustained.
   - Lista exaustiva de hot paths atuais (referência absoluta a arquivo:linha): callbacks DLL em `dll/wrapper.py`, `_trades_to_table` em `storage/parquet_writer.py` (pós-2.2), dedup `_apply_dedup` (pós-2.2), `IngestorThread.run` loop em `orchestrator/`.
   - Regras vetadas (do `from`): `structlog.get_logger().info/debug/...`, `logging.*`, `print`, `f"..."` apenas para log, `json.dumps` para serializar payload, `time.strftime`.
   - Padrões permitidos: contadores `Counter.inc()` (ADR-013, lock-free), `Queue.put_nowait` para streaming async para Pyro, `prometheus_client.Histogram.observe()` (atomic).
   - Auditoria mecânica: linter custom (script Python) que faz AST scan dos hot paths e falha se vê chamadas vetadas.

2. **AC2 — Auditoria mecânica via script:** Existe `scripts/audit_hot_path.py` que:
   - Lê whitelist de hot paths declarada em HOT_PATH_RULES.md (parse seção "Lista hot paths").
   - Faz AST scan em cada arquivo:função declarada como hot path.
   - Falha (exit 1) se encontra import/call de `structlog`, `logging`, `print`, `json.dumps`, `time.strftime` dentro do escopo.
   - Pre-commit hook adicionado (`.pre-commit-config.yaml` Story 0.2 estendido) — bloqueia commit que viola.
   - Test `tests/dev/test_audit_hot_path.py` com fixture sintética: arquivo violador → audit retorna fail; arquivo limpo → audit retorna pass.

3. **AC3 — `structlog` removido dos hot paths:** Áreas:
   - `dll/wrapper.py` — callbacks (`TNewTradeCallback`, `THistoryTradeCallback`) NÃO chamam `structlog`. Substituído por: contador atomic `dll_callback_invocations_total{type}` (ADR-013) + `Queue.put_nowait` para drain async em thread separada.
   - `storage/parquet_writer.py` — `_trades_to_table`, `_apply_dedup` (pós-2.2 vectorized) sem `structlog`. Métricas via Histogram observe (lock-free).
   - `orchestrator/IngestorThread` — loop principal usa eventos pré-allocated em deque, structlog só fora do loop.
   - Audit `scripts/audit_hot_path.py` PASS.

4. **AC4 — Per-chunk logging preservado:** `structlog` continua sendo usado em pontos per-chunk (boundaries entre chunks, não per-trade): `OrchestratorThread.complete_chunk`, `ParquetWriter.flush_buffer`, `Catalog.register_partition`. Eventos preservados: `chunk.started`, `chunk.completed`, `chunk.failed`, `partition.committed`. Fields incluem `correlation_id` via contextvar (ADR-010 §contextvars), `chunk_id`, `symbol`, `trades_count`, `duration_ms`.

5. **AC5 — Bench `bench_callback_to_disk` re-run pós-tuning:** Pyro roda `bench_callback_to_disk` (baseline existente Story 1.4.5) com configuração production (post-2.2 vectorized). Compara CPU usage breakdown via `cProfile` ou `py-spy`. Resultado documentado em `docs/perf/HOT_PATH_TUNING_2026-MM-DD.md`:
   - Antes: % CPU em structlog (esperado 30-50%).
   - Depois: % CPU em structlog (target < 5%).
   - Improvement absoluto (trades/s) + improvement relativo (%).
   - Flame graph antes/depois anexado (SVG ou PNG).

6. **AC6 — Async logging path para per-chunk eventos (opcional, se necessário):** Se análise revelar que `structlog` per-chunk ainda é gargalo em fluxos com muitos chunks pequenos (ex: símbolo com chunk_days=1), implementar `AsyncLogQueue` (queue + background thread drainer). NÃO usar `concurrent.futures.ThreadPoolExecutor` (fila simples basta). Este AC é condicional: PASS se Pyro mede gargalo OU PASS se mede que não há gargalo (negative result aceitável com evidência).

7. **AC7 — Tests + regression:**
   - **Unit:** `bench_callback_to_disk` usado como teste micro (assertion: CPU breakdown dentro do limite).
   - **Audit:** `audit_hot_path.py` rodado em CI (Story 0.2 pre-commit + GitHub Action se houver).
   - **Regression:** suite `bench/` rodada full pós-mudanças; nenhuma regressão > regression budget vs `BASELINES.md` v1.1.0-mock (atualizar para v1.0.0-real quando 1.8-followup PASS).
   - **Property test (Hypothesis):** invariante "número de logs emitidos = número de chunks completed" (não escala com número de trades) — 100 examples.

8. **AC8 — F-Q-1 follow-up (cobertura --cov bloqueada por duckdb x Python 3.14):** Investigação documentada em `docs/dev/COVERAGE_WORKAROUND.md` (NEW — pode ser muito curto):
   - Cenário do bloqueio (duckdb 1.x falha em hook coverage do Python 3.14 — referência QA report 1.7a §F-Q-1).
   - 3 opções avaliadas: (a) downgrade Python alvo para 3.12/3.13, (b) bump duckdb para versão compatível com 3.14, (c) plugin coverage customizado que skipa duckdb.
   - Recomendação Pyro+Aria com justificativa.
   - Decision deferred para outra story se workaround complexo (≥ 1d adicional). Este AC é PASS se documento existe com análise + recomendação clara, INDEPENDENTE de implementar o fix (que pode virar Story separada).
