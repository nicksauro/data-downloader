## Contexto / Problema

A revisão consolidada Frente 2 v1.4.0 (2026-05-16) identificou que o repositório **não tem CI** (`.github/workflows/` inexistente — `ls .github` retorna `No such file or directory`). Toda a disciplina R13 (gates pré-commit) e R14 (AUDIT por release) depende de execução humana sem barreira mecânica, com 5 riscos P0:

- **P0-R1 — Suite full não roda em PR.** ruff/mypy/pytest dependem do contributor lembrar de rodar. Pelo menos 2 ocasiões documentadas de skip (`docs/qa/RELEASE-GATE-v1.1.0-RESULTS.md` round 1).
- **P0-R2 — Build em Python 3.14** com `requires-python>=3.12`. Story 4.31 AC8 (`d4ea928 build(determinism): Python 3.12.x pinning guard`) adicionou guard em `scripts/build_release.py`, mas é guard reativo (dispara quando alguém roda). CI **previne** antes.
- **P0-R3 — `verify-dll-companions.py` falha durante build.** Evidência em `err1.log` (1 linha: `cp: cannot create regular file 'dist/data_downloader/scripts/verify-dll-companions.py': No such file or directory`). Script roda antes do bundle estar pronto.
- **P0-R4 — RELEASES.md sem auto-AUDIT por release.** Story 4.31 AC5 (`0e71040 docs(release): backfill RELEASES.md com v1.1.0..v1.3.0`) backfilled o histórico — agora precisamos garantir que toda release futura emite entry automático.
- **P0-R5 — SHA256 publicado fora-de-banda.** Hoje o squad gera SHA256 localmente e copia para `RELEASES.md`/`CHANGELOG.md` no commit pós-release. Auditabilidade fraca para terceiros (consumidor não tem trust path do binário).

ADR-027 §2 prescreve estrutura de 2 workflows (`test.yml` + `release.yml`) em runner `windows-latest` único, Python `3.12` single, com 4 jobs paralelos em test e 5 jobs sequenciais em release (incluindo auto-AUDIT bot commit).

**Esta story é independente da Frente 3 (Story 4.26 rollover-safety) — podem ser implementadas em paralelo.** Dependência downstream: stories de v1.4.1+ que exigirem secrets adicionais (code signing, S3 bucket de DLLs).

## Acceptance Criteria

- [x] **AC1 — `.github/workflows/test.yml` criado.** Triggers: `pull_request` (branches: main), `push` (branches: main), `workflow_dispatch`. Concurrency group por PR/branch (`group: test-${{ github.ref }}, cancel-in-progress: true`) para cancelar runs obsoletos quando novo commit é pushed. Runner: `windows-latest`. Setup: `actions/checkout@v5` + `actions/setup-python@v6` com `python-version: '3.12'` + pip cache (`cache: 'pip', cache-dependency-path: 'pyproject.toml'`).

- [x] **AC2 — 4 jobs paralelos em `test.yml`:**
  1. **`lint-type`** — `pip install -e ".[dev]"` → `ruff check src/ tests/` → `mypy --strict src/data_downloader/`. Timeout job: 10min.
  2. **`unit-tests`** — `pip install -e ".[test,dev]"` → `pytest tests/unit -v --maxfail=5 --timeout=60 --junitxml=junit-unit.xml`. Upload `junit-unit.xml` como artifact (retention 30 dias). Timeout: 15min.
  3. **`integration-property-tests`** — `pip install -e ".[test,dev]"` → `pytest tests/integration tests/property -v --maxfail=3 --timeout=180 --junitxml=junit-integration.xml -m "not smoke"`. Upload artifact. Timeout: 20min.
  4. **`pre-commit`** — `pip install pre-commit` → `pre-commit run --all-files`. Inclui bandit + pip-audit + detect-secrets (já configurados em `.pre-commit-config.yaml` pós-Story 4.31 AC6). Timeout: 10min.

- [x] **AC3 — `.github/workflows/release.yml` criado.** Triggers: `push` com `tags: ['v*']` e `workflow_dispatch` com `inputs: {version: {required: true}, skip_tests: {type: boolean, default: false}}`. Runner: `windows-latest`. Mesma configuração de Python/cache do test.yml.

- [x] **AC4 — 5 jobs sequenciais em `release.yml`:**
  1. **`validate-version`** — Extract version da tag (ou input). Validate `^\d+\.\d+\.\d+$`. Comparar com `version` em `pyproject.toml` + `__init__.py` + `installer/data_downloader.iss` (drift = fail). Validar que `CHANGELOG.md` tem seção `[v{version}]` matching regex de `scripts/github_release.py`. Outputs: `version`, `commit_sha`.
  2. **`full-test-suite`** (depends: validate-version) — Re-roda os 4 jobs de `test.yml` (reuse via `uses: ./.github/workflows/test.yml` se possível; fallback inline). Condicional: skip se `inputs.skip_tests == true` (workflow_dispatch escape hatch para hotfixes) + emite warning no `$GITHUB_STEP_SUMMARY`. Timeout total: 25min.
  3. **`build-windows`** (depends: full-test-suite) — Setup Python 3.12. `pip install -e ".[build,test,dev]"`. **DLL bootstrap fallback (v1.4.0):** tenta `aws s3 cp s3://${{ secrets.PROFITDLL_S3_BUCKET }}/profitdll/DLLs/Win64/ profitdll/DLLs/Win64/ --recursive` se secret existe; senão emite warning e skip bundle PyInstaller (gera `release-no-binary.json` placeholder). Quando bootstrap OK: roda `python scripts/build_release.py --version ${{ needs.validate-version.outputs.version }}` + `python scripts/verify-dll-companions.py dist/data_downloader/` para validar bundle. Upload `dist/data-downloader-v{version}-win64.zip` + `dist/build-manifest-v{version}.json` como artifacts. Timeout: 15min.
  4. **`build-installer`** (depends: build-windows, condicional: bundle exists) — Install InnoSetup via chocolatey (`choco install innosetup -y --no-progress`) OU download direto. Roda `iscc installer\\data_downloader.iss /DAppVersion=${{ needs.validate-version.outputs.version }}`. Upload `installer/Output/data-downloader-v{version}-setup.exe` como artifact. Timeout: 10min.
  5. **`auto-audit-and-release`** (depends: build-windows, build-installer) — Download artifacts. Re-compute SHA256 de cada artefato (sanity check vs manifest). Append entry em `docs/release/RELEASES.md` no formato:
     ```markdown
     ## v{version} — {YYYY-MM-DD}
     - **Released:** {iso_timestamp_utc} via [workflow #{run_id}]({workflow_url})
     - **Tag:** v{version}
     - **Commit:** {git_sha} ({short_sha})
     - **Artifacts (SHA256):**
       - `data-downloader-v{version}-win64.zip` — `{sha256}` ({size_human})
       - `data-downloader-v{version}-setup.exe` — `{sha256}` ({size_human})
       - `build-manifest-v{version}.json` — `{sha256}` ({size_human})
     - **CHANGELOG:** [GitHub Release]({release_url})
     ```
     Commit + push (`git push origin main`) usando `${{ secrets.GITHUB_TOKEN }}` com `[skip ci]` no message para evitar trigger loop. Invoca `python scripts/github_release.py --version ${{ needs.validate-version.outputs.version }}` para criar GH Release oficial com body do CHANGELOG section + assets anexados. Timeout: 5min.

- [x] **AC5 — `.github/branch-protection.md` documentado (não automatizado).** Markdown explicando configuração que humano deve aplicar via GH UI/API:

# Suite full (espelhar o que CI roda)
ruff check src/ tests/

**Riscos identificados:**
- **DLL bootstrap em CI** — `windows-latest` runner não tem ProfitChart. Sem S3 (ou alternativa) configurado, jobs `build-windows` ficam parciais (gera `release-no-binary.json`). v1.4.0 fechado com fallback; v1.4.1+ implementa solução completa.
- **Custo** — Windows runner ~$0.08/min. PRs com 4 jobs paralelos × ~15min = ~$5/PR. Pode escalar; monitorar via `actions/billing`.
- **Workflow loop** — auto-AUDIT bot push para main poderia retriggerar test.yml. Mitigado por `[skip ci]` no commit message + `if: github.actor != 'github-actions[bot]'` opcional em `test.yml` triggers.
- **actionlint false positives** — hook é strict; pode bloquear PR por warnings benignos. Configurar `.actionlint.yaml` se necessário para tolerar padrões legítimos.
- **Secrets em forks** — fork-based PRs não recebem secrets por padrão (segurança GH). `release.yml` bot push falha em fork. Documentar em SECRETS.md.
