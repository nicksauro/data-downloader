# Story 1.8 — Pyro baselines reais + regression budgets

## User Story

**As a** Pyro (perf-engineer) que precisa converter os baselines synthetic (Story 1.4.5) em baselines REAIS após smoke MVP verde
**I want** rodar `bench_chunking` (com download real WDOJ26 30 dias), `bench_callback_to_disk` real (com flame graph), e atualizar `BASELINES.md` + `REGRESSION_BUDGETS.md` com números observados
**So that** o gate de close do Epic 1 tenha baselines vinculados a comportamento real, não a fixtures sintéticos — e Epic 2/3/4 tenham referência para detectar regressão.

---

## Acceptance Criteria

1. **AC1:** Após Story 1.7b smoke verde, Pyro roda `bench_chunking`:
   - Tempo total download 30 dias WDOJ26
   - Throughput trades/s
   - Nº reconexões (eventos `download.99_reconnect_detected`)
   - Latência por chunk (p50/p95/p99)
2. **AC2:** Pyro roda `bench_callback_to_disk` REAL (não synthetic):
   - Mede latência callback→disk com DLL real
   - Captura flame graph via `py-spy` ou `cProfile`
   - Identifica top 5 hot paths
3. **AC3:** `docs/perf/BASELINES.md` ganha seção "Real baselines (Story 1.8)":
   - Comparação synthetic vs real (delta %)
   - Baseline número-por-número
   - Data + hardware + OS + Python version + DLL version
4. **AC4:** `docs/perf/REGRESSION_BUDGETS.md` revisado:
   - Budgets recalibrados baseados em baseline real
   - Threshold de FAIL atualizado
5. **AC5:** Pyro entrega flame graph como `docs/perf/flame_chunking.svg` + `docs/perf/flame_callback_to_disk.svg`.
6. **AC6:** Pyro abre 0 ou mais Hot Path Issues em `docs/perf/HOT_PATH_ISSUES.md`:
   - Cada issue: hot path identificado, % CPU, sugestão de otimização, severidade (info/warn/critical)
   - Issues `critical` viram stories de Epic 2
7. **AC7:** Pyro `*regression-check` confirma sem regressão vs baselines anteriores (synthetic).
8. **AC8:** Sol revisa que ingestion_ts_ns / chunk_id (Story 0.0 SCHEMA AC1) estão presentes nos Parquets gerados pelo smoke — usados por Pyro para reconstruir timeline.
9. **AC9:** Aria revisa flame graph e propõe ADRs adicionais se identificar débito arquitetural.
10. **AC10:** Morgan registra GO em release-readiness só após AC1-AC9 OK.
