# Story 2.4 — Prometheus Observability Exporter V2

## User Story

**As a** squad data-downloader (operadores que rodam jobs longos + Pyro que precisa de baselines objetivos + Aria que define fronteira observability)
**I want** exportar métricas runtime (counters, gauges, histograms já definidas em ADR-013) via endpoint HTTP `/metrics` no formato Prometheus
**So that** ferramentas externas (Grafana dashboards, Alertmanager, scrape de SRE) possam coletar dados objetivos do data-downloader sem depender de parsing de logs JSON ad-hoc (V1 era stdout JSON via structlog — útil para forense, inadequado para dashboard live).

**NOTA:** Esta story implementa a **V2 deferred** documentada em ADR-013 §"Export V2 (Epic 4 — multi-symbol): exporter HTTP" — antecipada para Epic 2 porque Story 1.7b release readiness exige métricas live para validar smoke MVP em ambiente de produção (humano + ProfitDLL real).

## Acceptance Criteria

1. **AC1 — Dependência autorizada:** `prometheus_client>=0.20` adicionado a `pyproject.toml` em `[project.dependencies]`. Autorização: ADR-013 §Decisão (Opção A escolhida; dep nova explicitamente autorizada). Pip install + lock file regenerado; `aiox doctor` (se aplicável) ou `pip check` clean.

2. **AC2 — Subpacote `observability/`:** Existe `src/data_downloader/observability/__init__.py` exportando registries e helpers + `prometheus_exporter.py` (HTTP server) + `metrics.py` (definições de Counter/Gauge/Histogram listadas em ADR-013 §Métricas V1). Estrutura limpa: nenhum import circular com `storage/`, `dll/` ou `core/` (Aria valida).

3. **AC3 — Métricas conforme ADR-013:** Registry contém pelo menos:
   - **Counters:** `trades_received_total{symbol,exchange}`, `parquet_writes_total{symbol}`, `parquet_bytes_written_total{symbol}`, `dll_reconnects_total{reason}`, `chunks_completed_total{symbol,status}`, `errors_total{type}`, `dedup_dropped_total{symbol}`, `dll_drops_total{reason}` (novo — finding ADR-013 V2).
   - **Gauges:** `queue_depth_dll`, `queue_depth_write`, `queue_depth_ui_progress`, `active_jobs`, `dll_state`, `partition_count{symbol}`.
   - **Histograms:** `chunk_duration_seconds{symbol}`, `callback_to_disk_seconds{symbol}` (com p99 buckets), `parquet_write_seconds{symbol}`, `dll_request_seconds`, `sqlite_commit_seconds`.
   - Todas com prefixo `data_downloader_` (cardinality control: `symbol` rotacionado por LRU top-50 + `other`).

4. **AC4 — Exporter HTTP configurável:** `prometheus_exporter.py` expõe `start_metrics_server(port: int) -> None` que invoca `prometheus_client.start_http_server(port)`. Porta default `9090`; override via env var `DATA_DOWNLOADER_METRICS_PORT` ou flag CLI `--metrics-port`. Se porta ocupada, log warning + tentar próxima até `port+10` ou abortar com mensagem clara.

5. **AC5 — Hook no orchestrator via Protocol pattern:** Orchestrator (Story 1.7a) publica eventos via `ProgressEmitter` Protocol (Aria contracts — finding em COUNCIL-05). `PrometheusEmitter(ProgressEmitter)` traduz cada evento em increment/observe na métrica correspondente. NÃO toca código do orchestrator diretamente — usa pattern de subscribe (Aria authority sobre fronteira). Test: emitter recebe eventos sintéticos, métricas refletem corretamente.

6. **AC6 — CLI flag `--metrics-port`:** `data-downloader download --metrics-port 9090 [--symbol WDOJ26 ...]` ativa exporter ANTES de iniciar download. Mensagem de boot: `"Métricas Prometheus expostas em http://localhost:9090/metrics"`. Se flag omitida, exporter NÃO inicia (zero overhead default — opt-in). Microcopy PT-BR validada por Uma (handoff via review).

7. **AC7 — Suite de testes:**
   - **Unit:** Cada Counter/Gauge/Histogram tem teste `inc()`/`set()`/`observe()` + `REGISTRY.collect()` retorna sample correto. Fixture `reset_registry` (introduzida em Story 1.1 amendment ADR-013) usada para isolamento.
   - **Integration:** `start_metrics_server(9091)` em test (porta efêmera), HTTP GET `/metrics` retorna 200 + body válido formato Prometheus exposition (parseável por `prometheus_client.parser.text_string_to_metric_families`).
   - **Format compliance:** test parseia output e confirma que cada métrica esperada está presente com tipo correto (Counter/Gauge/Histogram).
   - **Overhead test:** Pyro `bench_observability_overhead` (já existente — ADR-013 §Validações) confirma increment <500ns e exporter HTTP idle adiciona <2% CPU (regression budget — finding `REGRESSION_BUDGETS.md`).

8. **AC8 — Documentação ops:** `docs/dev/OBSERVABILITY.md` atualizado com:
   - Como ativar exporter (flag CLI + env var).
   - Lista completa de métricas exportadas (tabela Counter/Gauge/Histogram + significado).
   - Exemplo `curl http://localhost:9090/metrics`.
   - Exemplo `prometheus.yml` scrape config.
   - Notas sobre cardinality control (LRU símbolos).
   - Troubleshooting (porta ocupada, métricas zeradas, etc.).

## Dev Notes

### Referências
- `docs/adr/ADR-013-observability.md` — fonte primária; Opção A escolhida; V2 HTTP exporter deferred para Epic 4 mas antecipado para Epic 2.
- `docs/decisions/COUNCIL-05-*.md` (D3 — ProgressEmitter Protocol — Aria contracts).
- ADR-005 (thread model) — exporter roda em thread daemon `prometheus_client`; sem interferir hot path.
- ADR-010 (logging — R21) — counters substituem logs per-trade; este story REFORÇA R21.
- MANIFEST §R21 — não logar per-trade; usar counters.
- Plan Review finding **H22** — hot path tuning relacionado.
- `agents/architect.md` — Aria authority sobre fronteira (Protocol pattern).
- `agents/perf-engineer.md` — Pyro authority sobre overhead budget.
- Story 1.7a (orchestrator) — provedor dos eventos via ProgressEmitter.
- Story 1.7b (CLI smoke MVP) — consumidor da nova flag `--metrics-port`.

### Constraints CRÍTICAS

- **Hot path safe (O(1)):** Increment de Counter / set de Gauge / observe de Histogram tem custo amortizado <500ns (lock-free na lib). Tests verificam.
- **Opt-in (zero overhead default):** Sem `--metrics-port` flag NEM env var, exporter NÃO inicia. Counters in-memory continuam disponíveis (REGISTRY global), mas HTTP server não consome porta nem CPU.
- **Sem refactor de orchestrator:** PrometheusEmitter é subscriber via Protocol — Aria authority. Editar orchestrator diretamente = violação de fronteira.
- **REGISTRY singleton:** Tests precisam fixture `reset_prometheus_registry` para isolamento (REGISTRY é módulo global em `prometheus_client`). Quinn fixture já provista (ADR-013 §Validações requeridas).
- **Cardinality control:** label `symbol` tem ~500 contratos vivos potenciais. Mitigação obrigatória: LRU top-50 + `other` (ADR-013 §Cardinality).
- **R21 reforço:** Sem `dll_drops_total` no V1 era impossível medir saúde DLL; este story fecha o gap.

### Bloqueia

- Gate close Epic 2 (G-Quality-Final exige `curl http://localhost:9091/metrics` retornar 200 com métricas válidas).
- Release readiness V1 — operadores precisam de dashboard.

### NÃO bloqueia (paralelo)

- Story 2.3 (migration framework), Story 2.5 (calendar B3) — independentes.
- Story 2.7 (hot path tuning) — depende DESTA (medir overhead pré/pós).

### Foundation Impact

**MÉDIO.** Toca observability/ (módulo novo, sem dependentes diretos). Hook no orchestrator é via Protocol (subscribe pattern) — sem refactor de produção. Bug aqui afeta ops/dashboard mas não afeta integridade de dados (storage intacto).
