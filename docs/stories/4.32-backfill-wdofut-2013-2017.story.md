## Contexto / Problema

Em 2026-05-24 o arquivo WDOFUT existente cobre **2018-01-02 → 2026-05-22** integralmente (2083 dias em `chunk_ledger`, 115 parquets, ~22 GB em `D:\data-downloader\data\`). Auditoria de hoje confirmou:

- 2 gaps cosméticos legados (2018-05-31 Corpus Christi, 2018-07-09 feriado SP)
- 1 calendar mismatch (2022-12-26)
- Bug OOM da compactação mensal **CORRIGIDO** em `d53e4c1` (PR #8 aberto contra `main`)

O usuário Pichau quer **estender o arquivo para trás**, baixando WDOFUT de **2012-01-01 → 2017-12-31** — adicionando ~6 anos de histórico para uso em backtests/research mais longos. O range pré-2018 nunca foi tocado, então não há `chunk_ledger` rows; precisamos descobrir o limite real do ProfitDLL nesse período (data mais antiga em que a API ainda retorna trades) e proceder a partir daí.

Restrições conhecidas:

- **OOM bug** se rodar de branch antiga → MUST checkout `fix/v1.4.0-compaction-oom` ou merge PR #8 antes.
- **Calendar coverage** → `holidays.dat` do ProfitDLL (`profitdll/DLLs/Win64/holidays.dat`) cobre 2013-2035; tem que ser passado via env var `DATA_DOWNLOADER_HOLIDAYS_DAT_PATH` (caso contrário, fallback no calendar embutido stalla em dias não-pregão).
- **ProfitDLL data range** pré-2014 é incerto — Nelogica não documenta data de início da história de WDO; precisa probe.

## User Story

- **As a** consumer of WDOFUT historical data (quant research, backtest engine, signal generator)
- **I want** trades de 2012-2017 no mesmo arquivo parquet (`D:\data-downloader\data\history\F\WDOFUT\`)
- **So that** backtests possam usar uma janela histórica maior (~14 anos contínuos: 2012/13/14 → 2026), estendendo o arquivo atual 2018-2026 retroativamente
