# Story 0.0 — Sol cria docs canônicos (SCHEMA, CONTRACTS, INTEGRITY)

## User Story

**As a** squad data-downloader prestes a iniciar implementação do Epic 1
**I want** que Sol formalize 3 documentos canônicos (SCHEMA.md, CONTRACTS.md, INTEGRITY.md) em `docs/storage/` ANTES de qualquer story de implementação consumir essas decisões
**So that** Stories 1.3 (download primitive), 1.4 (storage), 1.6 (contract calendar) consultem fonte única de verdade — eliminando decisões implícitas e prevenindo schema drift.

---

## Acceptance Criteria

1. **AC1:** Existe `docs/storage/SCHEMA.md` documentando schema Parquet v1.0.0 completo:
   - Tabela de campos (nome, tipo pyarrow, nullable, descrição, exemplo)
   - Campos obrigatórios novos: `side` (uint8, derivado de trade_type), `ingestion_ts_ns` (int64, NOT NULL — timestamp do append no Parquet), `chunk_id` (string, NOT NULL — UUID do chunk de origem), `dll_version` (string, NOT NULL — coletado via GetDLLVersion)
   - Política de evolução (`schema_version` em metadata Parquet; semver MAJOR.MINOR.PATCH; quando bumpa cada um)
   - Política de migração (referência a Story H16 / framework futuro)
2. **AC2:** SCHEMA.md inclui seção "Catálogo SQLite v1.0.0" com DDL completo de TODAS as tabelas (`downloads`, `partitions`, `gaps`, `contracts`, `_schema_meta`, `_checksum_cache`) — fonte única para Story 1.5.
3. **AC3:** SCHEMA.md inclui seção "Dedup Key" — chave canônica `(symbol, exchange, timestamp_ns, sequence_within_ns)` onde `sequence_within_ns` é índice 0..N de trades dentro do mesmo nanossegundo (garante dedup determinístico mesmo com `trade_id` NULL).
4. **AC4:** Existe `docs/storage/CONTRACTS.md` documentando:
   - Tabela letras de mês CME/B3 (F=Jan..Z=Dec)
   - Regras de vigência por símbolo (WDO mensal, WIN trimestral H/M/U/Z)
   - Mapa inicial de contratos validados (mínimo: WDOH26, WDOJ26, WDOK26)
   - Formato seed (YAML embedded ou Markdown table parseável) consumido por Story 1.6
5. **AC5:** Existe `docs/storage/INTEGRITY.md` documentando:
   - Two-phase commit emulado: catalog `pending_commit` → arquivo `os.replace` → catalog `committed`
   - Política `fsync(parent_dir)` pós-replace (evita catalog avançar antes de filesystem persistir)
   - Reconciliação automática no startup do orchestrator
   - Política WAL checkpoint pós `register_partition` (evita perda em crash)
   - Detecção de drift A/B/C (catálogo→fs, fs→catálogo, checksum mismatch)
   - Threshold rewrite vs new file (evita O(n²) append+dedup) — quando partição cresce além de N MB, criar novo arquivo `MM.{N}.parquet`
6. **AC6:** Os 3 docs cross-reference entre si com links relativos.
7. **AC7:** Aria revisa e confirma que docs não conflitam com ADRs vigentes (ADR-002, ADR-004, ADR-006).
8. **AC8:** Quinn revisa que cada AC de Stories 1.3/1.4/1.5/1.6 tem rastreabilidade para um trecho de SCHEMA.md/CONTRACTS.md/INTEGRITY.md.
9. **AC9:** Morgan adiciona referência cruzada nos epic 1 doc e nas stories 1.3/1.4/1.5/1.6 (já fará no rework do roadmap).
10. **AC10:** Versão dos 3 docs é `v1.0.0`. Mudanças após esse marco exigem bump (referência a R4).
