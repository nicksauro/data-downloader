# Story 1.7 (HISTÓRICO) — Orchestrator: chunking + retry + CLI smoke (MVP GATE)

## User Story

**As a** usuário do data-downloader rodando via terminal Windows
**I want** um único comando `data-downloader download --symbol WDOJ26 --start 2026-03-01 --end 2026-03-30` que coordene tudo (DLL init → contrato vigente → chunking → download chunk-by-chunk → persistência → catálogo → cleanup), seja resumível se interromper, idempotente se re-rodar
**So that** eu tenho 30 dias de WDO baixados sem intervenção manual, prontos para query DuckDB — **e este comando funcionando = gate de Epic 1 = squad confiante para construir UI e features adicionais em cima de fundação sólida**.

## Acceptance Criteria

1. **AC1:** `src/data_downloader/orchestrator/orchestrator.py` expõe classe `DownloadOrchestrator(dll, catalog, writer)` com método `run(symbol, exchange, start_date, end_date) -> DownloadResult`.
2. **AC2:** Orchestrator usa Story 1.6 `vigent_contract` se `symbol` for root (ex: "WDO") — resolve para contract code (ex: "WDOJ26") cobrindo o range. Se range cruza vigência → erro `RangeRequiresMultipleContracts` (multi-contract = Epic 4; nesta story, requer-se symbol específico).
3. **AC3:** Chunking adaptativo: WDO = 5 dias por chunk (Nelo quirk empírico — limite de chunk size para WDO histórico); WIN = 1 dia; equity = 1 dia. Default = 1 dia se símbolo desconhecido.
4. **AC4:** Para cada chunk: registra job no catálogo (Story 1.5), chama `download_chunk` (Story 1.3), persiste via `ParquetWriter` (Story 1.4), atualiza catálogo, marca chunk como completo. Crash entre chunks deixa estado consistente (chunk não-completo é detectável e re-rodado).
5. **AC5:** Resume: se job_id passado E job existe E status != 'completed' → orchestrator chama `Catalog.resume_job(job_id)` e processa apenas chunks pendentes.
6. **AC6:** Idempotência (lei R5): se intervalo [start, end] inteiramente coberto por partições existentes (verificado via catálogo), orchestrator retorna `DownloadResult(status='cache_hit', trades_count=already_in_catalog, partitions=[...])` sem chamar DLL.
7. **AC7:** Retry policy:
   - Timeout default 1800s por chunk (Nelo quirk).
   - Quirk 99%: tolerar até 1800s sem progresso real após atingir 99% (não considerar timeout até total).
   - Em erro NL_*: retry com backoff exponencial (3 tentativas, base=30s); após 3 falhas, marca chunk como `failed` + log + segue próximo chunk (não para o job inteiro).
8. **AC8:** CLI `data-downloader download` (typer):
   - `--symbol` (required)
   - `--start YYYY-MM-DD` (required)
   - `--end YYYY-MM-DD` (required)
   - `--exchange F|B` (default F)
   - `--resume {job_id}` (opcional)
   - `--data-dir PATH` (default `./data`)
   - Output Rich: barra de progresso por chunk, summary final
9. **AC9:** Smoke test E2E (gated por env `PROFITDLL_KEY`): roda comando completo `download WDOJ26 --start 2026-03-01 --end 2026-03-30`. Verifica:
   - Comando termina sem exception não-tratada
   - `data/history/F/WDOJ26/2026/03.parquet` existe e é Parquet válido
   - Catálogo SQLite tem ≥1 entrada em `partitions` para WDOJ26
   - DuckDB lê todos os trades (`SELECT COUNT(*) FROM read_parquet('data/history/F/WDOJ26/2026/*.parquet')` > 0)
   - Re-rodar mesmo comando = no-op (saída "cache hit", nenhum chamada DLL)
   - Sol `*integrity-check --symbol WDOJ26` retorna clean (sem dups, sem gaps inesperados além de fim-de-semana)
10. **AC10:** Pyro registra baseline `bench_chunking` em `docs/perf/BASELINES.md`: tempo total do download de 30 dias WDOJ26, throughput trades/s, nº reconexões.

### Esqueleto do orchestrator

```python
# src/data_downloader/orchestrator/orchestrator.py
from dataclasses import dataclass, field
from datetime import date, datetime
from pathlib import Path
import structlog

from data_downloader.dll.wrapper import ProfitDLL
from data_downloader.storage.catalog import Catalog
from data_downloader.storage.parquet_writer import ParquetWriter
from data_downloader.orchestrator.chunker import chunk_range
from data_downloader.orchestrator.contracts import vigent_contract
from data_downloader.orchestrator.download_primitive import download_chunk
from data_downloader.orchestrator.retry import with_retry

log = structlog.get_logger(__name__)

@dataclass(frozen=True)
class DownloadResult:
    job_id: str
    symbol: str
    exchange: str
    requested_start: date
    requested_end: date
    actual_start: date | None
    actual_end: date | None
    trades_count: int
    chunks_completed: int
    chunks_failed: int
    partitions: list[Path]
    status: str  # 'completed' | 'partial' | 'failed' | 'cache_hit'
    duration_seconds: float

class DownloadOrchestrator:
    def __init__(self, dll: ProfitDLL, catalog: Catalog, writer: ParquetWriter):
        self.dll = dll
        self.catalog = catalog
        self.writer = writer

    def run(self, symbol: str, exchange: str, start: date, end: date,
            *, resume_job_id: str | None = None) -> DownloadResult:
        # 1. Resolver contract code (Story 1.6) se for root
        contract = self._resolve_contract(symbol, start, end, exchange)

        # 2. Cache hit?
        if self._fully_cached(contract, exchange, start, end):
            return self._cache_hit_result(contract, exchange, start, end)

        # 3. Resume ou novo job?
        job_id = resume_job_id or self.catalog.register_job(contract, exchange, start, end)
        pending = self._compute_pending(job_id, contract, exchange, start, end)

        # 4. Loop de chunks com retry
        results = []
        for chunk in pending:
            try:
                with_retry(
                    lambda: self._process_chunk(job_id, contract, exchange, chunk),
                    max_attempts=3,
                )
                results.append(('ok', chunk))
            except Exception as e:
                log.error("orchestrator.chunk_failed", chunk=chunk, error=str(e))
                self.catalog.register_gap(contract, exchange, chunk.start, chunk.end, reason='failed_chunk')
                results.append(('failed', chunk))

        return self._build_result(job_id, contract, exchange, start, end, results)
```
