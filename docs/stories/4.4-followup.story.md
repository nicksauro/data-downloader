# Story 4.4-followup — V1.1 polish

## Contexto

Story 4.4 (Auto-updater + packaging final V1 release) entregou pipeline
de build mock-first + UpdaterStub V1.0 + INSTALL.md + 2 WAIVERS:

1. `docs/qa/WAIVERS/4.4-vm-smoke-deferred-2026-05-04.md` (AC6 smoke VM
   limpa).
2. `docs/qa/WAIVERS/4.4-signing-deferred-2026-05-04.md` (AC5 code
   signing Caminho A).

Esta story-followup fecha **4 débitos consolidados** que requerem
humano + vendors externos para execução completa:

1. **Smoke VM Windows limpa** (AC6 da 4.4) — humano provisiona VM +
   roda smoke E2E + arquiva evidência.
2. **Code signing V1.1** (AC5 Caminho A) — aquisição EV cert + HSM +
   integração `signtool` no pipeline.
3. **Tufup auto-updater full** (ADR-017 §"Decisão final pendente") —
   key ceremony + tufup repository + UpdaterStub upgrade.
4. **Container Docker Windows CI** (ADR-009 Camada 4-5) — bit-exato
   verify-build em windows-2022 controlled.

Consolidação em 1 story-followup (não 4 separadas) porque os débitos
compartilham:

- Mesma audiência humano (devops + arquiteto).
- Mesmo gate "release V1.1 publishable".
- Mesmo deadline (V1.1 release window — Q3/2026 estimado).
- Sinergias técnicas (ex.: signing precisa estar pronto antes de tufup
  full porque TUF verification sem signing é teatro).

## User Story

**As a** humano operador (Gage role) responsável pela publicação V1.x
do data-downloader

**I want** executar smoke real em VM Windows limpa + adquirir EV cert
+ ativar tufup full + montar container CI

**So that** V1.1 release pode ser publicada com:
- Selo "Verified Publisher" no SmartScreen (sem warning).
- Auto-update automático com TUF signature verification + atomic apply
  + rollback resistance.
- Build determinístico bit-exato verificável em CI.
- Smoke E2E real validado em VM limpa para cada release V1.x.

## Acceptance Criteria

### Bloco A — VM smoke (AC6 da 4.4)

1. **AC1 — VM provisionada.**
   - Windows 10 build 19045+ ou Windows 11.
   - Sem Python instalado, sem ProfitDLL prévia, sem app data-downloader.
   - Snapshot tirado para reset rápido entre runs.

2. **AC2 — Pipeline release executado real.**
   - Humano roda `python scripts/build_release.py` (sem `--dry-run`)
     em ambiente dev (laptop ou CI).
   - PyInstaller produz `dist/data_downloader/` com tamanho 80-150 MB.
   - `dist/data-downloader-v1.0.0-win64.zip` criado deterministicamente.
   - `dist/build-manifest-v1.0.0.json` populado com SHA256 reais.
   - `python scripts/github_release.py --version 1.0.0` publica
     GitHub Release oficial (ou `--dry-run` para validar antes).

3. **AC3 — Smoke E2E na VM.**
   - Humano baixa zip da release page na VM.
   - Verify SHA256 contra manifest → match.
   - SmartScreen warning aparece → click-through "Mais informações" →
     "Executar mesmo assim" (Caminho B).
   - Setup `.env` com credenciais Nelogica reais.
   - `data_downloader.exe doctor` retorna OK (DLL + credenciais +
     conectividade).
   - Smoke download 1 dia: `data_downloader.exe download --symbol WDOJ26
     --start 2026-04-15 --end 2026-04-15` completa em <90s; partição
     Parquet criada com trades > 0.

4. **AC4 — Smoke update.**
   - Humano publica `v1.0.1-rc1` com bump trivial em pyproject.toml
     (ex.: typo fix em CHANGELOG).
   - Na VM, abre Settings → Updates → "Verificar atualizações".
   - UpdaterStub.check_for_updates() retorna info para v1.0.1-rc1.
   - Status label mostra "↑ Nova versão v1.0.1-rc1 disponível.".
   - Botão "Baixar manualmente" abre browser na release page.
   - Humano baixa novo zip, fecha app, substitui pasta, reabre →
     versão atualizada (`data_downloader.exe --version` confirma).

5. **AC5 — Evidência arquivada.**
   - Sanitizada (sem credenciais, sem hostname real, sem PII).
   - Salva em `docs/qa/SMOKE_EVIDENCE/4.4-{ISO}.md` com:
     - Screenshots (SmartScreen, app aberto, Settings Updates).
     - Logs sanitizados (`%APPDATA%\data_downloader\logs\app.log`).
     - SHA256 verificado.
     - Timing observado.

### Bloco B — Code signing V1.1 (AC5 da 4.4 Caminho A)

6. **AC6 — EV cert adquirido + HSM setup.**
   - Vendor: Sectigo OR DigiCert OR GlobalSign EV Code Signing.
   - Custo aprovado pelo usuário: ~$400/ano.
   - Lead time: 5-10 dias úteis (validação organização).
   - HSM: token físico (SafeNet eToken 5110) OR DigiCert KeyLocker
     cloud HSM. Preferência: KeyLocker (CI-friendly).

7. **AC7 — Signing integrado no pipeline.**
   - `.github/workflows/release.yml` adiciona step:
     ```yaml
     - name: Sign .exe
       env:
         SIGNING_CERT_BASE64: ${{ secrets.SIGNING_CERT }}
         KEYLOCKER_TOKEN: ${{ secrets.KEYLOCKER_TOKEN }}
       run: |
         smctl sign --keypair-alias data-downloader-key \
           --tool signtool \
           --input dist/data_downloader/data_downloader.exe \
           --tool-flags "/td sha256 /fd sha256 /tr http://timestamp.digicert.com /v"
     ```
   - `signtool verify /pa /v dist/data_downloader/data_downloader.exe`
     retorna `Successfully verified`.
   - Token escopado @devops only (GitHub Actions secrets locked).

8. **AC8 — Smoke pós-signing em VM limpa.**
   - Re-run smoke E2E (AC3) com binário assinado.
   - SmartScreen mostra **"Verified publisher: Squad data-downloader"**
     (após reputation building 24-48h).
   - `Get-AuthenticodeSignature` em PowerShell retorna `Status: Valid`.

9. **AC9 — ADR-016 atualizado.**
   - Status `accepted (deferred to V1 release)` →
     `accepted (active V1.1+)`.
   - Renewal reminder em release calendar (Gage `*release` checklist).

### Bloco C — Tufup auto-updater full (ADR-017 §"Decisão final")

10. **AC10 — Aria reabre ADR-017 + valida tufup vs alternativas.**
    - POC tufup em VM Windows (~2h Aria + Gage).
    - Comparar status PyUpdater fork (Opção B) + Velopack (Opção F)
      atualizado.
    - ADR-017 status `deferred to Epic 4` → `accepted — final tooling:
      tufup` (ou pivot documentado).

11. **AC11 — TUF key ceremony executada.**
    - Root key gerada offline + armazenada em cold storage (USB
      criptografado, hardware key, OR vault offline). Rotação: anos.
    - Targets / Snapshot / Timestamp keys em DigiCert KeyLocker cloud
      HSM (acesso restrito CI).
    - Documentação: `docs/release/TUF_KEY_CEREMONY.md`.

12. **AC12 — Tufup repository setup.**
    - Estrutura `repository/metadata/{root,targets,snapshot,timestamp}.json`
      versionada em GitHub Releases (asset OR `gh-pages` branch).
    - Pipeline release (`scripts/github_release.py`) atualiza targets
      + snapshot + timestamp em cada release.
    - Smoke: `tufup` Client em VM limpa consegue baixar update assinado.

13. **AC13 — UpdaterStub → UpdaterClient (full impl).**
    - `src/data_downloader/_updater/tufup_stub.py` substituído ou estendido
      por `client.py` (tufup Client wrapper).
    - `download_update()` baixa zip + verifica TUF signature (raise se
      falha) + retorna path local.
    - `apply_update()` extrai zip + atomic replace (`.tmp` + `os.replace`)
      + restart hint.
    - Rollback: se startup pós-update falha 3× consecutivas, restaura
      versão anterior (sentinel `~/.data_downloader/last_known_good.json`).
    - CLI `data-downloader self-update --check-only|--rollback` (ADR-017
      §"5. CLI integration").

14. **AC14 — Settings UI atualizada.**
    - `LBL_UPDATE_NOTICE_MANUAL_V1` removida (V1.1 auto-apply).
    - Botão "Aplicar agora" (após download completo) + restart hint.
    - Opt-out: checkbox "Verificar automaticamente ao iniciar"
      (default: ON em V1.1).

### Bloco D — Container Docker Windows CI (ADR-009 Camadas 4-5)

15. **AC15 — Container Docker Windows official.**
    - `Dockerfile.windows-build` baseado em `mcr.microsoft.com/windows/server:ltsc2022`.
    - Inclui Python 3.12.6 (versão exata) + uv + PyInstaller pinned.
    - Documentação: `docs/release/CONTAINER_BUILD.md`.

16. **AC16 — `.github/workflows/release.yml` em windows-2022 fixo.**
    - `runs-on: windows-2022` (não `windows-latest`).
    - Matrix `python-version: ['3.12.6']`.
    - Steps: checkout → install uv → uv sync --frozen → build_release.py
      → github_release.py.

17. **AC17 — verify-build (2× same SHA).**
    - CI roda `scripts/build_release.py` 2× no mesmo commit.
    - SHA256 do `data-downloader-v{version}-win64.zip` deve bater
      bit-exato. Falha pipeline se diferentes.
    - `tests/release/test_deterministic_build.py` (a criar) executa
      em CI matrix.
