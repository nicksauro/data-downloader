# Story 4.2 — Multi-asset support (WIN, equities)

## Contexto

Story criada por mini-council Aria + Sol + Felix em **COUNCIL-13** (Epic 4
prep). Sol é dona desta story por dois motivos: (1) o eixo crítico é
**`docs/storage/CONTRACTS.md`** + tabela `contracts` no catálogo SQLite —
território exclusivo de Sol; (2) WIN tem regra trimestral H/M/U/Z e equities
são "vigência infinita" — duas formas distintas de calendário que precisam
conviver no mesmo schema.

Hoje (Epic 1+2) o data-downloader baixa apenas WDO (mensal). O seed de
`CONTRACTS.md` já inclui WINH26/WINM26 + PETR4/VALE3 com
`validation_source=hypothesized` ou `manual` — falta o smoke real validando
que o pipeline (`vigent_contract → orchestrator → ProfitDLL → Parquet`)
funciona end-to-end para esses três asset shapes (futuros mensais, futuros
trimestrais, papéis à vista).

`chunker.py` (Story 1.7a) já contém chunking adaptativo com defaults
WDO=5d, WIN=1d, equity=1d. Esta story **não cria a regra**, apenas confirma
que ela está aplicada e cobre cada shape em testes.

Story 4.1 (broker) é pré-requisito porque a meta é "4 símbolos paralelos
heterogêneos" — se o broker não estiver pronto, esta story degrada para
"single-symbol cada asset", o que não fecha o gate G-Multi-Asset.

## User Story

**As a** Sol (custodian) garantindo longevidade do dado para projetos
downstream (backtest engine vai pedir WIN e equity, não só WDO)

**I want** que o pipeline atual de download/read/catalog funcione
identicamente para WINH26/WINM26 (trimestral, com rollover) e PETR4/VALE3
(equity sem vencimento), com chunking adaptativo correto e probe DLL
validado

**So that** a public_api expõe a mesma fronteira (`download`, `read`,
`read_continuous`, `vigent_contract`) sem cases especiais por asset class,
e o gate G-Multi-Asset (4 símbolos heterogêneos paralelos) pode ser
fechado pelo @qa.

## Acceptance Criteria

1. **AC1 — Chunking adaptativo confirmado por asset class.**
   `chunker.py` (Story 1.7a) já implementa `default_chunk_days(asset_class)`
   com {WDO: 5, WIN: 1, equity: 1}. Esta AC verifica:
   - Mapeamento `symbol_root → asset_class` está em
     `src/data_downloader/orchestrator/contracts.py` (ou módulo
     equivalente — Sol decide local).
   - Test `test_chunker_per_asset_class.py` cobre os 3 caminhos
     (WDOJ26 → 5d, WINH26 → 1d, PETR4 → 1d).
   - Override por flag CLI (`--chunk-days N`) continua funcionando.

2. **AC2 — Seed `CONTRACTS.md` expandida + carregada.**
   `docs/storage/CONTRACTS.md` §3 inclui:
   - **WIN:** H/M/U/Z para 2026 + 2027 (8 contratos mínimo). Vigência
     trimestral conforme regra §2.2 (validar via probe AC3).
   - **Equities:** PETR4 + VALE3 (já no seed) + ITUB4 + BBDC4 (4 mínimo).
     `vigent_from=1900-01-01`, `vigent_until=9999-12-31`,
     `validation_source=manual`.
   - `populate_contracts_from_seed` (Story 0.0) carrega tudo em
     `data/history/catalog.db.contracts` na inicialização.

3. **AC3 — Probe DLL valida WIN + equity (Story 1.6 *validate path).**
   - Probe contra `WINH26` na data `vigent_from + 5 dias úteis` retorna
     trades reais (não `NL_INVALID_TICKER`). `validation_source` atualiza
     para `dll_probe`.
   - Probe contra `PETR4` em qualquer dia útil recente retorna trades
     reais. `validation_source` atualiza para `dll_probe`.
   - Falha de probe (NL_*) marca `notes: "PROBE FAILED — investigar"`
     mas NÃO bloqueia merge — escalada para Sol + Nelo via issue.
   - Comando documentado: `data-downloader contracts validate --root WIN
     --year 2026` + `--symbol PETR4`.

4. **AC4 — `read_continuous` funciona para WIN (rollover trimestral).**
   `read_continuous("WIN", start, end)` concatena `WINH26 → WINM26 → WINU26
   → ...` segundo `vigent_from/vigent_until` em `contracts`. Test
   `test_read_continuous_win.py` verifica:
   - Concatenação ordenada por `timestamp_ns`.
   - Sem duplicatas no overlap (rollover policy COUNCIL-06 — Sol já
     definida; reusada).
   - Para equities (`read_continuous("PETR4", ...)`), retorna trades de
     PETR4 (uma única "vigência infinita" — caso degenerado mas correto).

5. **AC5 — Testes para WIN + PETR4 com fixtures sintéticos.**
   - `tests/unit/test_contracts_resolution.py` cobre:
     - `vigent_contract("WIN", date(2026,2,15))` → `"WINH26"`.
     - `vigent_contract("WIN", date(2026,3,20))` → `"WINM26"` (após
       vencimento de H).
     - `vigent_contract("PETR4", date(2026,4,1))` → `"PETR4"` (vigência
       infinita).
     - Overlap raises `AmbiguousContractError`.
   - `tests/integration/test_multi_asset_pipeline.py` simula download
     WINH26 + PETR4 com fixtures mock de DLL (sem broker — broker é
     Story 4.1; aqui foco é a camada de contracts/storage).
   - `tests/property/test_contract_calendar.py` (Hypothesis): para
     qualquer data válida, ou existe contrato vigente único, ou retorna
     None, ou raises `AmbiguousContractError` — nunca retorno errado
     silencioso.

6. **AC6 — Smoke gated com WINH26 + PETR4 (humano roda).**
   `tests/smoke/test_multi_asset_smoke.py` (gated `skipif(not env_has_dll())`):
   - Download 1 dia de `WINH26` (2026-02-15 ou data próxima útil) →
     completa sem erro; partition gravada; integrity check passa.
   - Download 1 dia de `PETR4` (qualquer dia útil recente) →
     completa sem erro; partition gravada; integrity check passa.
   - Evidência sanitizada salva em
     `docs/qa/SMOKE_EVIDENCE/4.2-{ISO}.md` (mesmo padrão Story 1.7b).
   - Humano roda; resultado anexado à story.

## Dev Notes

- **Sem novo schema:** Schema Parquet (Story 1.4) é genérico; campo
  `symbol` já carrega ticker arbitrário (WDOJ26, WINH26, PETR4). Zero
  migração de dado.
- **Sem nova fronteira pública:** `public_api` (download/read/read_continuous/
  vigent_contract) já é parametrizada por symbol — funciona para qualquer
  asset class supported pela DLL. Esta story é validação + seed expansion,
  não API change.
- **Risco DLL quirks WIN/equity:** ProfitDLL pode ter quirks específicos
  para WIN (ex: campos extras no callback, formato de tempo diferente).
  Nelo audita via probe (AC3). Quirks descobertos = atualização de
  `dll/wrapper.py` ou parser, escalado para issue separada se severo.
- **Equity ticker stability:** Equities podem trocar de ticker (split,
  fusão). Esta story NÃO trata isso (escopo Story futura). Documentar
  em `CONTRACTS.md` §6.2 (já existe).
