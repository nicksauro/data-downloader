## Contexto / Problema

A revisão consolidada Frente 1 v1.4.0 (2026-05-16) identificou que a tabela `_pending_commits` (criada por `catalog.py:166-175` no DDL v1.0.0) é **populada** durante `register_partition` (`catalog.py:863-987`) mas **NUNCA lida** após crash:

```bash
grep -rn "recover_pending\|pid_alive" src/ tests/  # 0 hits
```

`docs/storage/INTEGRITY.md §4.3` (linhas 272-303) prescreve `recover_pending_commits()` com validação SHA256/size + reativação ou quarentena. Doc existe; implementação não.

`cleanup_orphans` (`catalog.py:1502-1551`) só toca arquivos `.tmp.*` no FS — **não deleta entries de `_pending_commits`**. Após N crashes, a tabela acumula órfãos indefinidamente. INT-10 (`check_pending_commits_clean`) sempre verde ou sempre vermelho, sem semântica útil.

ADR-026 §2.2 prescreve método público `recover_pending_commits()` chamado em `__post_init__` ANTES de `cleanup_orphans` e `reconcile`, com PID liveness via `psutil.pid_exists + create_time` (defesa contra PID recycling), validação SHA256/size, e quarentena para arquivos corrompidos.

**Esta story é independente** — pode rodar primeiro na frente. Dependências downstream: 4.23 (inversão Fase 1/replace — consome o protocolo aqui implementado) e 4.24 (writer race — usa `_pending_commits` como advisory lock).

## Acceptance Criteria

- [x] **AC1 — Método público `Catalog.recover_pending_commits() → PendingRecoveryReport`** implementado em `src/data_downloader/storage/catalog.py`. Dataclass nova `PendingRecoveryReport(recovered: tuple[str, ...], cleaned: tuple[tuple[str, str], ...], quarantined: tuple[tuple[str, str], ...], skipped: tuple[tuple[str, str], ...])`. Imutável (frozen dataclass). Disponível no `__all__` do módulo.

- [x] **AC2 — PID liveness algorithm com defesa contra recycling.** Helper `_pid_alive(pid: int, started_at: datetime) → bool`:
  1. Se `pid is None` → False.
  2. Se `psutil` indisponível (ImportError) → fallback: `True` apenas se `started_at > now - 1h` (timestamp-only).
  3. `psutil.pid_exists(pid)` → se False → False.
  4. `psutil.Process(pid).create_time()` (UTC epoch) → comparar com `started_at`. Se `create_time > started_at` (PID reciclado, processo diferente) → False.
  5. Senão → True.
  Helper unit-testável isoladamente (mockando psutil).

- [x] **AC3 — Recovery resolution table** para cada row de `_pending_commits`:
  | Estado | Ação |
  |--------|------|
  | PID vivo | skip (`pid_alive`) |
  | PID morto + arquivo ausente em `data_dir/final_path` | DELETE pending; `cleaned` (motivo `no_file`) |
  | PID morto + arquivo presente + sha256+size match | UPSERT em `partitions` (via helper interno `_register_recovered_partition`) + DELETE pending; `recovered` |
  | PID morto + arquivo presente + sha256 OR size mismatch | mover para `data/_quarantine/{utc_iso_compact}/{partition_path}`; DELETE pending; `quarantined` (motivo `sha_mismatch` ou `size_mismatch`) |

- [x] **AC4 — Quarantine convention.** Helper `_quarantine_partition(path: Path) → Path` cria `data/_quarantine/{YYYYMMDDTHHMMSSZ}/{relative_partition_path}` preservando hierarquia interna. `os.replace` (atômico). Loga `catalog.recovery.quarantined` com path origem + destino. Se quarantine falhar (disco cheio, etc.), loga `catalog.recovery.quarantine_failed` e DELETE da pending row é abortada para retry no próximo boot.

- [x] **AC5 — Integração em `__post_init__`.** Chamar `self.recover_pending_commits()` ANTES de `self.cleanup_orphans()` e `self.reconcile()` em `catalog.py:586-611`. Wrapped em try/except `OSError|sqlite3.Error` com log `catalog.recover_pending.failed` (não levanta — boot sempre continua). Se `report` não-vazio (`recovered + cleaned + quarantined > 0`), log `catalog.recover_pending.startup` com counts.

- [x] **AC6 — Re-registro idempotente (`_register_recovered_partition`).** Helper interno lê metadata do parquet on-disk (`pyarrow.parquet.read_metadata`) para extrair `row_count`, `first_ts_ns`, `last_ts_ns`, `schema_version`. Resolve `PartitionKey` via `parse_partition_path` (`partition.py`). UPSERT em `partitions` usando o mesmo SQL de `register_partition` mas com checksum/size pegos da row de `_pending_commits` (eram `expected_*` e agora validados). Setta `written_at = recovery_now` e `job_id` da pending row.

- [x] **AC7 — CLI subcomando `data-downloader catalog recover-pending [--dry-run]`.** Adicionar ao `cli.py`. `--dry-run` chama método novo `dry_run_recovery_pending() → PendingRecoveryReport` (mesmo workflow mas sem mutações: pula DELETE/quarantine/UPSERT). Exit codes: `0` se `report.is_clean` (ou após sucesso); `2` se `quarantined != ()`; `3` em erro operacional. Print formatado com counts por categoria.

- [x] **AC8 — `cleanup_orphans` complementar.** Aumentar `catalog.py:1502-1551` para, ao final, executar:
  ```sql
  DELETE FROM _pending_commits
   WHERE pid IS NOT NULL
     AND started_at < datetime('now', '-1 day')
  ```
  Apenas como defense-in-depth (recovery já deveria ter pegado). Loga count deletado se >0. Não toca rows com `started_at` recente (poderia ser write ativo).

- [x] **AC9 — Testes unitários (mínimo 8).** `tests/unit/test_pending_commits_recovery.py` (12 testes entregues):
  1. `_pid_alive` com psutil mockado: pid_exists=True + create_time<started → True.
  2. `_pid_alive` com pid_exists=False → False.
  3. `_pid_alive` PID reciclado: pid_exists=True + create_time>started → False.
  4. `_pid_alive` fallback sem psutil + started_at recente → True.
  5. `recover_pending_commits` row com pid morto + arquivo ausente → `cleaned`.
  6. `recover_pending_commits` row com pid morto + arquivo+sha matches → `recovered` + partição em `partitions`.
  7. `recover_pending_commits` row com pid morto + arquivo presente + sha mismatch → `quarantined` + arquivo movido para `_quarantine/`.
  8. `recover_pending_commits` row com pid vivo → `skipped`.

- [x] **AC10 — Testes integration (mínimo 3).** `tests/integration/test_recovery_boot.py` (3 testes entregues):
  1. Setup: cria pending row + arquivo on-disk válido. Instancia novo `Catalog` (boot path). Verifica partição registrada + pending limpo.
  2. Setup: cria pending row + arquivo corrompido (sha diferente). Boot. Verifica quarantine criada + pending limpo.
  3. Smoke CLI: invoca `data-downloader catalog recover-pending --dry-run` via `subprocess` e valida exit code + output (counts).

- [x] **AC11 — INT-10 atualizado.** `docs/storage/INTEGRITY.md §1` linha 29 (INT-10) mantém-se mas adiciona-se §4.5 documentando o protocolo de recovery (PID liveness, quarantine, CLI). Atualizar referência da implementação (`recover_pending_commits` em vez de "doc-only").

**Riscos identificados:**
- Adicionar `psutil` se não estiver disponível pode quebrar builds frozen (PyInstaller). Validar antes via `python -c "import psutil"`.
- Performance no boot com >100 pending rows: SHA256 de cada arquivo. Mitigação: log INFO com tempo total + count; consideração futura de cache se necessário.
- Quarantine pode falhar se `data_dir` está em FS read-only ou disco cheio. Tratar como warning + retry no próximo boot (idempotente).
