**Status:** Done

**Owner:** @dev (Dex), RCA por @profitdll-specialist (Nelo) + Felix/Dex

## Contexto / Problema

O app GUI instalado (v1.1.0, `console=False`) **crashava ao tentar baixar histórico**. Logs reais (Pichau, 2026-05-12):

- `_internal/Erro.log` (ProfitDLL): `CreateDataLoader Failed | Access violation at ... Read of address FFFFFFFFFFFFFFFF` em `System.@IntfCopy ← BaseDataLoaderU.CreateBaseDataLoader ← InitializationManagerU.TInitializationManager.InitializeAfterLogin ← ConnectorThreadU.TConnectorThread.LoginResult`.
- `PostMessageError.log`: `WM_VALID_LICENSE` / `WM_REPLAY_SYNC_CLOCK` / `WM_LOGIN_RESULT` perdidos com `SysLastError=1444` (`ERROR_INVALID_THREAD_ID`) e `1400` (`ERROR_INVALID_WINDOW_HANDLE`).
- Windows Event Log: `data_downloader.exe` crash em `Qt6Core.dll 6.11.0.0`, exception `0xc0000409` (`__fastfail`/abort).

O **CLI** (`data_downloader-cli.exe download ...`) nunca foi afetado — faz 1 init de DLL por processo.

## Causa raiz (duas, ambas corrigidas)

1. **Duplo init da ProfitDLL no mesmo processo (Q08-E).** "Testar Conexão" fazia `with ProfitDLL() as dll: dll.initialize_market_only(...)` e chamava `finalize()` ao sair do `with`; "Baixar" criava **outra** instância e re-inicializava. A ProfitDLL Classic não é re-inicializável — `Finalize` não zera o estado interno; a `ConnectorThread` da 1ª sessão morre mas a 2ª init posta mensagens pra ela (`ERROR_INVALID_THREAD_ID`) e `CreateDataLoader` pega ponteiro de interface morto. **Bug latente desde v1.0.5** (quando "Testar Conexão" foi adicionado). Ver `docs/dll/QUIRKS.md` Q08-E e `docs/adr/ADR-022-single-session-sequential-policy.md`.
2. **Frozen windowed (`console=False`) → stdio inválido.** Em modo windowed o processo não tem console: `sys.stdout`/`sys.stderr` são `None` e os handles OS-level `STD_OUTPUT/ERROR_HANDLE` ficam inválidos. A ProfitDLL Delphi, em `InitializeAfterLogin`/`CreateDataLoader`, faz operações que dependem desses handles válidos → access violation. O CLI EXE (`console=True`) tem handles válidos. Ver `docs/dll/QUIRKS.md` Q-DRIFT-39 (novo).

## Acceptance Criteria

- [x] AC1 — A ProfitDLL é inicializada **no máximo uma vez por processo**: "Testar Conexão" e "Baixar" reusam a mesma instância (`data_downloader/dll/session.py::get_dll()`, lazy + thread-safe).
- [x] AC2 — `finalize()` da DLL só é chamado uma vez, no encerramento (`atexit` + `MainWindow.closeEvent` via `shutdown_dll()`); idempotente.

## Addendum — fix #21b (2026-05-12): modo de init Test Connection ↔ Download alinhado

O fix #21 (singleton) resolveu o **crash do processo** mas introduziu uma **falha funcional**: `get_dll()` reusa a instância no MODO em que foi inicializada (ignora `**init_kwargs` se a instância já existe), e os dois call sites passavam modos diferentes:

- `_build_real_dll` (download): `minimal_handshake` resolvido de `DATA_DOWNLOADER_DLL_MINIMAL_HANDSHAKE` → default `False` (modo completo, validado: smoke real WDOFUT 5d = 2.8M trades).
- `_TestConnectionWorker.run` ("Testar Conexão"): `minimal_handshake=True` hard-coded (espelho do probe Q-DRIFT-12).

Resultado: usuário clica "Testar Conexão" primeiro → DLL inicializada minimal → clica "Baixar" → `get_dll()` retorna a instância minimal (ignora o kwarg) → `download_chunk` registra `SetHistoryTradeCallbackV2`, trades chegam, mas a DLL crasha INTERNAMENTE ao traduzir (`ConnectorMarketDataLibraryU.TranslateTrade | Access violation ... PopulateTradeV0`, "SILENT MODE" — não mata o processo) → `status=failed trades=0`. Confirmado no repro `docs/qa/repro-q08e.log`.

**Fix #21b:** fonte única do modo de init — `data_downloader.dll.session.resolve_dll_init_mode() -> dict` lê `DATA_DOWNLOADER_DLL_MINIMAL_HANDSHAKE` e retorna `{"minimal_handshake": bool}`. Ambos os call sites (`_build_real_dll`, `_TestConnectionWorker.run`) chamam `get_dll(..., **resolve_dll_init_mode())` → mesmo modo garantido (por default `minimal_handshake=False` = modo completo, igual ao download; "Testar Conexão" só lê status/versão, então ter os callbacks de trade é inócuo). Defensivo: `get_dll()` guarda os `init_kwargs` da inicialização e, se uma chamada subsequente pedir um modo divergente (kwargs em `_MODE_AFFECTING_KWARGS`), loga `warning dll.session.mode_mismatch requested=... active=...` e reusa no modo ORIGINAL (NÃO re-inicializa — a DLL Classic não tolera).

Outros callers verificados: `cli.py:668`/`cli.py:1948` (`contracts validate-contract` / healthcheck) usam `with ProfitDLL()` one-shot em processo dedicado — fora de escopo (não compartilham o singleton). Nenhum outro caller de `get_dll()` em `src/`.

**File List (addendum):**
- `src/data_downloader/dll/session.py` — `resolve_dll_init_mode()`, `_DLL_INIT_KWARGS`, warning `dll.session.mode_mismatch`
- `src/data_downloader/public_api/download.py` — `_build_real_dll` usa `resolve_dll_init_mode()`
- `src/data_downloader/ui/screens/settings_screen.py` — `_TestConnectionWorker.run` usa `resolve_dll_init_mode()` (antes `minimal_handshake=True`)
- `tests/unit/test_dll_session.py` — cobertura `resolve_dll_init_mode` (env truthy/falsy/default) + warning de mismatch + `_DLL_INIT_KWARGS` lifecycle

**Validação #21b:** `pytest tests/unit/test_dll_session.py` 22 passed; `pytest tests/unit -k "dll or session or download"` 294 passed; ruff limpo nos arquivos tocados; imports OK. Repro ajustado (`docs/qa/repro-q08e-fix21b.log`): ambos os `get_dll` usam `resolve_dll_init_mode()` → mesma instância, ZERO `mode_mismatch`, **nenhum `Erro.log` novo da DLL** (não crashou), processo não morreu. (Download `status=failed` off-hours porque MARKET_DATA não conecta fora do pregão B3 — não é o crash do bug; o caminho de tradução de trades só reproduz em horário de pregão.)
