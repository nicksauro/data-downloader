## Contexto / Problema

A revisão consolidada Frente 1 v1.4.0 (2026-05-16) identificou dois races concorrentes no caminho de write/compact, ambos resolvíveis com a mesma técnica (claim atômico em tabela SQLite como advisory lock).

### P0-S3 — Writers concorrentes em mesma partição diária

ADR-025 §2.2 declara que o caminho diário (`{ex}/{sym}/{YYYY}/{MM}/{DD}.parquet`) é **write-once** (`parquet_writer.py:347`):

```python
is_daily = partition.day is not None
if not is_daily and path.exists():
    # caminho mensal legacy — read-merge-rewrite
    ...
else:
    # caminho diário (write-once) OU mensal novo
    table = deduped_new_table
```

Dois processos/threads escrevendo `2026/03/15.parquet` simultaneamente (cenário: UI rodando + CLI em background com mesmo símbolo) terão last-writer-wins via `os.replace` — trades do primeiro **somem sem detecção**. O `chunk_ledger` UPSERT (`record_chunk`) é defesa tardia: acontece DEPOIS do `register_partition`, então a inconsistência (arquivo do segundo + ledger pode estar inconsistente) só é detectável via reconcile.

**Solução:** após inversão do two-phase commit (Story 4.23), `_pending_commits` vira advisory lock natural. A Fase 1 acontece ANTES do `os.replace` e usa o claim WHERE-guarded já preparado em 4.23 AC4. Esta story implementa o retry policy do writer ao receber `ConcurrentWriterError`.

### P0-S4 — `maybe_compact_month` race cross-process

`Catalog.maybe_compact_month` (`catalog.py:1178-1338`) marca `compactions(symbol,year,month).started_at=now` via UPSERT:

```sql
INSERT INTO compactions(...) VALUES(..., started_at=now, completed_at=NULL, error=NULL)
ON CONFLICT(symbol, exchange, year, month) DO UPDATE SET
  started_at = excluded.started_at,   -- BUG: sobrescreve in-flight
  completed_at = NULL,
  error = NULL
```

(linhas 1232-1242)

Dois processos chamando `maybe_compact_month("WDOJ26", "F", 2026, 3)` simultaneamente: ambos UPSERT, ambos veem os mesmos `{DD}.parquet` no disco, ambos chamam `compact_month`. Resultado: dois `os.replace` no mesmo `{MM}.parquet` (segundo ganha); ambos chamam `DELETE` dos diários (segundo `Path.unlink()` falha com `FileNotFoundError` ou em Windows `PermissionError` se algum handle ainda aberto). Inconsistência catalog↔FS detectada posteriormente por reconcile, mas com janela de exposição alta.

**Solução (ADR-026 §2.4):** substituir o UPSERT por claim atômico com WHERE-guard idêntico ao usado em `_pending_commits` (4.23 AC4):

```sql
INSERT INTO compactions(...) VALUES(...)
ON CONFLICT(symbol, exchange, year, month) DO UPDATE SET
  started_at = excluded.started_at,
  completed_at = NULL,
  error = NULL
WHERE compactions.completed_at IS NOT NULL
   OR compactions.started_at < datetime('now', '-1 hour')
```

Após o INSERT, `SELECT started_at FROM compactions WHERE ...` e comparar com o `now` passado. Se != → outro processo claim'd dentro da janela de 1h. No-op, retorna `False`.

**Esta story depende de 4.22 (recovery) e idealmente de 4.23 (inversão).** A integração com o context manager `pending_commit` é onde o retry do writer fica natural. Pode rodar em paralelo a 4.23 nas partes que tocam `compactions` apenas.

## Acceptance Criteria

### Writer race (P0-S3)

- [x] **AC1 — `ConcurrentWriterError` exception class.** Adicionar em `src/data_downloader/public_api/exceptions.py`:
  ```python
  class ConcurrentWriterError(IntegrityError):
      """Outro processo está escrevendo a mesma partition_path.

      Levantada por Catalog.pending_commit() quando o claim em
      _pending_commits detecta pending row de PID diferente vivo
      e dentro do timeout de staleness (1h).
      """
  ```
  Exportada em `public_api/exceptions/__init__.py`. Documentar quando é levantada (referência a ADR-026 §2.3).

- [x] **AC2 — Claim refinado em `pending_commit.__enter__`** (Story 4.23 AC4 já estabelece a base; esta AC formaliza a detecção de colisão real). Após o INSERT WHERE-guarded:
  ```python
  row = conn.execute(
      "SELECT pid, started_at FROM _pending_commits WHERE partition_path = ?",
      (rel_path,),
  ).fetchone()
  if row is None:
      # Não deveria acontecer — INSERT garantido pelo upsert.
      raise IntegrityError(f"Unexpected: pending row missing for {rel_path}")
  if int(row["pid"]) != os.getpid():
      raise ConcurrentWriterError(
          f"Concurrent writer detected for {rel_path}",
          details={"holder_pid": int(row["pid"]), "holder_started_at": str(row["started_at"])},
      )
  ```

- [x] **AC3 — Retry policy em `ParquetWriter.write` quando `catalog is not None`.** Wrap do bloco `with catalog.pending_commit(...)` em loop com backoff:
  ```python
  _RETRY_DELAYS_MS = (100, 500, 2000)  # 3 attempts total
  last_exc: ConcurrentWriterError | None = None
  for attempt, delay_ms in enumerate(_RETRY_DELAYS_MS):
      try:
          with catalog.pending_commit(...) as handle:
              os.replace(tmp_path, path)
              _fsync_directory(path.parent)
              handle.complete(write_result)
          break  # sucesso
      except ConcurrentWriterError as exc:
          last_exc = exc
          if attempt == len(_RETRY_DELAYS_MS) - 1:
              raise  # último attempt — propaga
          time.sleep(delay_ms / 1000.0)
  ```
  Log INFO em cada retry (`parquet_writer.concurrent_writer.retry`).

- [x] **AC4 — Cleanup de tmp em retry exhaustion.** Quando todos os retries falham e `ConcurrentWriterError` é propagada, o `tmp_path` deve ser deletado (não tentar replace de novo) — o `except Exception` block existente em `parquet_writer.py:423-433` já cobre isso. Validar.

### Compact single-flight (P0-S4)

- [x] **AC5 — Reescrever claim em `maybe_compact_month`.** Em `catalog.py:1228-1242` substituir o `INSERT ... ON CONFLICT DO UPDATE SET started_at = excluded.started_at, completed_at=NULL, error=NULL` por versão WHERE-guarded:
  ```sql
  INSERT INTO compactions(symbol, exchange, year, month, started_at, completed_at, error)
  VALUES (?, ?, ?, ?, ?, NULL, NULL)
  ON CONFLICT(symbol, exchange, year, month) DO UPDATE SET
    started_at = excluded.started_at,
    completed_at = NULL,
    error = NULL
  WHERE compactions.completed_at IS NOT NULL
     OR compactions.started_at < datetime('now', '-1 hour')
  ```

- [x] **AC6 — Pós-INSERT verification.** Após o claim, SELECT a row e comparar `started_at` com o `now` original passado ao INSERT. Se diferente → outro processo claim'd. No-op, retorna `False` (sem chamar `compact_month` nem mutar `partitions`/`compactions` adicionalmente). Log INFO `catalog.maybe_compact_month.claim_lost` com `holder_started_at`.

- [x] **AC7 — Path completo com claim:**
  ```python
  def maybe_compact_month(self, symbol, exchange, year, month, *, dll_version=""):
      if not self.is_month_complete(...):
          return False
      now = _utcnow_iso()
      # Claim atômico (AC5 + AC6)
      with self._transaction():
          conn = self._conn_or_raise()
          conn.execute(<<SQL AC5>>, (symbol, exchange, year, month, now))
          row = conn.execute(
              "SELECT started_at FROM compactions WHERE symbol=? AND exchange=? AND year=? AND month=?",
              (symbol, exchange, year, month),
          ).fetchone()
          if row is None or str(row["started_at"]) != now:
              _LOG.info("catalog.maybe_compact_month.claim_lost", extra={...})
              return False
      # Claim conquistado — segue o pipeline existente (compact_month + UPSERT mensal + DELETE diários + UPDATE completed_at)
      try:
          result = compact_month(...)
          ...
      except Exception as exc:
          ...
  ```

- [x] **AC8 — `_recover_inflight_compactions` integra com staleness.** O método existente (`catalog.py:1641-1792`) processa rows com `completed_at IS NULL`. Após esta story, claims stale (>1h) são reclamáveis. Adicionar comentário no docstring referenciando ADR-026 §2.4. Não precisa mudar a lógica de recovery (que decide completar vs reverter baseado em `monthly_rows >= daily_total_rows`).

## Notas de execução

**Dependências de stories:**
- **4.22 (recovery)** DEVE estar `Done` antes — sem recovery, claims de PIDs mortos não são limpos.
- **4.23 (inversão)** DEVE estar `Done` antes — o claim em `_pending_commits` ANTES do `os.replace` é estabelecido em 4.23 AC4. Esta story formaliza a detecção de colisão (`ConcurrentWriterError`) e o retry policy.

**Riscos identificados:**
- Threading em testes integration é frágil em CI (especialmente Windows). Usar `threading.Event` para sincronizar starts e `threading.Barrier` para garantir paralelismo real. Timeout generoso (10s) para evitar flakes.
- Backoff fixo (100ms→500ms→2s) pode ser insuficiente para writes muito longos (mês completo de WDOFUT compactado: ~5-10s). Para `compact_month`, retry NÃO é apropriado — perde o claim e retorna `False` é o comportamento correto (outro processo está compactando, deixa fazer). Apenas o writer (chunk de 1 dia ~50-500ms) retry faz sentido.
- `ConcurrentWriterError` propagada ao orchestrator: marcar chunk como `failed` no `chunk_ledger` para retry futuro, NÃO abortar o download inteiro. Verificar callsite no orchestrator e adicionar tratamento explícito.
- Performance: o pós-INSERT SELECT adiciona ~1ms por write/compact. Negligível.

**Métricas de sucesso pós-deploy:**
- Zero ocorrências de "trades silenciosamente perdidos" em downloads paralelos (auditoria via `chunk_ledger.trades_count` vs `DuckDB COUNT(*)` em parquet glob).
- Zero ocorrências de `compact_month.daily_cleanup_failed` em logs por race (manter como warning, contar).
- `_pending_commits` permanece <=10 rows em qualquer momento (limpeza recovery + retry funcionando).
