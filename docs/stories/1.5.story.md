# Story 1.5 — Catálogo SQLite + checkpoint/resume

## User Story

**As a** orchestrator (Story 1.7) que pode crashar/cancelar a qualquer momento
**I want** um catálogo SQLite que registra cada job de download, cada partição escrita, gaps detectados; e um mecanismo de checkpoint/resume que permite re-iniciar de onde parou
**So that** download interrompido não exige re-baixar tudo, e qualquer agente pode responder "o que está baixado?" sem abrir Parquets.

---

## Acceptance Criteria

1. **AC1:** Sol formaliza schema do catálogo em `docs/storage/SCHEMA.md` (DDL referência em `agents/storage-engineer.md` `expertise.catalog_schema`).
2. **AC2:** `src/data_downloader/storage/catalog.py` expõe classe `Catalog(db_path: Path)` que cria/migra schema na inicialização (idempotente — re-init não duplica).
3. **AC3:** SQLite configurado com `PRAGMA journal_mode=WAL`, `synchronous=NORMAL`, `cache_size=-200000`, `temp_store=MEMORY`.
4. **AC4:** `Catalog` expõe métodos:
   - `register_job(symbol, exchange, requested_start, requested_end) -> str` (retorna job_id UUID, status='pending')
   - `update_job_progress(job_id, status, **fields)`
   - `register_partition(partition: WriteResult, job_id: str)` — atualiza `partitions` table
   - `register_gap(symbol, exchange, gap_start, gap_end, reason)`
   - `get_completed_partitions(symbol, exchange) -> list[Partition]`
   - `get_pending_chunks(job_id) -> list[ChunkRange]`
   - `reconcile() -> ReconcileReport` — varre filesystem vs catálogo, reporta drift
5. **AC5:** Job lifecycle: `pending` → `in_progress` → `completed` | `failed` | `partial`. Status escrito a cada chunk via WAL.
6. **AC6:** Operação registrar_partition é **idempotente** — mesma partition_path re-registrada atualiza row_count/timestamps; não duplica.
7. **AC7:** Cleanup de `.tmp.{uuid}` órfãos: na inicialização do `Catalog` (ou via método `cleanup_orphans()`), varre `data/` por arquivos `.tmp.*` antigos (>5min) e deleta com log.
8. **AC8:** Função `resume_job(job_id) -> ResumePlan` — dado job interrompido, calcula chunks já completos (presente em `partitions`) vs faltantes (em `requested` range mas não em `partitions`).
9. **AC9:** Reconciliação `reconcile()` detecta:
   - Arquivos Parquet em `data/history/` sem entrada em `partitions` (drift A → catálogo desatualizado)
   - Entradas em `partitions` sem arquivo correspondente (drift B → arquivo deletado)
   - Checksum em `partitions` divergente do arquivo atual (drift C → arquivo modificado externamente)
10. **AC10:** Property test: para qualquer sequência de `register_job → register_partition` repetida N vezes (mesmos args), o estado final do catálogo é equivalente a executar uma vez (idempotência R5).
11. **AC11:** Reconcile **automático** no startup do orchestrator (não só on-demand) — se diff detectado, log warn + auto-corrige drift A (catálogo desatualizado, arquivo presente), aborta com erro em drift B/C. Detalhes em `docs/storage/INTEGRITY.md` (Story 0.0).
12. **AC12:** WAL checkpoint (`PRAGMA wal_checkpoint(TRUNCATE)`) executado após cada `register_partition` — evita perda em crash entre write e checkpoint default. Trade-off documentado em INTEGRITY.md.
13. **AC13:** Two-phase commit emulado para `register_partition`:
    - Fase 1: `INSERT INTO partitions (..., status='pending_commit')`
    - Fase 2: arquivo `os.replace` + fsync(parent_dir) (Story 1.4 AC5)
    - Fase 3: `UPDATE partitions SET status='committed' WHERE partition_path=?`
    - Cleanup de `pending_commit` órfãos no startup (consistente com cleanup tmp órfãos AC7)
    - Detalhes em `docs/storage/INTEGRITY.md` (Story 0.0).

## Dev Notes

### Referências
- ADR-002 (SQLite WAL config)
- `agents/storage-engineer.md` — `expertise.catalog_schema` (DDL canônico)
- Lei R5 (idempotência), R6 (catálogo é fonte única)

### Constraints CRÍTICAS
- **WAL mode** habilitado para concorrência leitor/escritor.
- **Transações curtas** — não segurar lock por > 100ms.
- **Idempotência** — register_* é UPSERT (`INSERT ... ON CONFLICT DO UPDATE`).
- **Cleanup tmp** evita .tmp.{uuid} órfão se crash entre write tmp e replace (Story 1.4 escreve atomicamente, mas crash exato no meio gera órfão).

### Schema v1.0.0 (referência — Sol formaliza)

```sql
CREATE TABLE IF NOT EXISTS _schema_meta (
  version TEXT PRIMARY KEY,
  applied_at TIMESTAMP NOT NULL
);

CREATE TABLE IF NOT EXISTS downloads (
  job_id TEXT PRIMARY KEY,
  symbol TEXT NOT NULL,
  exchange TEXT NOT NULL,
  requested_start TIMESTAMP NOT NULL,
  requested_end TIMESTAMP NOT NULL,
  actual_start TIMESTAMP,
  actual_end TIMESTAMP,
  status TEXT NOT NULL,
  trades_count INTEGER,
  started_at TIMESTAMP,
  completed_at TIMESTAMP,
  error TEXT,
  dll_version TEXT
);

CREATE TABLE IF NOT EXISTS partitions (
  partition_path TEXT PRIMARY KEY,
  symbol TEXT NOT NULL,
  exchange TEXT NOT NULL,
  year INTEGER NOT NULL,
  month INTEGER NOT NULL,
  row_count INTEGER NOT NULL,
  first_ts_ns INTEGER NOT NULL,
  last_ts_ns INTEGER NOT NULL,
  schema_version TEXT NOT NULL,
  checksum_sha256 TEXT NOT NULL,
  file_size_bytes INTEGER NOT NULL,
  written_at TIMESTAMP NOT NULL,
  job_id TEXT
);

CREATE TABLE IF NOT EXISTS gaps (
  symbol TEXT NOT NULL,
  exchange TEXT NOT NULL,
  gap_start TIMESTAMP NOT NULL,
  gap_end TIMESTAMP NOT NULL,
  reason TEXT,
  detected_at TIMESTAMP NOT NULL,
  PRIMARY KEY (symbol, gap_start)
);

CREATE TABLE IF NOT EXISTS contracts (
  symbol_root TEXT NOT NULL,
  contract_code TEXT NOT NULL,
  vigent_from TIMESTAMP NOT NULL,
  vigent_until TIMESTAMP NOT NULL,
  validated_at TIMESTAMP NOT NULL,
  validation_source TEXT NOT NULL,
  PRIMARY KEY (symbol_root, contract_code)
);

CREATE TABLE IF NOT EXISTS _checksum_cache (
  partition_path TEXT PRIMARY KEY,
  mtime_ns INTEGER NOT NULL,
  size INTEGER NOT NULL,
  sha256 TEXT NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_partitions_symbol ON partitions(symbol, year, month);
CREATE INDEX IF NOT EXISTS idx_gaps_symbol ON gaps(symbol);
```
