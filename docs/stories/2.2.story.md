# Story 2.2 — Perf Write Optimization (vectorize ParquetWriter)

## User Story

**As a** Pyro (perf-engineer) responsável por throughput sustentado de escrita Parquet em produção
**I want** vectorizar o hot path interno do `ParquetWriter` (`_trades_to_table`, `validate_record`, `dedup`, `_sha256_file`) usando `pa.compute` + `pa.array` direto + `hashlib` streaming
**So that** o gap de **-72%** medido em Story 1.4.5 / re-confirmado em Story 1.8 (production writer 27_638 trades/s vs target V1 100k/s) seja eliminado, atingindo **>= 100k trades/s sustained** sem alterar contrato externo nem schema canônico Parquet.

**NOTA:** Story criada por mini-council Pyro+Sol+Aria em **COUNCIL-10**. Refatoração interna ao módulo `data_downloader.storage.parquet_writer` — sem mudança de fronteira (`public_api`/`SCHEMA.md` intactos). Aria já endossou via COUNCIL-02 sign-off recomendação 4.

## Acceptance Criteria

1. **AC1 — Throughput target atingido:**
   `bench_parquet_write` (production writer canônico — snappy + row_group=100k + validate + dedup + fsync + sha256) atinge **trades/s p50 >= 100_000** (gap fechado vs baseline 27_638). `bench_parquet_write` JSON registra git_sha pós-vectorização e diff vs baseline `1.1.0-mock`.

2. **AC2 — Vectorização `_trades_to_table`:**
   Conversão dict→`pa.Table` usa `pa.array` direto a partir de columns dict pré-acumuladas (1 array por coluna, sem loop Python por trade). Testes unitários comparam saída vectorizada vs saída loop Python (path antigo mantido como referência via flag `_use_legacy=True` durante transição).

3. **AC3 — Vectorização `validate_record`:**
   Validação roda via `pa.compute.greater_than`/`less_than`/`is_in`/`and_kleene` sobre coluna inteira (boolean masks). Trades inválidos são separados em `pa.Table` separado e registrados via `metrics.invalid_count`. Comportamento equivalente ao path loop (mesmas regras: price>0, qty>0, exchange in {F,B}, ts_ns positivo, etc.).

4. **AC4 — Vectorização `dedup`:**
   Dedup usa `pa.compute.unique` sobre array de chave canônica computada como hash int64 combinado **OU** `duckdb.sql("SELECT DISTINCT ...")` — **medir as duas opções e escolher pareto-otimal**. Resultado preserva INV-2: `dedup(L ++ L) == dedup(L)`.

5. **AC5 — SHA256 streaming:**
   `_sha256_file` consome arquivo via `iter(lambda: f.read(8192), b'')` em vez de carregar arquivo inteiro. RSS peak para arquivos > 100MB reduzido (medido em `bench_parquet_write` peak_rss_delta). Comportamento idêntico (mesmo hash byte-a-byte).

6. **AC6 — Property tests Hypothesis (Aria recomendação 7 COUNCIL-02):**
   Suite de property tests em `tests/property/test_parquet_writer_vectorized.py` cobre:
   - INV-2: `dedup(L ++ L) == dedup(L)` em paths vectorizado e legacy.
   - INV-3: read-after-write consistency (`read(write(L)) == sorted_dedup(L)` — INV-7).
   - Equivalência: validate_record vectorizado vs loop produzem mesma classificação (válido/inválido) para cada trade gerado por Hypothesis.

7. **AC7 — Regression check em todos os benchs:**
   Suite completa (bench_parquet_write, bench_parquet_read, bench_dedup, bench_callback_to_disk, bench_chunking, bench_multi_symbol) re-rodada pós-vectorização. **Nenhuma regressão > 10%** (default budget) em nenhum bench. Melhoras documentadas em `BASELINES.md` v1.2.0-vectorized.

8. **AC8 — Sem mudança de fronteira:**
   Diff de `src/data_downloader/public_api/` = vazio. Diff de `docs/storage/SCHEMA.md` = vazio. Aria revisa via `*review-design` para confirmar sem ADR amendment necessário.
