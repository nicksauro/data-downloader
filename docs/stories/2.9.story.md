# Story 2.9 — Logging strategy ADR-010 implementada

## User Story

**As a** Aria (autora ADR-010) e Quinn (que precisa logs auditáveis), e operadores (que precisam logs JSON parseáveis e sem secrets vazados)
**I want** **implementar formalmente ADR-010** (`structlog` + contextvars `correlation_id`/`job_id`/`chunk_id`/`symbol` + redaction de secrets `NL_USERNAME/NL_PASSWORD/NL_KEY` + format JSON estruturado), preenchendo gaps onde a implementação atual cobre parcialmente (estado atual: `correlation_id` aparece em alguns pontos mas não consistente; redaction não tem cobertura formal; format JSON não é homogêneo)
**So that** logs de produção sejam (a) auditáveis (todo evento traceable via `correlation_id`), (b) seguros (zero credencial em log — INV-credenciais), (c) machine-parseable (JSON estruturado consumível por SRE/Grafana/CloudWatch), preparando ground para Epic 3 UI live log view (Felix + Uma) e release V1 forense.

**NOTA:** Esta story **complementa** Story 2.7 (hot path tuning). 2.7 remove structlog do hot path; 2.9 garante que **fora do hot path**, structlog é usado conforme ADR-010 (contextvars + redaction + JSON). Implementação atual (Story 1.2/1.7a) usa structlog mas NÃO sistematicamente contextvars, NÃO tem redaction processor, e NÃO tem JSON renderer fixo.

## Acceptance Criteria

1. **AC1 — Pipeline structlog formal:** Existe `src/data_downloader/observability/logging_config.py` com função `configure_logging(level: str, json_output: bool, redact: bool) -> None` que registra:
   - `structlog.contextvars.merge_contextvars` (incluir contextvars em todo log)
   - `structlog.processors.add_log_level`
   - `structlog.processors.TimeStamper(fmt='iso', utc=True)`
   - `structlog.processors.StackInfoRenderer`
   - `structlog.processors.format_exc_info`
   - Custom processor `_redact_secrets` (AC3)
   - Renderer final: `JSONRenderer` se `json_output=True`, `ConsoleRenderer` se `False` (dev mode).
   Chamada única no boot do CLI (`cli.py` `main()`) e no boot do public_api (`download_handle.start()`).

2. **AC2 — Contextvars preenchidos em todos os pontos:** Auditoria + correção: TODOS os entry points do orchestrator preenchem contextvars:
   - `OrchestratorThread.run` → `bind_contextvars(job_id=..., correlation_id=...)`
   - `complete_chunk` → `bind_contextvars(chunk_id=..., symbol=..., exchange=...)`
   - `IngestorThread.run` → herda contextvars do parent thread (estratégia documentada — `contextvars.copy_context()` ou bind explícito).
   - Limpeza: `clear_contextvars()` no exit (try/finally).
   - Test: invocar 2 jobs concorrentes → logs de cada job têm `job_id` correto, sem cross-contamination.

3. **AC3 — Redaction de secrets:** Processor `_redact_secrets` substitui valores de keys conhecidas por `***REDACTED***`:
   - Keys: `nl_password`, `nl_username`, `nl_key`, `password`, `secret`, `token`, `api_key`, `auth`.
   - Heurística: case-insensitive match no key name; valor substituído mas key preservado (preserva schema).
   - Recursive: inspeciona dicts aninhados.
   - Test: log com payload `{user: 'demo', nl_password: 'sec', nested: {nl_key: 'xyz'}}` → output JSON com `nl_password: '***REDACTED***'` + `nested.nl_key: '***REDACTED***'` + `user: 'demo'` (não tocado).
   - Performance: < 50µs por log call (não está em hot path mas deve ser razoável).

4. **AC4 — JSON renderer canônico:** Em `json_output=True` (default em production), cada log entry é uma linha JSON com:
   - Campos canônicos: `timestamp`, `level`, `event` (mensagem), `logger`, `correlation_id`, `job_id`, `chunk_id`, `symbol`, `exchange` (quando aplicáveis via contextvars).
   - Campos extras: arbitrários do call site (ex: `trades_count=1234`, `duration_ms=567`).
   - Campos especiais: `stack` (se exception), `exc_info` (formatted).
   - Test: parse cada linha de output com `json.loads`, schema validation via JSONSchema (NEW: `docs/dev/LOG_SCHEMA.json`).

5. **AC5 — CLI flag + env var:** CLI `data-downloader --log-format json|console --log-level INFO|DEBUG ...`. Env vars: `DATA_DOWNLOADER_LOG_FORMAT`, `DATA_DOWNLOADER_LOG_LEVEL`. Default: `json` em production, `console` em dev (heurística: TTY → console, não-TTY → json). Override sempre vence. Microcopy PT-BR validada (Uma handoff via review — strings de error reuse MICROCOPY_CATALOG.md).

6. **AC6 — Migration cirúrgica de call sites existentes:** Audit grep `structlog.get_logger()` no codebase atual; cada call site verifica:
   - Bind contextvars antes de call (correlation_id no mínimo).
   - Não passa secrets como field (cobre via redaction processor mas defesa em profundidade).
   - Eventos seguem naming convention `{component}.{action}` (ex: `orchestrator.chunk_started`, NÃO `Started chunk!`).
   Refactor mínimo onde necessário; documentar no Change Log da story.

7. **AC7 — Tests + property:**
   - **Unit:** Cada processor (timestamp, level, redact, JSON render) testado isoladamente.
   - **Integration:** End-to-end log emission (CLI command → captured stdout → parse JSON → assert fields).
   - **Property test (Hypothesis):** secret redaction — para qualquer dict com key matching pattern, valor é redacted; outras keys preservadas. 100 examples.
   - **Cross-thread test:** 2 OrchestratorThread concorrentes → contextvars isolados (job_id distinto em cada log).

8. **AC8 — Documentação ops:** `docs/dev/LOGGING.md` (NEW) com:
   - Tabela de campos canônicos (correlation_id, job_id, chunk_id, etc).
   - Como configurar (CLI flag + env var).
   - Exemplos de log linha (JSON + console mode).
   - Schema JSONSchema para validation.
   - Lista de secrets redacted.
   - Referência a ADR-010 + R21.
