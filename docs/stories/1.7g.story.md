# Story 1.7g — P0 release-blockers — schema integrity + volume completude

## Contexto

Após COUNCIL-36 (Pax/@po), dois bloqueadores P0 inegociáveis foram
identificados que **impedem o release v1**:

### Bloqueador 1 — Silent column drop é BUG, não trade-off

COUNCIL-32 (Nelo) descobriu que o schema parquet v1.0.0
(`src/data_downloader/storage/schema.py`) **descarta silenciosamente**
três colunas que o resolver popula corretamente em runtime:

- `buy_agent_name` (ex: "XP Investimentos") — populado por `AgentResolver.resolve()`
- `sell_agent_name` (ex: "BTG Pactual") — idem
- `trade_type_name` (ex: "AgressionBuy") — derivável do enum `TTradeType`

O comentário em `download_primitive.py` L153-155 admite explicitamente:
> "Campos NÃO fazem parte do schema Parquet v1.0.0 (Sol authority — adição requer bump SCHEMA_VERSION); writer atual descarta se não encontrar coluna."

**Decisão Pax (autoridade ownership):** descarte silencioso de colunas é
**BUG**, não trade-off aceitável. Schema bump v1.0.0 → v1.1.0 é
release-blocker. Adicionalmente, o writer deve **falhar loudly** ao
receber colunas não mapeadas (defense-in-depth contra futuras regressões
deste tipo).

### Bloqueador 2 — Volume capturado parece ~50% abaixo do esperado

Smoke real 1.7d capturou 603.770 trades em 4 dias calendário (= 2 dias
úteis: seg 04/05 + ter 05/05 parcial; 01/05 = feriado, 02-03/05 = fds).

Dia útil completo (04/05, 9h25m de pregão) → 307.010 trades.

Usuário (autoridade ownership) afirma que **1 dia útil WDOFUT = 600-700k
trades**. Pro-rated full day no smoke = ~326k. **Gap ≈ 50%**.

Isso implica perda silenciosa em algum ponto da cadeia
`subscribe_ticker → callback V2 → IngestorThread → TranslateTrade → result.trades → parquet`.

Hipóteses ranqueadas (COUNCIL-36 §1):
- H1 (30%) — `subscribe_ticker` em ordem ruim vs `GetHistoryTrades`
- H2 (30%) — LAST_PACKET prematuro (DLL declara fim antes do tempo)
- H3 (20%) — TranslateTrade descartando trades reais como sentinelas Q-DRIFT-34
- H4 (15%) — janela 5 dias é cap silencioso da DLL
- H5 (5%) — buffer overflow callback V2 em alta liquidez

**Decisão Pax:** investigação volume é P0. Não shipa v1 sem confirmar
captura ≥500k trades em 1 dia útil completo de WDOFUT.

### Bloqueadores complementares (P0 docs)

- **SCHEMA.md L33** (Sol) — comentário trade_type errado: dizia
  "1=auction, 2=normal, 3=cross"; correto é `1=CrossTrade`,
  `2=AgressionBuy`, `3=AgressionSell`, `4=Auction` (Nelo COUNCIL-32 §3.1).
  Tabela completa do enum `TTradeType` (14 valores) deve ser adicionada.
- **wrapper.py L731-745** — comentário Q-DRIFT-35 errado: dizia
  "`-2147483636 == 0x80000004 reinterpretado`"; correto é
  `-2147483636 == NL_NOT_FOUND == 0x8000000C` (semântica intencional da
  DLL, não corrupção ABI — Nelo COUNCIL-32 §2.2).

## User Story

**As a** dono do produto data-downloader (Pax via Pichau)
**I want** garantir que o parquet entregue ao analista contém TODAS as
colunas resolvíveis pela DLL (sem descartes silenciosos) E que o volume
de trades capturados reflete a realidade do mercado (≥500k/dia útil
para WDOFUT)
**So that** análises downstream (microestrutura, agressões por corretora,
volume por tipo de trade) sejam confiáveis e o release v1 possa shipar
com integridade de dados garantida.

## Acceptance Criteria

1. **AC1 — Schema parquet v1.1.0 com colunas de nomes (NOT NULL com fallback string)**.
   Em `src/data_downloader/storage/schema.py`:
   - `SCHEMA_VERSION = "1.1.0"` (bump de "1.0.0").
   - Adicionar 3 campos `pa.field(..., pa.string(), nullable=False)`:
     - `buy_agent_name`
     - `sell_agent_name`
     - `trade_type_name`
   - Fallbacks determinísticos quando resolver retorna None ou ID
     desconhecido:
     - `buy_agent_name` / `sell_agent_name`: `f"Agent#{id}"` (mesmo
       padrão do `AgentResolver.resolve()` para NL_NOT_FOUND).
     - `trade_type_name`: lookup do enum `TTradeType` (14 valores), com
       fallback `f"TradeType#{n}"` se valor estiver fora do range 0-13.
   - Atualizar `validate_record()` (INT-3/4/5) para aceitar/exigir os
     novos campos.

2. **AC2 — Writer FALHA LOUDLY se schema receber colunas não mapeadas (no silent drop)**.
   Em `src/data_downloader/storage/writer.py` (ou módulo equivalente):
   - Antes de gravar batch, validar `set(record_keys) ⊆ set(schema_columns)`.
   - Se houver chave em `record_keys` que não esteja em `schema_columns`,
     **raise `SchemaIntegrityError`** (nova exception class) com mensagem
     clara listando as colunas órfãs.
   - **NÃO ignorar silenciosamente.** O comportamento atual ("writer
     descarta se não encontrar coluna") é o BUG que esta AC corrige.
   - Adicionar teste unit que valida o raise: passar dict com chave
     `unknown_field=foo` deve quebrar com `SchemaIntegrityError`.

3. **AC3 — Smoke real WDOFUT 1 dia útil retorna ≥500k trades (validação volume)**.
   Quinn (@qa) executa smoke real standalone (mesmo padrão 1.7d) com:
   - Símbolo: WDOFUT/F
   - Janela: 1 dia útil completo (`now - 1 dia útil 00:00 BRT` → `now - 1 dia útil 23:59 BRT`).
   - Exemplo: se hoje (terça) for útil, janela = segunda 00:00 → segunda 23:59 BRT.
   - **Critério PASS**: `trades_count >= 500_000`.
   - **Critério FAIL**: `trades_count < 500_000` → escalar para
     mini-council Aria + Nelo + Pax (não tentar resolver autonomamente).
   - Persistir parquet ad-hoc em `data/scratch/smoke-1.7g-{uuid}/wdofut.parquet`
     para análise DuckDB.

4. **AC4 — Distribuição diária no parquet ≥500k para cada dia útil dentro da janela**.
   Smoke adicional com janela 5 dias (mesmo padrão 1.7d) — para CADA dia
   útil presente no parquet:
   - `SELECT date(timestamp_ns/1e9) AS d, count(*) AS n FROM wdofut.parquet GROUP BY d ORDER BY d`
   - Cada dia útil completo deve ter `n >= 500_000`.
   - Dias parciais (primeiro/último da janela) ignorados — apenas dias
     com `min(timestamp_ns)` antes de 09:30 BRT e `max(timestamp_ns)`
     depois de 17:30 BRT contam como "completos".
   - Se algum dia útil completo tiver `n < 500_000` → root-cause volume
     gap (H1/H2/H3/H4/H5) precisa ser identificado e fixado nesta story
     ou escalado para mini-council.

5. **AC5 — SCHEMA.md tabela TTradeType de 14 valores**.
   Em `docs/SCHEMA.md` L33 (e onde mais aparecer):
   - Substituir comentário errado ("1=auction, 2=normal, 3=cross") pela
     tabela canônica do manual `LegacyProfitDataTypesU.pas` L33-46:

     | ID | Nome | Notas |
     |----|------|-------|
     | 0 | ttZero | reserved/sentinel |
     | 1 | ttCrossTrade | cross trade |
     | 2 | ttAgressionBuy | comprador agride |
     | 3 | ttAgressionSell | vendedor agride |
     | 4 | ttAuction | leilão |
     | 5 | ttSurveillance | vigilância |
     | 6 | ttExpit | expit |
     | 7 | ttOptionsExercise | exercício de opções |
     | 8 | ttOverTheCounter | OTC |
     | 9 | ttDerivativeTerm | termo derivativos |
     | 10 | ttIndex | índice |
     | 11 | ttBTC | BTC |
     | 12 | ttOnBehalf | on behalf |
     | 13 | ttRLP | retail liquidity provider |

   - Adicionar referência cruzada: `Veja src/data_downloader/storage/schema.py::TRADE_TYPE_NAMES (mapping canônico)`.

6. **AC6 — Comentário Q-DRIFT-35 wrapper.py corrigido (NL_NOT_FOUND)**.
   Em `src/data_downloader/dll/wrapper.py` L731-745 (ou onde estiver o
   comentário Q-DRIFT-35):
   - Substituir `-2147483636 (== 0x80000004 reinterpretado como c_int signed)` por
     `-2147483636 (== NL_NOT_FOUND == 0x8000000C — código semântico da DLL para "agent_id desconhecido", NÃO corrupção ABI)`.
   - Manter o fix de signatures (`argtypes`/`restype` explícitos para
     `GetAgentNameLength`/`GetAgentName` em path `minimal_handshake`) —
     continua sendo boa prática ctypes em x64. Apenas corrigir o
     **diagnóstico textual** do comentário.
   - Adicionar nota: `Veja COUNCIL-32 §2.2 e error_taxonomy.py::_NL_NOT_FOUND`.

## Risks

| # | Risco | Mitigação |
|---|-------|-----------|
| R1 | AC1 quebra readers existentes do parquet v1.0.0 | Migration helper em `storage/schema.py` (read v1.0.0 → cast v1.1.0 com `Agent#{id}` para nomes ausentes). Não bloqueia release v1 do downloader. |
| R2 | AC3 (volume) confirma gap real grande → trabalho de 1.7g vira escopo aberto | Escalar para mini-council Aria + Nelo + Pax; AC3 fail é gate de escalação, não tentar resolver dentro da story. |

| R4 | trade_type_name fallback `TradeType#{n}` não é desejado em produção | Validar com Pax via comment review do PR antes de merge; alternativa é raise se valor fora de 0-13. |
