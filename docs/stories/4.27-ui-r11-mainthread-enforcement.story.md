## Contexto / Problema

Revisão consolidada do squad (BIG COUNCIL Frente 4, 2026-05-16) identificou
4 violações P0 de R11 ("MainThread Qt < 16 ms") nos relatórios da frente
UI/UX. O P0-U1 já foi resolvido em Story 4.31 AC3 (`MainWindow.__init__`
agora chama `QApplication.instance().installEventFilter(self)` — PR #1
aguardando merge). Esta Story 4.27 fecha os **3 P0s restantes** + um conjunto
de P1s relacionados ao mesmo princípio.

A decisão arquitetural entre `QTimer.singleShot(0, slot)` (Defer) e
QObject+QThread+signal (Worker) está formalizada em **ADR-029 — UI Threading
Pattern: Defer-vs-Worker**. Regra resumida: **qualquer operação que toque
I/O (disco/SQLite/rede/DLL) ou cresça com o tamanho do dado do usuário
DEVE rodar em Worker**. Defer só é aceitável para custo sabidamente bounded
em memória (<1 ms).

### P0s endereçados nesta story

- **P0-U2 — `Catalog(...)` + `list_jobs` + `resume_job` em `DownloadScreen.__init__`**
  (`src/data_downloader/ui/screens/download_screen.py:218-223, 513-554`):
  SQLite I/O síncrono no MainThread → boot freeze 5-200 ms em catalog
  grande. `_check_for_interrupted_download` abre `Catalog(...)`, chama
  `list_jobs(statuses=("in_progress","partial"), limit=1)` e
  `resume_job(job.job_id)` antes do primeiro paint. Decisão (ADR-029):
  **mover para `CatalogAdapter` (já em QThread)** + novo slot
  `check_interrupted_jobs(data_dir)` + signal `interrupted_job_found(payload)`.
  Banner aparece após paint inicial (50-150 ms de latência aceitável).

- **P0-U3 — `_refresh_storage_status` em `SettingsScreen.__init__`**
  (`src/data_downloader/ui/screens/settings_screen.py:858, 875-907`):
  `shutil.disk_usage` + `sqlite3.connect` + `SELECT COUNT(*) FROM partitions`
  síncronos no MainThread → boot freeze 5-300 ms. Decisão (ADR-029):
  **Worker dedicado `_StorageStatusWorker`** (padrão consistente com
  `_TestConnectionWorker`/`_IntegrityWorker`/`_ReconcileWorker` já existentes
  no mesmo arquivo). Labels (`_free_space_label`, `_catalog_status_label`)
  ficam com placeholder "—" até worker responder.

- **P0-U4 — `data_dir.rglob("*.parquet")` em `StorageIndicator`**
  (`src/data_downloader/ui/widgets/storage_indicator.py:137-159, 290-338`):
  scan recursivo de filesystem no MainThread, disparado a cada 30 s
  + após cada `partition_registered`. Em 7 anos × N símbolos (50 k+
  arquivos) → freeze 500 ms - 2 s recorrente. Decisão (ADR-029):
  **dois passos complementares** —
  - (a) trocar `_parquets_used_gb(data_dir)` (rglob+stat) por
    `SELECT COALESCE(SUM(file_size_bytes), 0) FROM partitions`
    no catálogo (O(N) sobre N=partitions << N=arquivos, tipicamente <1000);
    fallback para `rglob` apenas quando catalog ausente.
  - (b) executar a query em **`StorageIndicatorWorker`** (QObject+QThread)
    com debounce 250 ms para coalescer `partition_registered` rajadas;
    UI consome último valor cached, refresh assíncrono.

### P1s endereçados (mesma fronteira arquitetural)

- **P1-O1 — `onboarding_wizard.py:399-409` `save()` blocking no MainThread**
  (`env_path.write_text` síncrono dentro do slot `_on_next_clicked` da página
  CREDS). Custo medido <50 ms em SSD, pode atingir 200 ms em HDD/encrypted
  volumes. Decisão: **Defer aceitável** (operação bounded, write único
  pequeno) **OU Worker one-shot** (mais defensivo). Escolha em AC7.

- **P1-D1 — `download_adapter.py:177` `events()` em loop sem backpressure**
  (signal queue acumula em rajada quando UI lenta). Decisão: adicionar
  **debounce/coalesce no progress signal** (similar ao `partition_registered`
  do CatalogAdapter — 100 ms suficiente para 10 Hz progresso visual).

- **P1-M1 — `metrics_panel.py:165-176` `shutdown()` com `BlockingQueuedConnection`
  wait(1500)**. Hoje em closeEvent o teardown sincroniza com a worker
  thread por até 1.5 s, perceptível como "app trava ao fechar". Decisão:
  reduzir wait timeout para 500 ms + cleanup async via `QTimer.singleShot`
  (best-effort se thread não responde).

- **P1-C1 — `closeEvent` racing em `main_window.py:704-731`** (shutdown de
  3 adapters sequencial → pode somar 5 s em pior caso). Decisão: paralelizar
  via `concurrent.futures` ou simplesmente reduzir individual `wait()` para
  500 ms cada.

### Fora de escopo (P2s opcionais — não bloqueiam Frente 4)

- Colorblind prefixes (já parcialmente em Story 4.31 AC9).
- Accessibility labels (Story 4.31 AC10).
- `progress_card.py:1014` / `download_screen.py:1011` `closeEvent` em
  widgets filhos (dead code — closeEvent de child widgets não é entregue
  pelo Qt; chamar manualmente ou remover). Tag para Story 4.27-followup.

## Acceptance Criteria

- [x] **AC1 — P0-U2: DownloadScreen `__init__` <16 ms (R11).**
  - Mover `_check_for_interrupted_download` (`download_screen.py:513-554`)

  - Adicionar signal `interrupted_job_found(object)` no

    rajada de `partition_registered` colapsa em 1 query SQLite.

### Riscos identificados

1. **Race condition em U2:** se usuário clicar "Baixar" antes do worker
   responder com `interrupted_job_found`, o banner pode aparecer
   simultaneamente ao download iniciar. Mitigação: no slot
   `_on_interrupted_job_found`, checar `self._download_active` e suprimir
   banner se já houver download em curso.

2. **Catalog ausente no boot:** `data_dir/_internal/catalog.db` não existe
   no primeiro launch. Worker U4 fallback para `rglob` (atual). Smoke
   first-run continua aceitável (data_dir vazio → rglob retorna 0
   instantaneamente).

3. **Worker StorageIndicator em test offscreen:** `QThread` em pytest sem
   event loop ativo pode não dispatch signals. Garantir teste usa
   `qtbot.waitSignal` ou `QSignalSpy`; padrão já existente em
   `tests/integration/test_storage_indicator.py`.

4. **Backpressure em P1-D1 pode atrasar percepção de "100%":** se debounce
   100 ms cair exatamente no último progress event, último update chega
   100 ms tarde. Mitigação: sempre emitir o último valor síncrono em
   `events()` esgotar.

5. **Microcopy "carregando" pode poluir Settings:** placeholder "—" já é
   neutro. Se decidir-se por `LBL_STORAGE_LOADING`, validar com Uma para
   não duplicar microcopy do MetricsPanel.

### Incógnitas a validar com profile

- Custo real do `SELECT SUM(file_size_bytes)` em partition tables com 5-10
  mil rows. Hipótese: <10 ms. Se falhar, considerar índice cobrindo.
- `shutil.disk_usage` em path de rede (NAS): pode ser mais lento que 5 ms.
  Worker já mitiga.
- Race entre `CatalogAdapter.check_interrupted_jobs` e
  `CatalogAdapter.partition_registered` callback durante boot — ambos
  rodam na thread do adapter, então serializados pela event loop.
  Confirmar via test integration.
