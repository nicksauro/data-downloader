# Story 4.7 — Settings Save persiste credenciais

## Contexto

**Bug release-blocker descoberto por Pichau** ao testar o `.exe` v1.0.2
publicado em github.com/nicksauro/data-downloader/releases:

1. Usuário abre app → Ctrl+, → SettingsScreen.
2. Digita `PROFITDLL_KEY`, `PROFITDLL_USER`, `PROFITDLL_PASS` nos campos.
3. Clica botão "Salvar" (BTN_SAVE_SETTINGS, Ctrl+S).
4. Toast verde "Configurações salvas".
5. **Fecha .exe e reabre** → campos VAZIOS de novo.

### Causa raiz (settings_screen.py:674-718, atual)

`_on_save_clicked` constrói `env_data` com os 3 PROFITDLL_*, mas só
escreve `config.toml` com `dll_path` + `data_dir`. As credenciais
**são silenciosamente descartadas** (existe `# vão para .env via outro
fluxo` mas esse fluxo NÃO EXISTE no produto v1.0.2).

Workflow `.env` é dev-only (cwd ou repo root). Para usuário final do
`.exe` que mora em `C:\Program Files\` (read-only), não há mecanismo
para o app criar/atualizar `.env` no diretório certo.

## User Story

**As a** usuário final do data-downloader.exe (não-dev, não tem
.env no path)

**I want** digitar minhas credenciais Profit na tela Settings, clicar
Salvar, fechar o app, reabrir o app e ver as credenciais ainda
presentes (mesmo que mascaradas com [Mostrar])

**So that** consigo usar o produto sem precisar editar arquivos `.env`
manualmente nem reentrar credenciais a cada sessão.

## Acceptance Criteria

1. **AC1 — Persiste em path estável (Windows-friendly).**
   - Save grava credenciais em `~/.data-downloader/.env` (ou
     `%USERPROFILE%\.data-downloader\.env` em Windows).
   - Diretório criado com `mode=0o700` se ainda não existir (chmod
     em POSIX, no-op em Windows mas best-effort).
   - Arquivo gravado com `encoding="utf-8"`.

2. **AC2 — `_bootstrap_env` em cli.py lê o path acima como fallback.**
   - Load order documentado: `cwd/.env` > `<exe-dir>/.env` >
     `~/.data-downloader/.env`.
   - Se `~/.data-downloader/.env` é o único presente, vars são
     carregadas em `os.environ` antes do CLI/UI iniciar.

3. **AC3 — Reload sem reiniciar.**
   - Após Save, `os.environ[var] = value` é setado para a sessão atual
     também — `_set_dll_status` reflete configurado imediatamente.
   - Toast "Configurações salvas" continua aparecendo.

4. **AC4 — Próxima abertura do .exe carrega as credenciais.**
   - Given .env em `~/.data-downloader/.env` com 3 vars válidas;
   - When `.exe` é executado;
   - Then `SettingsScreen._load_initial_values()` lê `os.environ` e
     popula os 3 campos (mascarados — `EchoMode.Password`).

5. **AC5 — Idempotência + segurança.**
   - Save 2x sequenciais não duplica linhas (sobrescreve).
   - Arquivo NÃO é commitado (já em `.gitignore`, mas adicionar
     `~/.data-downloader/` em INSTALL.md "files we create").
   - Comentário no topo do `.env` gerado: `# Generated by
     data-downloader Settings UI — do not commit`.

6. **AC6 — Empty state correto na primeira execução.**
   - Given fresh install, `~/.data-downloader/.env` ausente;
   - Then `_load_initial_values` mostra empty state hint (já
     implementado, AC valida que não regrediu).

7. **AC7 — Teste pytest cobre o fluxo.**
   - `tests/unit/test_settings_screen_save_credentials.py`:
     mock `Path.home()`, dispara `_on_save_clicked`, verifica
     arquivo `.env` criado com formato `KEY=value\n`.
   - Roda na suite local Windows do Pichau (smoke).

## Out of Scope

- DPAPI / keyring (ADR-018 considera isso V1.1+ — credenciais
  cifradas pelo OS). V1.0.3 mantém plain text com perms 0o700.
- Auto-test connection após Save (cobre Story 4.X separada — UX
  polish: toast "Configurações salvas, testando conexão...").

## Risks

| Risco | Mitigação |
|-------|-----------|
| Path `~/` não resolve em Windows raros | Usa `Path.home()` PEP-519 (resolve em todas plataformas). |
| Permissão 0o700 quebra em Windows | Best-effort `os.chmod`; documentar expectativa em INSTALL.md. |
| Migração de usuários que já têm `.env` em cwd | `_bootstrap_env` mantém cwd como prioridade — zero breaking. |
