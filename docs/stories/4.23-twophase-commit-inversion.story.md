# Story 4.23 — Two-phase commit invertido para ordem prescrita (P0-S2)

## Contexto / Problema

A revisão consolidada Frente 1 v1.4.0 (2026-05-16) identificou que a implementação atual do two-phase commit em `Catalog.register_partition` (`src/data_downloader/storage/catalog.py:913-980`) está **invertida** em relação ao protocolo prescrito por `docs/storage/INTEGRITY.md §4`:

**Ordem real (atual — INCORRETA):**
1. `ParquetWriter.write()` chama `os.replace(tmp, final)` em `parquet_writer.py:419` (FS commit).
2. Caller (orchestrator) chama `Catalog.register_partition(write_result, partition)`.
3. `register_partition` faz INSERT em `_pending_commits` (Fase 1).
4. `register_partition` faz UPSERT em `partitions` + DELETE de `_pending_commits` (Fase 3).

**Ordem prescrita (INTEGRITY.md §4 — CORRETA):**
1. Fase 1 — INSERT em `_pending_commits` (claim com expected_sha256 + expected_size).
2. Fase 2 — `os.replace(tmp, final)` (FS commit atômico).
3. Fase 3 — UPSERT em `partitions` + DELETE de `_pending_commits`.

**Janela atômica vazia entre `os.replace` e Fase 1:** se o processo crasha entre o passo 1 (atual) e o passo 3 (atual), o arquivo está em disco mas não há registro em `_pending_commits` nem em `partitions`. Recovery (Story 4.22) não tem como saber que o arquivo é "pending" — ele aparece em reconcile como drift A (orphan parquet) que pode ser registrado mas SEM validação de SHA esperado (a info se perdeu). Resultado: parquet órfão entra no catálogo com checksum recomputado on-disk, mesmo se for parcialmente escrito (caso raro mas possível se o sync de pages do FS falhou parcialmente).

**Solução (ADR-026 §2.1):** introduzir context manager `Catalog.pending_commit()` que enquadra a operação completa. Writer recebe injeção opcional de `catalog: Catalog | None`. Quando passado, faz Fase 1 ANTES de `os.replace` e Fase 3 DEPOIS, tudo via context manager.

**Esta story depende de 4.22** — o protocolo de recovery precisa estar funcional para que a Fase 1 antes do replace tenha valor (sem recovery, o pending viraria órfão eterno).

## Acceptance Criteria

- [x] **AC1 — Dataclass nova `PendingCommitHandle`** em `src/data_downloader/storage/catalog.py`. Atributos: `rel_path: str`, `partition: PartitionKey`, `expected_sha256: str`, `expected_size: int`, `job_id: str | None`, `pid: int`, `started_at: str` (ISO). Método `complete(write_result: WriteResult) → None` que executa Fase 3 (UPSERT em `partitions` + DELETE pending). Marca handle como `_completed = True` para detectar uso indevido (raise `RuntimeError` se `complete` chamado 2x).

- [x] **AC2 — Context manager `Catalog.pending_commit()`** com assinatura:
  ```python
  @contextlib.contextmanager
  def pending_commit(
      self,
      rel_path: str,
      partition: PartitionKey,
      *,
      expected_sha256: str,
      expected_size: int,
      job_id: str | None = None,
  ) -> Iterator[PendingCommitHandle]:
      ...
  ```
  - `__enter__`: executa Fase 1 (INSERT em `_pending_commits` com claim atômico — ver AC4). Retorna `PendingCommitHandle`.
  - Bloco do caller: faz `os.replace(tmp, final)` (Fase 2) + chama `handle.complete(write_result)` (Fase 3).
  - `__exit__`: se exception propagada do bloco, NÃO deleta a pending row (recovery na próxima boot tratará). Se exit normal e `handle._completed is False`, log warning `catalog.pending_commit.not_completed` (caller esqueceu de chamar `complete`).

- [x] **AC3 — `ParquetWriter.write()` aceita `catalog: Catalog | None = None`.** Assinatura nova em `src/data_downloader/storage/parquet_writer.py:write`:
  ```python
  def write(
      self,
      trades: list[TradeRecord],
      partition: PartitionKey,
      *,
      dll_version: str = "",
      chunk_id: str | None = None,
      catalog: Catalog | None = None,
      job_id: str | None = None,
  ) -> WriteResult:
  ```
  - Quando `catalog is None`: comportamento atual preservado (write + return `WriteResult`; caller responsável por `register_partition`).
  - Quando `catalog is not None`: usa o context manager. Calcula SHA256 + size do `tmp_path` ANTES do `os.replace`, abre `pending_commit`, faz `os.replace` dentro do bloco, chama `handle.complete(write_result)`.

- [x] **AC4 — Claim atômico em `_pending_commits` (preparação para Story 4.24 / P0-S3).** O SQL do INSERT no `pending_commit.__enter__` muda para:
  ```sql
  INSERT INTO _pending_commits(partition_path, started_at, expected_sha256, expected_size, job_id, pid)
  VALUES (?, ?, ?, ?, ?, ?)
  ON CONFLICT(partition_path) DO UPDATE SET
    started_at = excluded.started_at,
    expected_sha256 = excluded.expected_sha256,
    expected_size = excluded.expected_size,
    job_id = excluded.job_id,
    pid = excluded.pid
  WHERE _pending_commits.started_at < datetime('now', '-1 hour')
     OR _pending_commits.pid = excluded.pid
  ```
  Após o INSERT, `SELECT pid FROM _pending_commits WHERE partition_path = ?` e comparar com `os.getpid()`. Se diferente → raise `ConcurrentWriterError(partition_path)` (nova subclasse de `IntegrityError` em `public_api/exceptions.py`). Esta AC apenas estabelece a base; o retry policy do writer é Story 4.24.

- [x] **AC5 — `register_partition` reescrito como wrapper deprecated.** A assinatura pública é preservada, mas o corpo é reescrito para:
  1. Detectar se o arquivo já está em disco (writer já fez `os.replace` — caminho legacy). Se sim: faz Fase 1 + Fase 3 em sequência (comportamento atual, mantém compat).
  2. Adiciona `warnings.warn(DeprecationWarning, "register_partition is deprecated; pass catalog to ParquetWriter.write()", stacklevel=2)`. Remoção planejada para v1.5.0.

- [x] **AC6 — Orchestrator atualizado.** Em `src/data_downloader/orchestrator/orchestrator.py`, o callsite de `parquet_writer.write(...)` (procurar via `Grep "writer.write\\|parquet_writer.write"`) passa a injetar `catalog=self.catalog, job_id=self.job_id`. Remove o chamada subsequente a `self.catalog.register_partition(write_result, partition, job_id=...)`. Preserva `maybe_compact_month` (chamado APÓS o write).

- [x] **AC7 — Compact_month também usa pending_commit.** `src/data_downloader/storage/parquet_writer.py::compact_month` (linhas 455-611) ganha kwarg `catalog: Catalog | None = None`. Quando passado, enquadra o `os.replace` do `{MM}.parquet` com `pending_commit`. O DELETE dos diários permanece fora (responsabilidade do caller / `maybe_compact_month`). Atualizar `Catalog.maybe_compact_month` (`catalog.py:1178-1338`) para passar `catalog=self`.

- [x] **AC8 — INTEGRITY.md §4 atualizado.** Adicionar nota explícita ao final de §4.2: "Implementação: `Catalog.pending_commit()` context manager (`catalog.py`); writer e compact_month usam injeção opcional. `register_partition` mantido como wrapper deprecated para compat v1.3.x." Atualizar §4.3 com link para `recover_pending_commits` (já adicionado em 4.22).

- [x] **AC9 — Testes unit (mínimo 6).** `tests/unit/test_pending_commit_context.py`:
  1. `pending_commit` happy path: INSERT pending → `handle.complete(write_result)` → UPSERT partitions + DELETE pending.
  2. `pending_commit` com exception no bloco: pending row preservada (recovery na próxima boot resolverá).
  3. `pending_commit` exit normal sem `complete()`: log warning + pending row preservada.
  4. `pending_commit` chamado 2x para mesmo `partition_path` por mesmo PID: aceitar (claim idempotente).
  5. `pending_commit` para `partition_path` com pending de PID diferente vivo (mock): raise `ConcurrentWriterError`.
  6. `pending_commit` para `partition_path` com pending stale (>1h): claim sobrescreve, sucesso.

- [x] **AC10 — Testes integration (mínimo 4).** `tests/integration/test_twophase_commit_inversion.py`:
  1. Writer com `catalog` injetado: invoca write → verifica que `_pending_commits` foi populado ANTES do `os.replace` (mockar `os.replace` para inspect state pré-replace).
  2. Writer sem `catalog`: comportamento legacy preservado (write retorna WriteResult, caller pode chamar `register_partition` manualmente).
  3. Crash simulado entre `os.replace` e `handle.complete`: arquivo presente, pending presente, `partitions` ausente. Próximo boot (recovery 4.22) recupera para `partitions`.
  4. `compact_month` com `catalog` injetado: pending row criada para `{MM}.parquet` ANTES do replace; após complete + DELETE diários, estado consistente.

- [x] **AC11 — Backward-compat com Story 4.21 / v1.3.x.** Property tests em `tests/property/test_parquet_per_day_hybrid_property.py` continuam passando (50 seeds, 3 invariantes). Smoke: instalar wheel v1.4.0 em ambiente com catálogo v1.3.0 e verificar que `register_partition` (deprecated) ainda funciona com DeprecationWarning emitido.

- [x] **AC12 — Suite verde.** `ruff` 0 / `mypy --strict` 0 / `pytest` 0 fail. Validar contagem total de testes pré e pós (delta esperado: +10 mínimo).

**Riscos identificados:**
- Acoplamento writer↔catalog quebra invariante de "writer puro" (ParquetWriter não conhecia Catalog em v1.3.x). Trade-off justificado em ADR-026 §2.1. Mitigação: injeção é opcional (`catalog=None` preserva comportamento antigo).
- DeprecationWarning em `register_partition` pode quebrar callsites externos (public_api, ui). Auditar via Grep "register_partition" antes de implementação e atualizar todos os callers para passar `catalog` ao writer.
- `compact_month` lidando com pending_commit: o `{MM}.parquet` é mensal (`partition.day is None`); recovery precisa entender que pending de mensal também é válido (já é — `_pending_commits.partition_path` é genérico). Confirmar com 4.22.
- Test pollution: tests que criam pending rows e não limpam podem afetar suites subsequentes. Usar fixtures isoladas com `tmp_path` + `Catalog(auto_reconcile=False, auto_cleanup_orphans=False)` para inspeção determinística.
