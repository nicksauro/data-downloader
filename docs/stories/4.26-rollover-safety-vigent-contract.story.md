## Contexto / Problema

A revisão consolidada Frente 3 v1.4.0 (2026-05-16) identificou que `_resolve_contract` em `src/data_downloader/orchestrator/orchestrator.py:785-799` resolve o contrato vigente **uma única vez** usando `config.start.date()`:

```python
def _resolve_contract(self, config: JobConfig) -> str:
    if not config.resolve_contract:
        return config.symbol
    return vigent_contract(
        self._catalog,
        config.symbol,
        config.start.date(),  # <-- 1 resolve, no início do job
        exchange=config.exchange,
    )
```

Em `Orchestrator.run` (linha 482) o resultado é usado para **todos os chunks** do loop:

```python
contract_code = self._resolve_contract(config)  # ex.: "WDOG26"
chunks = self._compute_chunks(config, contract_code, ...)
for chunk in chunks:
    self._process_chunk(..., contract_code=contract_code, chunk=chunk, ...)
```

**Cenário concreto do bug (P0-D1):**

- `download("WDO", start=2026-01-15, end=2026-06-15)` (6 meses cobrindo 4-5 rollover events).
- `vigent_contract(WDO, 2026-01-15)` → `WDOG26` (fevereiro/26, conforme `docs/storage/CONTRACTS.md:140-162`).
- Orchestrator usa `WDOG26` em **todos** os ~125 chunks diários.
- Chunks dentro da vigência de `WDOG26` (~15/jan até ~28/fev) retornam trades reais.
- Chunks fora da vigência (~01/mar em diante) — DLL aceita o ticker vencido, **retorna ret=0 e 0 trades silenciosamente**. Comportamento documentado em `docs/dll/QUIRKS.md` Q-DRIFT-32:

  > **Q-DRIFT-32 (VALIDATED 2026-05-05):** `GetHistoryTrades(WDOJ26, 'F', ...)` retorna ret=0 mas servidor NÃO despacha trades — callback V2 nunca dispara, mesmo em pregão aberto com janela curta. Causa raiz: `WDOJ26` é abril/2026 (já vencido em 2026-05-05). `WDOFUT` é ticker continuous-future agregado pela Nelogica que sempre aponta para o contrato vigente.

- Catálogo registra `chunk_ledger` status `no_trades` para dias fora-de-vigência — semanticamente "este dia não tem trades" — quando na verdade é "este dia tem trades mas pedimos com contrato errado".

A perda é **silenciosa** porque:
1. `chunk_ledger` `no_trades` é status legítimo (fins-de-semana, feriados).
2. Retries do orchestrator não disparam (DLL retornou sucesso).
3. ProgressCard mostra 100% completo.

Pichau identificou um caso real onde 6 meses de WDO entregou apenas o primeiro mês de dados + 5 meses de `no_trades` falsos — investigação consumiu ~3h até atribuir a Q-DRIFT-32.

### Workaround tácito atual

A convenção de fato no squad é **"usar `WDOFUT`/`WINFUT` para downloads históricos"** — tickers continuous-future cadastrados em `CONTRACTS.md` como `symbol_root == contract_code == "WDOFUT"` com vigência `[1900-01-01, 9999-12-31]` (`docs/storage/CONTRACTS.md:100-114`). Calls com `WDOFUT` passam pelo lookup, recebem `WDOFUT` de volta, e nunca cruzam rollover. Funciona — mas:

1. **Não é mecânica.** Qualquer chamador legítimo com raiz simples cai no buraco. Documentação `CONTRACTS.md:135-138` apenas sugere; API pública não enforça.
2. **Não cobre uso legítimo de contratos específicos** (research/auditoria histórica multi-mês com seguimento de rollover natural).

ADR-028 §2 prescreve política híbrida: **fail-loudly default + opt-in per-chunk re-resolution.** Validation de `_validate_config` detecta rollover-spanning e levanta `AmbiguousRolloverError` com mensagem prescritiva listando 3 opções. Per-chunk mode (`JobConfig.resolve_contract_per_chunk=True`) re-resolve por chunk; bypass da validação.

**Esta story é independente da Frente 2 (Story 4.25 CI Pipeline) — podem ser implementadas em paralelo.** Dependência downstream: none (frente isolada do storage/CI).

## Acceptance Criteria

- [x] **AC1 — `AmbiguousRolloverError` exception nova.** Em `src/data_downloader/public_api/exceptions.py`, subclasse de `InvalidContract` com:
  ```python
  class AmbiguousRolloverError(InvalidContract):
      def __init__(
          self,
          symbol_root: str,
          start: date,
          end: date,
          contracts_in_range: list[str],
      ) -> None: ...
  ```
  Atributos `symbol_root`, `start`, `end`, `contracts_in_range` acessíveis no instance. Mensagem multi-linha prescritiva listando os contratos detectados + as 3 opções de remediation (continuous future, specific contract, opt-in per-chunk). Disponível em `__all__` + `_TAXONOMY` mapping (`ERR_AMBIGUOUS_ROLLOVER`).

- [x] **AC2 — `Catalog.list_contracts_in_range` método novo.** Em `src/data_downloader/storage/catalog.py`:
  ```python
  def list_contracts_in_range(
      self,
      *,
      root: str,
      start: date,
      end: date,
  ) -> list[Contract]:
      """Lista Contracts cuja vigência intersecta [start, end] para root."""
  ```
  Query SQL com overlap semantics (`vigent_from <= end AND vigent_until >= start`). Ordena por `vigent_from ASC`. Retorna lista vazia se raiz não tem vigência cobrindo o range (fail-late no `vigent_contract` posterior). Adiciona ao `__all__` do módulo.

- [x] **AC3 — `JobConfig.resolve_contract_per_chunk: bool = False`.** Em `src/data_downloader/orchestrator/orchestrator.py` `@dataclass(frozen=True) class JobConfig`, novo campo:
  ```python
  resolve_contract_per_chunk: bool = False
  """Se True, re-resolve vigent_contract por chunk (opt-in para multi-rollover).
  Bypassa rollover-spanning validation. Default False = fail-loudly (Q-DRIFT-32 defense).
  """
  ```
  Update docstring de `JobConfig` listando o novo campo.

- [x] **AC4 — `_validate_no_rollover_in_window` helper + integração em `_validate_config`.** Novo método em `Orchestrator`:
  ```python
  def _validate_no_rollover_in_window(self, config: JobConfig) -> None:
      """Levanta AmbiguousRolloverError se [start, end] cruza rollover na raiz."""
      contracts = self._catalog.list_contracts_in_range(
          root=config.symbol,
          start=config.start.date(),
          end=config.end.date(),
      )
      if len(contracts) > 1:
          raise AmbiguousRolloverError(
              symbol_root=config.symbol,
              start=config.start.date(),
              end=config.end.date(),
              contracts_in_range=sorted(c.contract_code for c in contracts),
          )
  ```
  Chamado de `_validate_config` quando `config.resolve_contract == True AND config.resolve_contract_per_chunk == False`.

- [x] **AC5 — Per-chunk resolution no `Orchestrator.run` loop.** Refator do loop `for chunk_index, chunk in enumerate(chunks)` (orchestrator.py ~555) para resolver `chunk_contract_code` por chunk quando `config.resolve_contract_per_chunk == True`:
  ```python
  if config.resolve_contract_per_chunk:
      contract_code_global = config.symbol  # placeholder semântico
  else:
      contract_code_global = self._resolve_contract(config)

  chunks = self._compute_chunks(config, contract_code_global, resume_pending_chunks)

  for chunk_index, chunk in enumerate(chunks):
      if config.resolve_contract_per_chunk:
          chunk_contract_code = vigent_contract(
              self._catalog, config.symbol, chunk.start.date(),
              exchange=config.exchange,
          )  # propaga InvalidContract se sem vigência
          log.info("orchestrator.per_chunk_contract_resolved",
                   job_id=job_id, chunk_start=chunk.start.date().isoformat(),
                   contract_code=chunk_contract_code, symbol_root=config.symbol)
      else:
          chunk_contract_code = contract_code_global

      self._process_chunk(..., contract_code=chunk_contract_code, chunk=chunk, ...)
  ```
  `_process_chunk` já aceita `contract_code: str` (signature inalterada — `orchestrator.py:913`).

- [x] **AC6 — `_compute_chunks` awareness de per-chunk mode.** Em modo per-chunk, `_compute_chunks` recebe `config.symbol` (raiz) em vez de um contract_code resolvido. A query de `chunk_ledger.completed_days` precisa filtrar por **conjunto de contract_codes possíveis na raiz** ao invés de um único code:
  - Em modo per-chunk: `Catalog.completed_days_by_root(root, exchange, start, end)` consulta `_pending_commits`/`partitions`/`chunk_ledger` para todos os contract_codes cuja raiz é `root`.
  - Em modo single-contract (default): usa o callsite atual `completed_days(contract_code, ...)` — sem mudança.
  - O cache-hit check no `Orchestrator.run` (linha 502) também precisa de awareness — em per-chunk mode, "cache hit" significa "todos os dias úteis do range cobertos por partitions de QUALQUER contract_code da raiz".
  - Sol decide se adiciona método novo `Catalog.completed_days_by_root` OU expande `completed_days` com parâmetro opcional `roots: set[str]`. Privilegiar API menor (expansão de método existente preferível).

- [x] **AC7 — Testes unitários (mínimo 6).** `tests/unit/test_rollover_safety.py` (NEW):
  1. `_validate_no_rollover_in_window` com 1 contrato no range → no raise.

**Riscos identificados:**
- **Breaking change parcial.** Chamadas com `WDO`/`WIN`/etc. cruzando rollover param de funcionar silenciosamente entregando lixo; passam a falhar cedo com `AmbiguousRolloverError`. Mensagem é prescritiva, mas comunicação no CHANGELOG/release notes v1.4.0 deve ser proeminente.
- **`_compute_chunks` refactor.** Hoje recebe `contract_code` único e filtra ledger por esse code. Per-chunk mode quebra essa premissa. Risco: cache hit em re-run de job per-chunk pode dar false miss se a query não considerar contract_codes heterogêneos.  Quinn deve cobrir explicitamente em integration test.
- **UI dialog edge cases.** "Usar WDOFUT" só funciona se `{root}FUT` existe em CONTRACTS.md (WDO, WIN, IND, DOL têm; outras raízes futuras como BGI não). Implementar guard que grey-out a opção quando não houver continuous future cadastrado.
- **Tests existentes podem assumir bug.** Auditar tests/* que invocam download com raízes (`WDO`, `WIN`) e ranges grandes — ajustar para usar `WDOFUT` OU `resolve_contract_per_chunk=True` explicit. Hipótese: poucos casos (Sol audita em AC7 cleanup).
- **Performance per-chunk.** 1 query SQLite por chunk para lookup. Em job de 7 anos × 1 dia/chunk = ~1800 chunks → ~1800 queries adicionais. Custo desprezível (~1ms cada) mas mensurar via log se houver suspeita.
