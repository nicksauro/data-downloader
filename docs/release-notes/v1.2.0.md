# data-downloader v1.2.0 — download robusto + períodos extensos (desde 2018)

> BIG COUNCIL round 3 (6 agentes). Foco: a ProfitDLL traduz ~100% dos trades; resume funcional; baixar desde 2018; UX para downloads longos; remoção de dead-code. Build `[git_sha]` · 2026-05-12.

## Destaques

### 🎯 `translate_failures` ≈ 0 — a DLL não perde mais trades
O callback V2 de trade histórico só enfileirava o `handle` (`a_pTrade: Pointer`) e o `TranslateTrade` era feito depois no IngestorThread — mas o handle só é válido **dentro do escopo do callback** (a DLL recicla o buffer interno do pacote). Com milhões de trades enfileirados, ~0.01% dos handles ficavam stale → `ConnectorMarketDataLibraryU.PopulateTradeV0` lia memória liberada → access violation interna da DLL (SILENT MODE, "Error Number 1") → `TranslateTrade` retornava lixo → trade perdido. **Fix:** o callback agora chama `TranslateTrade` dentro do escopo (padrão do exemplo Nelogica) e enfileira o `TradeFields` já copiado, não o handle. R3/ADR-005 emendado. Cinto-e-suspensório: retry do chunk se `completeness_pct < 99.99%` (máx. 2); métrica `download_chunk_completeness_pct` exportada. (Q-DRIFT-40, Story 4.20.)

### 🔁 Resume funcional
- `--resume <job_id>` na CLI deixou de ser no-op.
- **Auto-resume:** se há um job incompleto (`pending`/`in_progress`/`partial`/`failed`) pro mesmo `(symbol, exchange, range)`, retoma automaticamente; ao fim de um job `partial`, imprime `--resume {job_id}`.
- `download(symbol, start, end, *, resume_job_id=...)` na public API.
- **Fresh-path skip:** re-rodar `download` do mesmo range é idempotente e barato — consulta o novo `chunk_ledger` (1 row por dia útil baixado) e baixa só os dias faltantes. Fim da **perda silenciosa** do resume mensal antigo (um mês "tocado" no dia 12 contava como completo e os outros ~21 dias não eram baixados). `CATALOG_VERSION` 1.1.0 → 1.2.0 (migração automática).

### 📅 Períodos extensos (desde 2018)
- `_PARTITION_ROW_LIMIT` 5M → 50M — um mês movimentado (WDOFUT ≈ 10-13M trades) não aborta mais o job com `IntegrityError` por volta do 10º dia útil. *(A solução definitiva — parquet-por-dia, ADR-025 — fica para a v1.2.1: ver `docs/qa/V1.2.0-PLAN.md`. O paliativo cobre o blocker; o custo é ~3h extra de I/O num download de 7 anos — wall-clock é dominado pela B3, ~30-90h.)*
- **PeriodPicker** com presets: "Mês corrente" / "Último ano" / "Ano completo: YYYY" / "Tudo desde 2018" / "Personalizado"; aviso de duração para ranges grandes ("⚠ ~N dias úteis, ~Xh-Yh de download — você pode pausar e retomar").
- **ProgressCard long-haul:** tempo restante (ETA), tempo decorrido, throughput (trades/s · chunks/h), trades baixados, trades perdidos (`0` verde / `N (0.XX%)` amarelo), chunks com retry; "Calculando plano de download…" quando o plano ainda não foi computado; log view limitado a 2000 blocos (não acumula em downloads de horas); botão "Abrir Pasta" durante o download.
- **Retomar ao reabrir:** se o app foi fechado/crashou durante um download, ao reabrir aparece um banner "Retomar download de {symbol}? (parou em X/Y chunks)" com [Retomar] / [Começar do zero] / [Descartar].

### 🧹 Dead-code removido
- `orchestrator/broker/` (~2034 LOC) — o multi-process broker pós-ADR-022 (`-p N>1`) usava um *mock factory* por default → gravava parquet com **dados falsos** silenciosamente. Agora `-p N>1` é deprecado (licença Nelogica é single-session) → cai no loop sequencial single-symbol com aviso. + 4 test files do broker removidos.
- `ui/shortcuts.py` — placeholders nunca chamados (`Ctrl+R`/`Ctrl+L`/`Ctrl+E`/`Ctrl+O` sem handler).
- Microcopy `BTN_REPEAT_LAST` órfão; waiver `test_pool_lifecycle_broker_dead_code.md` obsoleto.

### Outros
- `busy_timeout=5000` no perfil SQLite (`wal_checkpoint(TRUNCATE)` não falha mais com `database is locked` quando a UI tem read-tx aberta).
- `get_pending_chunks`: `OR`-query → `UNION` de SELECTs indexados (+ `idx_partitions_job`) — não vira mais full scan no resume.
- `test_smoke_imports::test_root_package_exposes_version` deriva da fonte (não enrijece o literal); `ruff UP047` (PEP-695) → `ignore` (migração diferida v1.3.0).
- Suite: ruff 0 / mypy --strict 0 (87 source files) / **1194 unit + 508 integration+property, 0 fail** (3 skips ambientais: plataforma não-Windows).

## Como atualizar

- Tem v1.1.x ou v1.0.x instalado? Baixe `data-downloader-Setup-v1.2.0.exe` e execute — InnoSetup faz upgrade in-place; `~\.data-downloader\.env` e os dados Parquet/catálogo são preservados (a migração do catálogo 1.1.0→1.2.0 é automática na 1ª abertura).
- Instalação nova? Setup.exe → preencha `~\.data-downloader\.env` (`PROFITDLL_KEY`/`PROFITDLL_USER`/`PROFITDLL_PASS`) → `data_downloader-cli.exe --healthcheck` (deve dizer `healthcheck OK`) → abra o app.
- Portable: `data-downloader-v1.2.0-win64.zip`.
- **Baixar histórico longo:** UI → escolha o símbolo → PeriodPicker preset "Tudo desde 2018" (ou "Ano completo: 2018", etc.) → BAIXAR. Pode fechar o app e retomar depois (banner ao reabrir), ou via CLI: `data_downloader-cli.exe download --symbol WDOFUT --start 2018-01-01 --end 2026-05-01 --exchange F` (e `--resume <job_id>` se interromper).

## Pendências conhecidas (v1.2.1+)

- **ADR-025 parquet-por-dia** (`{ex}/{sym}/{YYYY}/{MM}/{DD}.parquet`) — substitui o paliativo `_PARTITION_ROW_LIMIT` 50M; elimina o O(N²) do append mensal e dá resume granular nativo. Planejado para v1.2.1.
- **Pause/Resume mid-download na UI** (parar sem cancelar) — o backend tem `cancel`; falta o `suspend`/`resume` no orchestrator + botão na UI.
- **Reconnect-without-reinit** — se a sessão Nelogica cair no meio de um download de 30-90h, os chunks falham e o job termina `partial` (recuperável via `--resume`, mas não automático). Reconexão automática (sem re-init da DLL — Q08-E) é P1 da v1.2.1.
- **CatalogScreen heatmap de cobertura** + filtro por ano (pra 7 anos × N símbolos).
- **Smoke real (Pichau, em pregão B3):** validar `translate_failures` ≈ 0 e completeness ~100% num download de 1 mês WDOFUT (CLI + UI); resume real (matar o `.exe` no meio, reabrir, retomar).
