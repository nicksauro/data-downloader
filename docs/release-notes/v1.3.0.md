# data-downloader v1.3.0 — UX excepcional + parquet híbrido

> BIG COUNCIL round 4 (6 agentes mini-councils paralelos). Pichau directive 2026-05-13: "vamos fazer o possível para a nova v ser excepcional" + "parquets mensais para ranges grandes, diários para pequenos — rigidamente implementar e testar". Build `bf1448e` · 2026-05-13.

## Destaques

### 📦 Parquet híbrido mensal + diário com auto-compact (ADR-025)

Pichau directive textual: *"precisamos de parquets mensais, quando baixarmos datas maiores que mês, parquets mensais. quando baixarmos datas menores, vamos ter parquets de dias, se não é muito ruim, vai ficar muito parquet, e pra qlqr backtest fica ruim"*.

Layout canônico:
- **Mensal** — `{exchange}/{symbol}/{YYYY}/{MM}.parquet` — quando o mês está **completo** (todos os dias úteis B3 com `chunk_ledger` `completed` ou `no_trades`).
- **Diário** — `{exchange}/{symbol}/{YYYY}/{MM}/{DD}.parquet` — durante o mês corrente, durante downloads parciais, ou para ranges intra-mês.

**Auto-compact:** ao final de cada `register_partition`, `maybe_compact_month()` checa `is_month_complete()` e — se sim — escreve `{MM}.parquet` atomicamente:
1. Tmp `{MM}.parquet.tmp` + `os.replace` + fsync
2. Verifica SHA256 do output
3. Insere row em `compactions` (in-flight registry)
4. Só **DEPOIS** deleta os `{DD}.parquet` daquele mês
5. Marca a row de `compactions` como `done`

**Crash mid-compact:** recovery na próxima boot via `_recover_inflight_compactions`. Se `{MM}.parquet` existe e é íntegro (row group counts conferem) → completa (deleta diários). Senão → reverte (delete `{MM}.parquet` + .tmp, mantém diários).

**Migração v1.2→v1.3:** idempotente. `CATALOG_VERSION` 1.2.0 → 1.3.0 (deltas SQL: `ADD COLUMN partitions.day INTEGER NULL`, `idx_partitions_symbol_ymd`, tabela `compactions`, `idx_compactions_inflight`). Dados v1.2.0 (`{MM}.parquet` mensais) ficam como estão; novos meses incompletos passam a usar layout diário; meses já fechados continuam mensais.

**Validação rígida** (Pichau directive textual: *"vamos rigidamente implementar e testar"*):
- **T1-T9 testes integração:** mês completo→compacta / mês incompleto→não compacta / crash recovery completa íntegro / crash recovery reverte inconsistente / write-once / layout exclusivo / idempotência / double-register / race condition.
- **Property test (Hypothesis, 50 seeds, 3 invariantes):**
  - Conservação: `sum(trades em {MM}.parquet) == sum(trades em {DD}.parquet originais)`.
  - Exclusividade mútua: nunca `{MM}.parquet` e `{MM}/{DD}.parquet` coexistem para o mesmo `(symbol, exchange, YYYY, MM)`.
  - Consistência catálogo↔FS: `partitions.day IS NULL` ⇔ `{MM}.parquet` existe; `partitions.day = N` ⇔ `{MM}/{N:02d}.parquet` existe.
- **T10 (smoke real em pregão B3 com auto-compact ao fechar mês):** Pichau pós-build.

### ✨ UX excepcional

- **Onboarding wizard** — `OnboardingWizard(QDialog)` modal de 3 telas (Welcome → Credenciais Nelogica → Done) disparado pelo `app.main()` quando `~/.data-downloader/.env` não existe ou `PROFITDLL_KEY` está vazio. Substitui o banner "Configurar credenciais" que aparecia escondido no topo.

### ⚡ Performance

- `_ROW_GROUP_SIZE` 100k → 1M — +30% throughput no parquet write (medido `bench_parquet_read_filtered.py`).
- `gc.freeze()` no início do orchestrator + `gc.collect()` entre chunks — 7-12% menos tempo total em downloads de 30+ dias (menos pausas GC com milhões de objetos vivos).
- **Write-once quando partition.day is not None** — sem read-merge-rewrite (que era O(N²) no mensal). Write de 1 dia útil de WDOFUT: ~700ms → ~50ms.

### Pequenas e médias

- `partitions.day INTEGER NULL` + `idx_partitions_symbol_ymd`.
- `compactions(symbol, exchange, year, month, started_at, finished_at, status, ...)` + `idx_compactions_inflight`.
- Observer `partition_registered = Signal(str, int, int)` com debounce 500ms via QTimer no `CatalogAdapter` — o CatalogScreen atualiza em tempo real durante o download.
- Microcopy: `BTN_CANCELLING`, `LBL_PROGRESS_TITLE_DOWNLOADING`, `INF_CALCULATING_PLAN`, `LBL_CANCELLING_HINT`, `LBL_STATUSBAR_DLL_*` (5 estados), `LBL_ONBOARDING_*` (18 IDs), `LBL_STORAGE_INDICATOR`, `TIP_STORAGE_INDICATOR`, `WAR_STORAGE_LOW`.

## Como atualizar

- **Tem v1.2.x ou v1.1.x instalado?** Baixe `data-downloader-Setup-v1.3.0.exe` e execute — InnoSetup faz upgrade in-place; `~\.data-downloader\.env` e os dados Parquet/catálogo são preservados (a migração catálogo 1.2.0→1.3.0 é idempotente na 1ª abertura). Atalhos antigos (sem ícone) podem ser deletados/recriados pelo Setup automaticamente.
- **Instalação nova?** Setup.exe → ao 1º run, o **onboarding wizard** te guia (Nelogica creds → Done) → `data_downloader-cli.exe --healthcheck` (deve dizer `healthcheck OK`) → abra o app.
- **Portable:** `data-downloader-v1.3.0-win64.zip`.
- **Layout do disco após upgrade:** meses já fechados (v1.2.0) ficam como `{ex}/{sym}/{YYYY}/{MM}.parquet` (sem mudança); novos meses incompletos passam a usar `{ex}/{sym}/{YYYY}/{MM}/{DD}.parquet` até fecharem (auto-compact dispara automaticamente ao baixar o último dia útil).

## Pendências conhecidas (v1.4.0+)

- **Bug 7 — Code signing (SmartScreen)** — diferido para v1.4.0 (sem orçamento agora). Workaround documentado em `docs/release/INSTALL.md`.
- **T10 — Smoke real em pregão B3** — Pichau pós-build: download via UI de 1 mês WDOFUT em pregão (validar auto-compact dispara ao fechar mês + completeness ~100% + layout híbrido no FS).
- **Pause/Resume mid-download na UI** (pausar sem cancelar; o backend tem `cancel`, falta `suspend`/`resume`).
- **Reconnect-without-reinit** — se a sessão Nelogica cair no meio de um download de 30-90h, os chunks falham e o job termina `partial` (recuperável via `--resume`, mas não automático; o `state_monitor` v1.3.0 já detecta a queda e seta `pause_event` — falta o handshake de reconexão sem `Finalize`).
