## 4. Achados (Findings)

### F-N-1 — INFO — Q18-OPEN aguarda probe humano (Story 4.2-followup)

- **Arquivo:** `docs/dll/QUIRKS.md` Q18-OPEN.
- **Descrição:** Vigência exata WIN H/M/U/Z permanece hipotetizada
  (regra B3 5º dia útil X-3 → quarta mais próxima 15/X). Manual silencioso
  sobre calendário oficial. Validação empírica requer probe real contra
  `WINH26 + 5 dias úteis` em ProfitDLL ao vivo.
- **Manual ref:** `manual_profitdll.txt` §3.1 linha 1747 (silencioso).
- **Sugestão de fix:** Story 4.2-followup AC2 — humano roda probe e
  atualiza `validation_source` por contrato. Q18-OPEN move para
  `validated` ou `ambiguous` (se ±5d de divergência observada).
- **Probe proposto:** `data-downloader contracts validate --root WIN
  --year 2026` (já implementado Story 1.6).
- **Severity rationale:** INFO — estrutura está correta (registro Q18-OPEN
  + WAIVER + 4.2-followup story-debt). Tracking apenas até resolução
  humana.

### F-N-2 — LOW — `_resolve_sample_date` fallback `today()-7d` ignora calendário B3

- **Arquivo:** `src/data_downloader/orchestrator/contracts_probe.py:199-228`.
- **Descrição:** Para equity, `today() - timedelta(days=7)` pode cair em
  fim-de-semana ou feriado se operador rodar probe ad-hoc num cenário
  fronteira (e.g. segunda pós-feriado prolongado). Probe retornará 0
  trades → `ProbeResult(success=False, reason="no_trades")` — operador é
  avisado, mas pode perder ciclo de probe. **Mesmo problema que F-S-6 do
  audit Story 1.6** (futuros) — agora estendido a equities.
- **Manual ref:** CONTRACTS.md §6.3 (DST B3 — limite >= 2020).
- **Sugestão de fix:** Quando Story 2.X (`bizdays-integration`) entregar
  `is_b3_business_day` cross-checked com Nelogica `holidays.dat`, usar
  `last_b3_business_day_before(date.today() - 7d)` no fallback. Aplicar
  identicamente para futuros (F-S-6) e equities (F-N-2).
- **Severity rationale:** LOW — log + retry semântico cobrem o caso.
  Tracking para Story 2.X.

### F-N-3 — INFO — Probe equity reusa `download_chunk` sem alteração de wrapper

- **Arquivo:** `src/data_downloader/orchestrator/contracts_probe.py:133-141`.
- **Descrição:** `probe_contract` chama `download_chunk` (Story 1.3)
  passando `exchange="B"` para equity. Wrapper DLL (`dll/wrapper.py`)
  continua opaco a asset class — recebe `TAssetIDRec(ticker, bolsa)` e
  delega à DLL nativa. Lei R3 (probe não toca callbacks diretamente)
  preservada — reusa primitive `download_chunk` que já obedece R3.
- **Manual ref:** `manual_profitdll.txt` §3.1 linha 1673 (literal "PETR4
  → B").
- **Sugestão de fix:** N/A — comportamento correto. Tracking apenas para
  documentar que equity probe é estruturalmente idêntico a futuros probe
  (mesma rota código, exchange diferente).
- **Severity rationale:** INFO — design consistency.

### F-N-4 — LOW — `is_equity_ticker` regex não cobre UNT (units, sufixo 11)

- **Arquivo:** `src/data_downloader/orchestrator/chunker.py:75 (EQUITY_TICKER_RE)`.
- **Descrição:** Regex `^[A-Z]{4}\d$` exige exatamente 1 dígito final.
  Tickers UNT (units, e.g. `SANB11`, `TAEE11`, `BIDI11`) têm sufixo `11`
  (2 dígitos) e NÃO casam. Atualmente é OK porque seed v1.1.0 não inclui
  UNT, mas ressalva CONTRACTS.md §2.3 ("tickers podem mudar (split,
  fusão)") sugere que UNT pode aparecer em V1.x.
- **Manual ref:** Convenção B3 — UNT são "11" (BR units mistas: ON+PN
  típicos).
- **Sugestão de fix:** Quando UNT for incluído no seed (Story V1.x), bump
  regex para `^[A-Z]{4}(\d|11)$` ou `^[A-Z]{4}\d{1,2}$`. Test matrix
  estendida cobre UNT. F-N-4 trackeia explicitamente.
- **Regression test sugerido:** Adicionar `test_is_equity_ticker_canonical_b3[SANB11]`
  quando regex for ampliada.
- **Severity rationale:** LOW — escopo atual (V1.0) cobre apenas tickers
  3/4 dígito. Tracking para roadmap V1.x.

### F-N-5 — INFO — CLI `data-downloader contracts validate --symbol PETR4 --exchange B` documentado (não codificado)

- **Arquivo:** Story 4.2 AC3 §"Comando documentado" + WAIVER §"Plano de remediação".
- **Descrição:** O comando CLI está documentado mas o flag `--exchange`
  para `contracts validate` ainda não é implementado (CLI Story 1.6 só
  expõe `--root WIN --year 2026`). Smoke real humano (4.2-followup)
  precisará passar `--exchange B` para PETR4.
- **Manual ref:** Story 4.2 AC3 + COUNCIL-29 D3.
- **Sugestão de fix:** Story 4.2-followup AC3 ou Story 1.6-followup
  (preferível) — adicionar flag `--exchange {F,B}` em
  `contracts validate` CLI command. Default permanece "F" (retrocompat).
- **Severity rationale:** INFO — não bloqueia merge desta story (mock-first
  impl funciona). Tracking para humano antes de rodar probe real.
