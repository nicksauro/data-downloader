## 6. Findings

### F-A-1 — INFO — Story 2.2 deve documentar trajetória do flag `_use_legacy`

- **Arquivo:** `docs/stories/2.2.story.md` Subtask 2.2
- **Descrição:** Flag `_use_legacy_table_builder=False` por default + manter
  loop como fallback é boa prática, mas Story 2.2 não especifica QUANDO a
  flag será removida (em qual sprint, em qual versão).
- **ADR/INV ref:** ADR-005 (mudança de comportamento default exige clareza)
- **Sugestão de fix:** Adicionar à Dev Notes: "Flag legacy removida em Story
  2.X (sprint após estabilização — Pyro decide via *bench-vs-baseline 5
  ciclos consecutivos sem regressão)."
- **Trade-off:** Não bloqueia gate; melhoria de clareza para auditoria futura.

### F-A-2 — INFO — DuckDB intra-writer: documentar política de invocação

- **Arquivo:** `docs/stories/2.2.story.md` AC4 + Subtask 4.2
- **Descrição:** Se `dedup_duckdb` (SELECT DISTINCT) ganhar a comparação
  pareto-ótima, isso introduz nova invocação DuckDB no hot path do
  `WriterThread`. Não é dep nova (ADR-002), mas é um novo *padrão de uso*
  que merece nota arquitetural (DuckDB embedded = thread-safe per connection,
  cada writer thread precisa connection isolada).
- **ADR/INV ref:** ADR-002 + R21 (hot path)
- **Sugestão de fix:** Subtask 4.2 deve explicitar: "Se vencedor for
  duckdb DISTINCT, criar `_dedup_connection: duckdb.DuckDBPyConnection` por
  writer (lazy init) e documentar em ARCHITECTURE.md §thread-model que
  WriterThread agora opera connection DuckDB local."
- **Trade-off:** Não bloqueia design APPROVED; refinement para implementação.

### F-A-3 — LOW — Re-baseline contra v1.0.0-real: definir o que acontece se humano demora

- **Arquivo:** `docs/stories/2.2.story.md` §"NÃO bloqueia (paralelo)"
- **Descrição:** Story 2.2 declara que deve re-baseline contra v1.0.0-real
  antes do merge final. Mas se Story 1.7b-followup demora indefinidamente
  (humano não roda smoke), Story 2.2 fica bloqueada para merge mesmo com
  AC1-AC8 PASS contra v1.1.0-mock.
- **ADR/INV ref:** Política de WAIVERS (`docs/qa/WAIVERS/README.md`)
- **Sugestão de fix:** Adicionar fallback: "Se Story 1.7b-followup ainda
  pendente em 2026-XX-XX, Story 2.2 merge contra v1.1.0-mock com WAIVER
  `2.2-real-baseline-deferred` similar ao 1.7b. Re-baseline vira followup."
- **Trade-off:** Não bloqueia design; mitiga deadlock potencial.

### F-A-4 — INFO — COUNCIL-10 não menciona impacto em bench_callback_to_disk especificamente

- **Arquivo:** `docs/decisions/COUNCIL-10-perf-optimization-roadmap.md`
- **Descrição:** COUNCIL-10 afirma que vectorização (5x) levaria
  `bench_callback_to_disk` p99 de 1_510ms para ~300ms. Mas decomposição
  em 3 sub-targets (COUNCIL-02 §6 Aria) não está re-citada explicitamente
  em Story 2.2 ACs. Pyro implementa AC1 e Story 2.2 fecha sem garantir
  que p99 callback→disk também melhora.
- **ADR/INV ref:** COUNCIL-02 §6 (Aria endorsement decomposição target)
- **Sugestão de fix:** Story 2.2 AC7 (regression check) já inclui
  `bench_callback_to_disk` na suíte. Adicionar explicitamente em AC7:
  "p99 chunk-mode espera-se < 500ms pós-vectorização (gap reduzido de
  +1410% para ~+400%)."
- **Trade-off:** Não bloqueia; refinement de AC.

**Total:** 0 CRITICAL, 0 HIGH, 0 MEDIUM, 1 LOW, 3 INFO.

**Justificativa:**

1. **COUNCIL-10 é arquiteturalmente sã.** Decisão de criar Story 2.2 em vez
   de inflar Story 1.8 preserva trilha de auditoria (Story 1.4 gate fechado),
   permite property tests Hypothesis dedicados (Aria rec 7 COUNCIL-02), e
   mantém regression budget canônico. Sign-off Pyro+Sol+Aria é legítimo
   (todas as autoridades de domínio representadas).

2. **Story 2.2 não muda fronteira pública.** ADR-007a preservado
   (`__api_version__ = "0.3.0"` intacto). `SCHEMA.md` intacto (Sol audit
   `1.8-storage-2026-05-04.md` confirma 17 campos byte-a-byte). Refactor
   confinado a `data_downloader.storage.parquet_writer` — sem propagação
   cross-layer. **Sem ADR amendment necessário.**

3. **Vectorização é refactor interno disciplinado.** AC2-AC5 atacam 4 hot
   paths sistematicamente; AC6 (property tests Hypothesis) é o gate apropriado
   para refactor de risco; AC7 (regression check) protege os outros 5
   benchmarks; AC8 (sem mudança fronteira) é AC verificável (diff
   `public_api/` = vazio, diff `SCHEMA.md` = vazio).

4. **Trade-offs respeitam princípios:** "menor blast radius primeiro"
   (Opção B vectorização antes de Opção C streaming-append); "não inventar
   feature" (refactor de algo medido, não palpite); "fronteira é sagrada"
   (sem mudança public_api / SCHEMA / Catalog API).
