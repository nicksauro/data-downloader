## 4. Achados (Findings)

### F-N-1 — INFO — Tabela canônica completa para escopo data-downloader (sem ação)

- **Arquivo:** `src/data_downloader/dll/error_taxonomy.py:129-336`
- **Descrição:** 39 codes mapeados cobrem **100% dos códigos retornados
  por funções `ProfitDLL.*` que data-downloader chama** (init, finalize,
  subscribe history, callbacks). Nelo confirma: NÃO falta nenhum código
  relevante para esta camada (códigos de trading — `NL_ORDER_*` —
  apareceriam apenas em modo trading que data-downloader não usa).
- **ADR/INV ref:** R5 (escopo data-downloader é market-only).
- **Sugestão de fix:** Sem ação. Story futura de trading-enabled (não
  planejada) precisaria estender tabela com `NL_ORDER_REJECTED`,
  `NL_ORDER_INVALID_QTY`, etc. Documentar via ADR no momento.

### F-N-2 — INFO — Justificativas em prosa são audit trail de longo prazo

- **Arquivo:** `src/data_downloader/dll/error_taxonomy.py` (entries 79-336)
- **Descrição:** Nelo destaca que o **3º item do tuple
  `(name, category, justification)`** é o investimento mais valioso
  desta tabela. Daqui a 6 meses, qualquer dev/agent que questionar
  "por que `NL_NOT_FOUND` é AMBIGUOUS e não TRANSIENT?" tem a resposta
  inline (`"pode ser warming OU pediu errado"`). Documenta intenção,
  não só decisão.
- **ADR/INV ref:** ADR sob princípio "ADR não é burocracia, é memória
  institucional" (Aria).
- **Sugestão de fix:** Sem ação. Boa prática a manter em adições
  futuras.

### F-N-3 — INFO — UNKNOWN como NO-RETRY é escolha conservadora R7 (Nelo endossa)

- **Arquivo:** `src/data_downloader/dll/error_taxonomy.py:339-377`
- **Descrição:** Decisão de tratar códigos não-mapeados como `UNKNOWN`
  → NO retry (em vez de "default TRANSIENT") foi discutida em
  COUNCIL-20 §D1 trade-off. Alternativa rejeitada: assumir TRANSIENT
  para "pode ser flake transitório de release nova". Nelo APROVA a
  decisão: release nova da ProfitDLL pode introduzir códigos com
  semântica completamente diferente (ex.: novo `NL_THROTTLE_BACKOFF`
  com semântica X), e retry-ar cegamente mascara comportamento. R7
  vence — agente deve auditar antes de classificar.
- **ADR/INV ref:** R7 (fail fast em erro lógico).
- **Sugestão de fix:** Sem ação. Quando próxima release ProfitDLL for
  validada, Nelo revisa `profit.h` mais recente e estende
  `NL_CATEGORY_MAP` com novos códigos.

### F-N-4 — INFO — Q02-E completamente formalizado (workaround → policy)

- **Arquivo:** `docs/dll/QUIRKS.md` Q02-E (linhas 54-71)
- **Descrição:** Story 2.6 fecha o ciclo Q02-E corretamente: status
  `validated` declarado (linha 21+57), workaround documentado em
  duas camadas — (a) Story 1.3 `download_primitive` trata 99%
  transparente via timeout duro; (b) Story 2.6 circuit breaker NÃO
  conta progress=99% como falha porque NÃO é error code (`is`
  estado de fluxo). Test reprodutor `test_circuit_breaker_does_not_count_q02e_progress_99_as_failure`
  valida — 1 chunk completed após N callbacks 99% → breaker permanece
  CLOSED. Nelo confirma: QUIRKS.md agora é referência única e
  unambiguous.
- **ADR/INV ref:** Q02-E `validated` (Story 2.6 close).
- **Sugestão de fix:** Sem ação. Q02-E pode ser considerado fechado
  para escopo data-downloader.

**Total findings: 0 CRITICAL, 0 HIGH, 0 MEDIUM, 0 LOW, 4 INFO.**

## 5. Decisão

| Verdict             | Marcar |
|---------------------|--------|
| ✅ **APPROVED**     | **[x]** |
| 🟡 CHANGES_REQUESTED | [ ]    |
| 🔴 BLOCKED          | [ ]    |

**Justificativa:**

A Story 2.6 entrega uma **taxonomia NL_* canônica e completa** para o
escopo de data-downloader — 39 códigos categorizados em 4 categorias
(`TRANSIENT`, `AMBIGUOUS`, `PERMANENT`, `UNKNOWN`) com semântica que
respeita FIELMENTE:

1. **A fonte primária do manual ProfitDLL** (`profit.h` L217-222 +
   `main.py` L13-48). Todos os 33 códigos canônicos do `profit.h` estão
   presentes; mapeamento de cada um audita-se contra semântica
   documentada.
2. **Os 5 códigos LEGACY** (Story 1.2 `dll/errors.py`) são honrados com
   mesma categoria semântica que canônicos — sem divergência entre
   `decode_nl_error` (legacy lookup) e `categorize_nl` (taxonomia).
3. **A sentinela interna `_DLL_SENTINEL=-1`** (wrapper-only, NÃO vem
   da DLL) é catalogada como PERMANENT com justificativa explícita
   ("bug de ambiente — COMPANIONS_MISSING / UNSUPPORTED_PLATFORM").
4. **A defesa conservadora R7**: códigos não-mapeados retornam
   `UNKNOWN` (NO retry) com justificativa explícita explicando que
   release nova da ProfitDLL pode introduzir semântica nova. Decisão
   trade-off em COUNCIL-20 §D1 — Nelo aprova explicitamente.
5. **A separação Q02-E / NL_***: Q02-E (progress=99% reconnect) NÃO
   é mapeado na tabela porque NÃO é error code; é estado de fluxo.
   Docstring do módulo (linhas 21-26) explicita a separação; quirk é
   tratado em layer apropriada (`download_primitive` Story 1.3 +
   `CircuitBreaker` Story 2.6, NÃO `error_taxonomy.py`).
6. **Q02-E status atualizado para `validated`** em `docs/dll/QUIRKS.md`
   com workaround formalizado em duas camadas (Story 1.3 timeout duro
   + Story 2.6 breaker progress-aware). Test reprodutor existe e
   passa.
7. **Cada entry da tabela tem justificativa em prosa** — investimento
   no audit trail de longo prazo. Daqui a 6 meses, qualquer agente
   que questionar uma decisão tem a resposta inline.
8. **`is_retryable` é função conservadora** — retorna True APENAS para
   TRANSIENT óbvio. AMBIGUOUS exige análise política (cap diferente)
   e por isso fica fora de `is_retryable`, forçando caller a usar
   `categorize_nl` + decidir baseado em `RetryPolicy.retryable_categories`.

**Esta auditoria difere de `wrapper_review`** (que aplica os 22 itens
manual + ctypes + INV-1) — Nelo está homologando **tabela de
classificação semântica**, não wrapper de função DLL. Os 22 itens do
checklist `nl_taxonomy_review` (custom desta story) cobrem exatamente
o que Nelo exige: cobertura completa, justificativa em prosa,
classificação semanticamente correta, defesa R7 conservadora,
separação NL_* vs estado de fluxo (Q02-E).

**Próxima ação:** Aria pode prosseguir para `*review-design` da
RetryPolicy + CircuitBreaker (`2.6-design-2026-05-04.md`).
