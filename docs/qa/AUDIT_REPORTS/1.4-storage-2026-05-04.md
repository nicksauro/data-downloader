## 4. Achados (Findings)

### F-1 — MEDIUM — Catálogo SQLite não integrado (deferred Story 1.5)

- **Arquivo:** `src/data_downloader/storage/parquet_writer.py` (todo o pipeline)
- **Descrição:** O writer escreve Parquet atomicamente mas **não** registra a
  partição em `data/history/catalog.db`. Sem catálogo, projetos downstream
  precisam scan filesystem para saber "o que existe" — viola Lei R6 (catálogo
  é fonte única de verdade) e impede o protocolo two-phase commit emulado
  (INTEGRITY.md §4: INSERT em `_pending_commits` antes do `os.replace` →
  registrar em `partitions` após). Em caso de crash entre `os.replace` e
  registro, o arquivo existe mas o catálogo não sabe — recovery boot não
  consegue reconciliar via `_pending_commits` órfão (porque tabela não existe
  ainda).
- **Manual ref:** `docs/storage/SCHEMA.md` §5 (catálogo DDL completo);
  `docs/storage/INTEGRITY.md` §4 (two-phase commit), §5 (drift report).
- **Sugestão de fix:** Story 1.5 implementa `storage/catalog.py` com:
  1. Boot: cria `catalog.db` com PRAGMAs (SCHEMA.md §5) e DDL.
  2. Writer: hook pré-replace (INSERT em `_pending_commits(expected_sha256,
     expected_size, pid)`) + hook pós-replace (INSERT/UPDATE em `partitions` +
     DELETE de `_pending_commits` em transação).
  3. Boot: rotina `recover_pending_commits()` (INTEGRITY.md §4.3).
- **Regression test sugerido:**
  `tests/integration/test_storage_catalog_atomicity.py::test_crash_between_replace_and_catalog_recovers`.
- **Severity rationale:** MEDIUM (não CRITICAL) porque Story 1.4 está
  explicitamente escopada para writer + reader; catálogo é Story 1.5
  documentada e priorizada. Sem catálogo nesta gate, perda de dado é nula
  (arquivo Parquet sobrevive a crash); apenas reconciliação manual é
  necessária. Bloqueia produção (não pode sair sem catálogo) mas não bloqueia
  PASS de 1.4 isoladamente.

### F-2 — MEDIUM — `sha256_self` ausente do metadata Parquet (deferred)

- **Arquivo:** `src/data_downloader/storage/parquet_writer.py:171-183`
  (`_build_metadata`)
- **Descrição:** SCHEMA.md §4 requer chave `sha256_self` no metadata Parquet
  (redundância com `partitions.checksum_sha256` — finding M4). Atualmente o
  checksum é calculado (linha 327) e exposto em `WriteResult` mas não escrito
  em `_build_metadata`. Justificativa Sol §4 reconhece o problema chicken-and-
  egg (hash dos bytes do arquivo precisa estar dentro do arquivo): solução é
  escrita em duas passadas (write tmp → hash tmp → reescrever metadata + atomic
  replace).
- **Manual ref:** `docs/storage/SCHEMA.md` §4 (NOTA IMPORTANTE),
  `docs/storage/INTEGRITY.md` §3.
- **Sugestão de fix:** Implementar two-pass write em Story 1.5 ou 2.X:
  1. Escreve tmp com `_build_metadata()` sem `sha256_self`.
  2. Calcula SHA256 do tmp.
  3. Escreve tmp2 com metadata completo (incluindo `sha256_self`).
  4. `os.replace(tmp2, final)`.
  Custo: ~2x I/O na escrita; aceitável para garantia de auditoria.
- **Regression test sugerido:**
  `tests/integration/test_storage_writer_reader.py::test_metadata_has_sha256_self`.
- **Severity rationale:** MEDIUM porque a redundância secundária (catálogo)
  está coberta pelo `WriteResult.checksum_sha256` (caller pode persistir);
  sem `sha256_self` no metadata, auditoria forense de arquivo isolado (sem
  catálogo disponível) fica fragilizada.

### F-3 — LOW — Threshold de rewrite usa rows (5M) em vez de bytes (256MB)

- **Arquivo:** `src/data_downloader/storage/parquet_writer.py:73, 276-287`
- **Descrição:** AC8 da story menciona "se arquivo existente > N MB (default
  256MB, configurável), em vez de reescrever, criar `MM.{N}.parquet`
  (sequência incrementada)". Implementação usa `_PARTITION_ROW_LIMIT =
  5_000_000` rows e **raise `IntegrityError`** em vez de criar
  `MM.{N}.parquet`. Comportamento determinístico (Sol prefere bloquear a
  silenciar) mas diverge da AC e do finding H6 do plan review.
- **Manual ref:** `docs/stories/1.4.story.md` AC8;
  `docs/decisions/PLAN_REVIEW_2026-05-03.md` finding H6.
- **Sugestão de fix:** Story 2.X implementa sub-particionamento real
  (`MM.{N}.parquet` com `glob "MM.*.parquet"` no reader). Threshold deve usar
  `file_size_bytes` (256MB default), configurável via env var. Atual
  `_PARTITION_ROW_LIMIT` é proxy razoável (5M trades ≈ 150MB Snappy) mas
  documentar como interim.
- **Regression test sugerido:**
  `tests/integration/test_storage_partition_overflow.py::test_creates_subpartition_when_size_exceeded`.
- **Severity rationale:** LOW — comportamento atual é seguro (bloqueia em vez
  de degradar O(n²)), apenas não otimiza. Para Story 1.4 (smoke 30 dias WDOJ26
  ≈ ~10M trades / mês ≈ 1 partição = 1 arquivo), o limite 5M é levemente
  apertado. Pyro pode rebalancear no benchmark 1.4.5.

### F-4 — LOW — Cobertura de erro paths em `_fsync_directory` e cleanup

- **Arquivo:** `src/data_downloader/storage/parquet_writer.py:121-127, 338-348`
- **Descrição:** Linhas 121-127 (warning de `fsync_directory.unsupported`) e
  338-348 (cleanup de tmp em exceção) não estão cobertas pelos testes (vide
  cov report: linhas 121-127 e 338-348 missing). São paths defensivos
  (Windows fsync(dir_fd) + cleanup de exceção rara), mas baixa cov nesses
  ramos esconde regressões silenciosas.
- **Manual ref:** `docs/qa/TEST_PYRAMID.md` (cobertura de erro paths).
- **Sugestão de fix:** Adicionar testes em Story 2.1 (data validators):
  - mockar `pq.ParquetWriter.write_table` para raise → verificar que tmp é
    removido.
  - testar `_fsync_directory` com diretório read-only para exercitar branch
    de warning.
- **Regression test sugerido:**
  `tests/unit/test_storage_writer_error_paths.py`.
- **Severity rationale:** LOW — paths defensivos; cobertura módulo já em
  91.67% (>= 80% requerido).

**Justificativa:** Implementação está correta e segura para o escopo declarado
de Story 1.4 (writer + reader, sem catálogo).

- Schema v1.0.0 implementado bit-a-bit conforme SCHEMA.md (17 campos, tipos,
  nullability).
- Atomicidade respeitada: pipeline tmp → fsync(file) → SHA256 → fsync(dir) →
  `os.replace` → fsync(dir again). `os.replace` é atômico em Windows + Linux.
- Idempotência demonstrada: integration test (write 2x → arquivo único de N
  trades) + Hypothesis property INV-2.
- `schema_version` no metadata: PASS (verificado em integration test).
- Append+dedup: lê existente, union, dedup, reescreve atomicamente — preserva
  Lei R5.
- Bloqueio determinístico em overflow (>5M rows raise IntegrityError) é
  preferível a degradação O(n²) silenciosa.
- 4 findings (2 MEDIUM, 2 LOW) — todos com fix path documentado em stories
  futuras (1.5, 2.X). Nenhum bloqueia este escopo.

Os MEDIUMs (F-1: catálogo, F-2: `sha256_self`) são **deferred-by-scope** — o
plano explicitamente coloca catálogo em Story 1.5 e two-pass write em Story
2.X. Aceito como dívida tracked.
