## 4. Achados (Findings)

### F-1 — LOW — `_apply_migrations` não exercita branch de skip de versão antiga

- **Arquivo:** `src/data_downloader/storage/catalog.py:377-378` (loop `for version, ddl_statements in MIGRATIONS:`)
- **Descrição:** O check `if current_version is not None and _semver_le(version, current_version): continue` (linha 378) é defensivo para o cenário "já estamos em v1.1.0, não re-aplicar v1.0.0". Como `MIGRATIONS` tem apenas v1.0.0 hoje, esse branch nunca é exercitado pelos testes atuais. Não bloqueia (idempotência geral é provada via `test_init_is_idempotent_no_duplicate_schema`), mas a primeira mudança aditiva (Story que adiciona v1.1.0) deve incluir teste explícito.
- **Manual ref:** `docs/storage/MIGRATIONS.md` (framework); `docs/storage/SCHEMA.md` §6 (política minor/major).
- **Sugestão de fix:** Quando primeira migration v1.1.0 entrar (e.g., adição de campo aditivo no catálogo, ou tabela auxiliar nova), adicionar test `test_apply_migrations_skips_already_applied_versions` em `test_catalog_init.py` que registra `catalog_version=1.0.0` num DB existente, depois adiciona `("1.0.0", _DDL_V1_0_0)` + `("1.1.0", _DDL_V1_1_0_DELTAS)` e verifica que apenas v1.1.0 roda (DDL v1.0.0 não é re-executado).
- **Regression test sugerido:** `tests/unit/test_catalog_init.py::test_apply_migrations_skips_already_applied_versions`.
- **Severity rationale:** LOW — código seguro (idempotente sob `IF NOT EXISTS` em todas as DDL); risco apenas em cenário hipotético onde futura DDL não é idempotente. Tracking para Story de bump de catalog_version.

### F-2 — LOW — `_auto_register_from_disk` faz `import pyarrow.parquet` lazy dentro da função

- **Arquivo:** `src/data_downloader/storage/catalog.py:1029-1031` (`def _auto_register_from_disk(...)`)
- **Descrição:** O import `import pyarrow.parquet as pq` está dentro do método (lazy import) em vez de top-level. Isso é eficiente (evita custo de import quando reconcile não detecta drift A) mas viola convenção PEP 8 + estilo do projeto (top-level imports em outros módulos storage). Mais grave: defesa de erro do import (e.g., pyarrow corrompido) cai em `(OSError, ValueError)` que NÃO captura `ImportError` — se pyarrow faltar, propaga `ImportError` em vez de log warning + `return False`.
- **Manual ref:** Convenção interna `data_downloader.storage` — `parquet_writer.py` e `duckdb_reader.py` fazem `import pyarrow.parquet as pq` no topo.
- **Sugestão de fix:** Mover import para topo do módulo (`from data_downloader.storage.parquet_writer` já depende de pyarrow; nada novo). Se manter lazy por razão de boot perf, capturar `ImportError` explicitamente: `except (OSError, ValueError, ImportError) as exc:`.
- **Regression test sugerido:** `tests/unit/test_catalog_reconcile.py::test_auto_register_handles_pyarrow_import_failure` (mock `__import__` para raise ImportError; verificar log warning + `return False`).
- **Severity rationale:** LOW — pyarrow é dependência hard do projeto (sem ele, `parquet_writer.py` falha primeiro). Risco apenas em ambiente de desinstalação parcial. Não bloqueia.

### F-3 — LOW — Cobertura de erro paths em `_auto_register_from_disk` (linhas 1033-1078)

- **Arquivo:** `src/data_downloader/storage/catalog.py:1033-1078` (multiple defensive branches)
- **Descrição:** Branches de erro em `_auto_register_from_disk` (metadata read fail, no metadata, parse fail, invalid layout, parse failed, stat failed) são paths defensivos para arquivos Parquet corrompidos / com layout inválido. Cobertura agregada do módulo é 82% (catalog.py); algumas linhas faltantes incluem 1033-1038 (metadata_read_failed), 1041-1042 (no_metadata), 1049-1054 (metadata_parse_failed), 1060-1061 (invalid_layout), 1067-1069 (parse_failed), 1073-1078 (stat_failed). São paths defensivos, mas baixa cov nesses ramos esconde regressões silenciosas em recovery boot.
- **Manual ref:** `docs/qa/TEST_PYRAMID.md` (cobertura de erro paths); INTEGRITY.md §5.1 (drift A com metadata corrompida vira candidato a quarantine futura).
- **Sugestão de fix:** Adicionar 6 testes em Story 2.1 (data validators) cobrindo cada branch:
  1. `test_auto_register_skips_corrupt_metadata` (escrever arquivo `.parquet` com bytes randômicos).
  2. `test_auto_register_skips_no_metadata` (escrever Parquet vazio sem custom metadata).
  3. `test_auto_register_skips_invalid_metadata_values` (metadata com `row_count="abc"`).
  4. `test_auto_register_skips_invalid_layout` (arquivo em `data/history/F/file.parquet` — só 2 níveis).
  5. `test_auto_register_skips_invalid_year_month` (`F/WDOJ26/abcd/01.parquet`).
  6. `test_auto_register_skips_when_stat_fails` (mock stat para raise OSError).
- **Regression test sugerido:** `tests/integration/test_catalog_auto_register_error_paths.py`.
- **Severity rationale:** LOW — paths defensivos; cobertura módulo já em 84.80% (>= 80% requerido). Em produção, drift A com metadata corrompida seria reportado mas não auto-corrigido (comportamento correto).

### F-4 — LOW — `register_partition` recalcula `_get_meta("parquet_schema_min_supported")` por chamada

- **Arquivo:** `src/data_downloader/storage/catalog.py:661` (`schema_version = self._get_meta(...) or "1.0.0"`)
- **Descrição:** Cada chamada a `register_partition` faz uma query SQLite extra para ler `_schema_meta('parquet_schema_min_supported')`. Em produção (Story 1.7 orchestrator com chunks de 500k trades), isso é 1 query por partição — irrelevante. Mas o valor é constante durante a vida da instância `Catalog`; poderia ser cacheado em `__init__` para clareza/perf (microbench).
- **Manual ref:** SCHEMA.md §5.1 (`parquet_schema_min_supported` definido em init via INSERT ON CONFLICT DO NOTHING).
- **Sugestão de fix:** Em `__post_init__` (após `_apply_migrations`), cachear `self._parquet_schema_min_supported = self._get_meta("parquet_schema_min_supported") or "1.0.0"`. Usar atributo em `register_partition`.
- **Regression test sugerido:** N/A (otimização cosmética; comportamento idêntico).
- **Severity rationale:** LOW — overhead < 1ms/chamada (SQLite indexed read em tabela 4-row). Marginal. Tracking opcional.

  Property tests Hypothesis (2 properties × 15 examples cada) provam

  drift B (arquivo deletado externamente) e C (checksum diverge) APENAS

- **WAL checkpoint após cada register_partition (AC12)** ativo (linha 676):
  `PRAGMA wal_checkpoint(TRUNCATE)` força flush do WAL para o DB principal,
  trade-off ~10ms vs durabilidade — aceitável pois partições escrevem em
  granularidade mensal (não por trade). Falha do checkpoint loga warning
  mas não rollback (commit em `partitions` já persistiu).
- **Resume (AC8)** com `compute_pending_chunks` (granularidade mensal,
  alinhada ao layout ADR-004): 6 testes cobrindo casos canônicos + edge
  (range invertido = vazio; cobertura total = vazio; filtro por
