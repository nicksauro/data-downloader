## 4. Achados (Findings)

### F-1 — INFO — `bench_callback_to_disk` não simula DLL real (esperado)

- **Arquivo:** `benchmarks/bench_callback_to_disk.py` (pipeline mock)
- **Descrição:** O benchmark usa thread separada para emular callbacks DLL, mas
  não exercita o ConnectorThread real (que tem timing próprio + callback C
  via `WINFUNCTYPE`). Diferenças possíveis: (a) overhead de marshalling C↔Python
  via ctypes, (b) thread affinity diferente, (c) timer resolution Windows vs
  Linux. Isto é **esperado** para Story 1.4.5 (synthetic baseline) e
  documentado em F4 do COUNCIL-02 ("Bench precisa rodar com n_callbacks maior
  + sustained rate para validar H4 propriamente — escopo Story 1.7 (real
  ingestor + worker process)").
- **ADR/INV ref:** ADR-005 (thread model), INV-1 (callback DLL → queue.put_nowait),
  INV-3 (toda fila tem maxsize + política).
- **Sugestão de fix:** Story 1.7 (orchestrator real) deve re-rodar `bench_callback_to_disk`
  com DLL real (gated por `PROFITDLL_KEY`) para validar baseline real. Story 1.8
  (real DLL E2E) registrará v1.0.0-real em BASELINES.md.
- **Trade-off alternativo (se aplicável):** N/A — synthetic baseline é o passo
  correto antes de DLL real (custo + reprodutibilidade).

### F-2 — INFO — COUNCIL-02 não convocou Aria diretamente — Aria sign-off agora

- **Arquivo:** `docs/decisions/COUNCIL-02-parquet-writer-streaming-overhead.md` §
  "Aria consultation"
- **Descrição:** COUNCIL-02 declarou explicitamente: "Esta decisão NÃO altera
  fronteira de camada. Otimizações futuras (Opção B) ficam dentro de
  `data_downloader.storage.parquet_writer` — sem novo contrato externo. Aria
  consultada apenas se Opção C escalada." Aria concorda com a justificativa:
  Opção B (vectorização interna) é mudança de implementação, NÃO de fronteira.
  Não exige ADR amendment. Mas a recomendação para Story 1.7 (queue=100k
  default + métrica `dll_drops_total`) **toca fronteira** (configuração de
  bounded queue ADR-005) e merece registro arquitetural — minor amendment ao
  ADR-005 quando Story 1.7 for implementada.
- **ADR/INV ref:** ADR-005 (thread model bounded queues — `dll_queue maxsize=10000`
  default atual; recomendação muda para 100_000 para absorver writer pause).
- **Sugestão de fix:** Story 1.7 deve incluir um pequeno ADR amendment a
  ADR-005: bumpar default `dll_queue maxsize` de 10_000 para 100_000 (com
  override via env var `DATA_DOWNLOADER_DLL_QUEUE_SIZE`, range [1k, 1M] —
  já documentado em INTEGRITY.md §7). Pyro recomenda no COUNCIL-02
  recomendação 2; Aria endossa.
- **Trade-off alternativo (se aplicável):** Manter default 10_000 e exigir
  que CLI/configuração de produção bumpe — pior, porque deixa default vulnerável
  a drops em ambiente que não configurou. Default deve ser robusto.

### F-3 — INFO — Recomendação 6 do COUNCIL-02 (revisar target callback→disk p99) merece registro formal

- **Arquivo:** `docs/decisions/COUNCIL-02-*.md` § "Decisão tomada" item 6
- **Descrição:** COUNCIL-02 recomenda substituir target `latency_callback_to_disk_p99 < 100ms`
  por 3 sub-targets: (a) "0 drops sob carga normal", (b) "p99 callback→queue
  acceptance < 1ms", (c) "p99 chunk→disk (write completes) < 30s para chunk 500k".
  Esta é uma **revisão de target arquitetural** — toca contrato implícito de
  performance que projetos downstream poderiam esperar.
- **ADR/INV ref:** TARGETS_V1.md (target atual ainda diz "< 100ms" mas marca
  como `gap`); precisa ser revisado formalmente.
- **Sugestão de fix:** Story 1.7 (orchestrator) deve atualizar TARGETS_V1.md
  com os 3 sub-targets (a/b/c) substituindo o target original. Aria endossa
  arquiteturalmente: o target original não fazia distinção entre latência de
  callback (responsividade do processo) e latência de write (throughput de
  serialização) — separar é mais correto.
- **Trade-off alternativo (se aplicável):** Manter target único "< 100ms" e
  marcar `gap` permanentemente — pior, porque cria expectativa errada e mascara
  decomposição arquitetural válida. A decomposição em 3 sub-targets é arquiteturalmente
  superior.

## 5. Decisão

| Verdict             | Marcar |
|---------------------|--------|
| ✅ APPROVED         | [X]    |
| 🟡 CHANGES_REQUESTED | [ ]    |
| 🔴 BLOCKED          | [ ]    |

**Justificativa:** Story 1.4.5 entrega **baselines honestos** (synthetic) sem
mascarar gaps. Os benchmarks respeitam o thread model ADR-005 (queue + writer
thread separados), não tocam fronteiras de camada (não modifica `src/`),
não introduzem nova dependência transversal, não alteram schema nem wrapper DLL,
e respeitam o hot path R21 (sample_rate=5% para evitar overhead de medição).

Os 2 verdicts FAIL nos baselines (`bench_parquet_write` production gap -72% e
`bench_callback_to_disk` chunk-mode gap +22x) são **realidade arquitetural**,
não defeito do benchmark. COUNCIL-02 documenta a causa raiz (validate+enrich+
sequence+dedup+merge+sort+fsync+sha256 todos em loop Python no `ParquetWriter`)
e propõe remediação correta:

1. **Story 2.1 (perf-write-optimization)** — vectorizar `_trades_to_table`
   + `validate_record` + `dedup` (Opção B passos 1-3). Esperado fechar gap
   para ~150-200k trades/s (atinge target V1).
2. **NÃO Opção C** (streaming-append separado) — preserva atomicidade single-file
   e contratos atuais (INV-3 read-after-write consistency).
3. **Story 1.7 (orchestrator)** — métricas `dll_drops_total`,
   `ingest_queue_depth`, `write_queue_depth` + queue default 100k.
4. **Revisar target `latency_callback_to_disk_p99`** em Story 1.7 (3 sub-targets
   decompostos).

3 findings INFO documentam:
- F-1: limitação esperada do mock (validar com DLL real Story 1.7/1.8).
- F-2: COUNCIL-02 não convocou Aria diretamente, mas decisão NÃO cruza fronteira
  (justificado); recomendação queue=100k para Story 1.7 merece ADR amendment
  pequeno.
- F-3: revisão de target callback→disk p99 (3 sub-targets) merece registro
  formal em TARGETS_V1.md durante Story 1.7.
