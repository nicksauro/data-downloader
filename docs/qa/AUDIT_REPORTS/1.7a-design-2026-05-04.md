## 2. Escopo

- **Em escopo:**
  - **State machine** (`state_machine.py`) — implementação do
    amendment ADR-005 (2026-05-03 — `Idle/Running/DrainingDLL/DrainingWrite/Committed`)
    + **estado FAILED extra** que Dex adicionou como terminal
    alternativo. Validação de transições, `threading.Lock`,
    callback `on_change`.
  - **Chunker** (`chunker.py`) — pure function `chunk_date_range`
    com lookup por prefixo (CHUNK_DAYS map: WDO/WIN/IND/DOL=5,
    equity=1) sobre business days B3.
  - **Retry** (`retry.py`) — `with_retry` com backoff exponencial
    (1s, 4s, 16s) + jitter ±20% + `RetryError`. Erros transientes
    (OSError, TimeoutError) vs fatais (ValueError, KeyboardInterrupt).
  - **Orchestrator** (`orchestrator.py`) — `Orchestrator.run(config,
    *, resume_job_id=None) -> JobResult`. Pipeline: register_job
    → resolve contract → cache hit check → loop chunks (with retry
    + state machine).
  - **Cache hit semântica REAL** (range coverage por mês) — finding
    H8 fechado.
  - **correlation_id = job_id** em todos os logs (finding L2 fechado).
  - **Métricas via structlog events** (V1) — alinhado com ADR-013
    (deferred Prometheus V2).
- **Fora de escopo:**
  - CLI typer (Story 1.7b).
  - public_api facade (`Downloader` class) — Story 1.7b.
  - Smoke real contra DLL (Story 1.7b).
  - Schema interno do Parquet (Sol — não tocado).
  - Wrapper DLL (Nelo — não tocado).
  - DDL do catálogo (Story 1.5 — não tocado).
- **Componentes tocados:**
  - `orchestrator/` — `state_machine.py` (210 linhas NOVO),
    `chunker.py` (145 linhas NOVO), `retry.py` (165 linhas NOVO),
    `orchestrator.py` (620 linhas NOVO), `__init__.py` (re-exports).
- **Cruza fronteiras?** **SIM** — primeira composição completa de
  4 camadas em runtime: `dll/` (via download_chunk Story 1.3) →
  `orchestrator/` (esta story) → `storage/parquet_writer/` (Story
  1.4) + `storage/catalog/` (Story 1.5) + `storage/contracts/`
  (Story 1.6 lookup). Direção respeita ADR-007a — orchestrator
  IMPORTA dessas camadas; nenhuma camada importa de orchestrator.

## 4. Achados (Findings)

### F-A-1 — INFO — Cache hit pula state machine (vai direto para `cache_hit` sem RUNNING)

- **Arquivo:** `src/data_downloader/orchestrator/orchestrator.py:304-330`
- **Descrição:** Quando cache hit é detectado, orchestrator transita
  `IDLE → RUNNING` mas **NÃO** transita para `DRAINING_DLL/WRITE/COMMITTED`
  antes de retornar — apenas chama `update_job_progress(status="completed")`
  e retorna `JobResult(status="cache_hit")`. O comentário (linhas 319-321)
  reconhece: "Cache hit pula DRAINING — direto para COMMITTED via
  FAILED-skip. Por simplicidade, não usa state machine no cache hit
  (nada para drenar)". Esta divergência do happy path documentada
  pode confundir observadores externos (gauge `orchestrator_state`
  vai mostrar `RUNNING` para sempre se gauge é amostrado durante
  cache hit — embora isso seja improvável dado o cache hit ser síncrono
  e instantâneo).
- **ADR/INV ref:** ADR-005 amendment 2026-05-03 (state machine);
  ADR-005 amendment 2026-05-04 (FAILED state).
- **Sugestão de fix:** (1) Ratificar a divergência em ADR-005
  amendment v3 ("estado RUNNING auto-conclui se cache_hit detectado
  imediatamente após resolve_contract"); OU (2) transicionar
  explicitamente `RUNNING → DRAINING_DLL → DRAINING_WRITE → COMMITTED → IDLE`
  no cache hit path (5 transições "vazias" mas observáveis). Aria
  prefere (1) — simplicidade vence; cache_hit é estado terminal
  alternativo legítimo (semelhante a FAILED). Tracking: Story 2.X
  (refinamento da state machine para multi-symbol Epic 4).
- **Trade-off alternativo:** —
- **Severity rationale:** INFO — não bloqueia; documentação da
  divergência em ADR-005 amendment v3 fecha. Nenhum impacto em
  correctness.

### F-A-2 — LOW — `_handle_fatal_error` faz transição em 2 hops (FAILED + force_idle) sem atomicidade

- **Arquivo:** `src/data_downloader/orchestrator/orchestrator.py:723-732`
- **Descrição:** `_handle_fatal_error` chama `sm.transition(JobState.FAILED)`
  seguido por `sm.force_idle()` — duas operações separadas, ambas
  com lock. Se outra thread (improvável em V1, mas plausível em V2
  multi-symbol) consultar `sm.state` entre as duas chamadas,
  observará `FAILED` momentâneo. Não causa bug funcional (estado
  é consistente em cada lock-window), mas viola atomicidade
  semântica de "tratar erro" como única operação.
- **ADR/INV ref:** ADR-005 amendment 2026-05-04 (FAILED state);
  INV-11 (state machine thread-safe).
- **Sugestão de fix:** Adicionar método `JobStateMachine.fail_and_idle()`
  que faz ambas transições sob o mesmo lock. Tracking: Story 2.X
  (refinamento). V1 OK porque apenas OrchestratorThread usa o sm
  (Ingestor/Writer apenas leem `state` snapshot).
- **Trade-off alternativo:** Manter 2 transições separadas (atual)
  → simplicidade + mais eventos no log para debug; OU método
  composto → menos eventos mas semântica atômica. Aria prefere o
  composto para V2 multi-symbol; V1 OK como está.
- **Severity rationale:** LOW — defesa em profundidade; sem bug
  atual.

### F-A-3 — LOW — `metrics.callbacks_received` mede `len(chunk_result.trades)` em vez de callbacks DLL reais

- **Arquivo:** `src/data_downloader/orchestrator/orchestrator.py:159-160, 643`
- **Descrição:** Atributo `callbacks_received` é incrementado por
  `len(chunk_result.trades)` por chunk. Mas `chunk_result.trades`
  é a lista FILTRADA (após dedup + translate em IngestorThread Story
  1.3) — NÃO o número de callbacks DLL reais (que pode ser maior
  por causa de TC_IS_EDIT, TC_LAST_PACKET, ou dropped if queue full).
  Nome induz a erro — operador pode usar este número para auditar
  "quantos eventos a DLL gerou" e obterá número subestimado.
- **ADR/INV ref:** ADR-013 (observability — métricas reservadas V2);
  audit Story 1.3 §3.10 (callback observability).
- **Sugestão de fix:** Renomear para `trades_in_chunks_total` (mais
  preciso) ou expor SEPARADO `callbacks_received_total` que
  download_chunk reporta via novo campo `ChunkResult.callbacks_total`
  (Story 1.3 extensão Story 2.X). Aria prefere renomear nesta story
  (custo: 1 commit) OU documentar a semântica no docstring. Tracking:
  Story 2.X (Pyro pode usar para validar overhead de translate).
- **Trade-off alternativo:** —
- **Severity rationale:** LOW — naming/documentação; não bug
  funcional.

### F-A-4 — LOW — `_compute_chunks` resume path usa expansão por mês inflada

- **Arquivo:** `src/data_downloader/orchestrator/orchestrator.py:516-563`
- **Descrição:** No caminho resume, `_compute_chunks` itera mês-por-mês
  e gera chunks ATÉ `config.end` mesmo se a granularidade do catálogo
  é mensal (Story 1.5 partition layout). Isso pode gerar chunks
  redundantes na fronteira (e.g. último dia útil do mês previous
  + primeiro dia útil do current month, se o mês previous estava
  completo). Logic correctness não viola (writer faz dedup), mas
  custa I/O extra de DLL.
- **ADR/INV ref:** ADR-004 (partition layout monthly); INV-12 (fim
  de chunk).
- **Sugestão de fix:** Em vez de iterar mês-por-mês, calcular
  `set(complete_months) ∩ range_meses` e gerar chunks APENAS para
  meses NÃO em complete_months. O loop atual já faz isso (linha 533
  `if (year, month) in completed_keys: pass`) — mas re-emite chunks
  para o mês CORRENTE mesmo se ele tem partição parcial registrada.
  Em V1 OK (writer dedup + cache hit por month garantem
  correctness); em V2 multi-day-resume seria desperdício. Tracking:
  Story 2.X.
- **Trade-off alternativo:** —
- **Severity rationale:** LOW — correctness preservada; custo I/O
  marginal em V1.

### F-A-5 — INFO — `OrchestratorMetrics` é mutável (intencional) mas fica embedded em `JobResult` frozen

- **Arquivo:** `src/data_downloader/orchestrator/orchestrator.py:151-208`
- **Descrição:** `JobResult` é `@dataclass(frozen=True)` mas seu
  campo `metrics: OrchestratorMetrics` (default factory) é uma
  classe MUTÁVEL (`@dataclass` sem frozen). Caller pode mutar
  `result.metrics.chunks_failed = 999` e o "imutável" `JobResult`
  registra a mutação. Decisão deliberada (linha 156 docstring:
  "mutável intencionalmente — acumulador. Snapshot imutável é
  embutido em JobResult ao final"), mas o snapshot NÃO é embutido —
  é a mesma referência.
- **ADR/INV ref:** ADR-007a (public_api SemVer — frozen vs mutable);
  agents/architect.md `core_principles` "PÚBLICO ≠ INTERNO".
- **Sugestão de fix:** (1) Tornar `OrchestratorMetrics` também frozen
  e usar `dataclasses.replace` no acumulador (mais alocações mas
  imutável real); OU (2) fazer deep copy de `metrics` ao construir
  `JobResult` (snapshot literal); OU (3) documentar o trade-off no
  docstring (`metrics` é referência viva — caller NÃO deve mutar
  após receber). Aria prefere (3) para V1 e (1) para V2 quando
  expor `JobResult` em public_api (Story 1.7b). Tracking: Story
  1.7b.
- **Trade-off alternativo:** —
- **Severity rationale:** INFO — design trade-off documentável.

**Justificativa:** Story 1.7a entrega o **core do orchestrator**
com:

   integralmente** (`Idle → Running → DrainingDLL → DrainingWrite
   → Committed → Idle`) + **estado FAILED ratificado em ADR-005
   amendment v2 (2026-05-04)** como terminal alternativo.
   Transições explícitas em `VALID_TRANSITIONS`, `threading.Lock`
   thread-safe, callback `on_change` para observabilidade externa.
   16 testes unit cobrem todas as transições + concorrência.
2. **COUNCIL-05 D1-D9 todos corretos** — D1 state machine
   (PASS — implementação fiel ao amendment + extensão FAILED
   ratificada), D2 queue 100k (PASS — herdado de Story 1.3),
   D3 métricas via structlog (PASS — alinhado com ADR-013 deferred
   Prometheus V2), D4 chunking 5d futuros / 1d equity (PASS —
   `CHUNK_DAYS` map), D5 retry 3 tentativas exponencial+jitter
   (PASS — `with_retry`), D6 resume via `Catalog.resume_job` (PASS
   — Story 1.5 API), D7 cache hit range coverage REAL (PASS —
   `_is_full_cache_hit` granularidade mensal), D8 correlation_id
   = job_id (PASS — em todo log structlog), D9 logging events
   canônicos per-chunk (PASS — R21).
3. **INV-11 (separação física)** preservada — OrchestratorThread
   (caller de `Orchestrator.run`) ≠ IngestorThread/ProgressMonitor
   (criadas internamente por `download_chunk` Story 1.3) ≠
   ConnectorThread (DLL). Nomes únicos.
4. **INV-12 ("fim de chunk")** implementada via state transitions
   `DRAINING_DLL → DRAINING_WRITE → COMMITTED` — só transita após
   `writer.write` retornar OK E `catalog.register_partition`
   commitar (Story 1.5 AC12 garante WAL checkpoint).
5. **Cache hit semântica REAL (range coverage)** — `_is_full_cache_hit`
   itera (year, month) em `[start, end]` e exige que TODOS os meses
   tenham partição registrada. Distingue "partição existe" de
   "range coberto" — **finding H8 fechado**.
6. **correlation_id = job_id** em TODO log structlog (`job_id=...`)
   — começando em `register_job`, agregando logs do mesmo run.
   `chunk_id` (vindo de `download_chunk`) é sub-correlação per-chunk.
   **Finding L2 fechado**.
7. **Métricas via structlog (V1)** — alinhado com ADR-013 (deferred
   Prometheus V2 sem mudança de API). 6 métricas tracked

   `chunks_failed`, `dll_drops_total`, `state_transition`).
8. **66 testes novos** (16 unit state_machine + 24 unit chunker
   incluindo 3 property + 12 unit retry + 12 integration
   orchestrator + 2 property idempotência E2E) — todos PASS.
   Suite total **390 passed + 1 skipped** em 199s (Python 3.14).
9. **ruff CLEAN, mypy strict CLEAN** nos 4 arquivos novos.
10. **AC8 e AC9 consolidados** em `tests/integration/test_orchestrator.py`

    redução de fricção de manutenção.
