### 3.2 storage_pr_review

> Origem: `agents/storage-engineer.md` → `checklists.storage_pr_review`

- [x] **Idempotência preservada (re-rodar não duplica)?** **SIM.** Tripla defesa:
  1. `MigrationRunner.execute(run_id=X)` re-execução com mesmo `run_id` pula partições já em `status='migrated'` via `_is_already_migrated` (`_runner.py:559-568`) — checkpoint resumível AC5.
  2. `Migration.applies_to(current_version)` retorna `False` se a partição já está em `to_version` — runner não re-aplica (R5).
  3. `V100ToV110.transform` é no-op se `liquidity_classification` já existe na table (`v1_0_0_to_v1_1_0.py:62-63`) — defensivo.
- [x] **Append-only respeitado (sem overwrite silencioso)?** **SIM com semântica controlada.** Migration sobrescreve em disco propositalmente — é a operação que ela executa — mas:
  1. Backup `.parquet.bak` é criado **antes** do overwrite (`_runner.py:469-492`, AC7), preservando o original byte-a-byte.
  2. Overwrite usa `os.replace` atômico via `ParquetMigration.write_new` (`_base.py:296`) — NÃO é truncate+write inline; é tmp + fsync + replace, espelhando o pipeline do `ParquetWriter` (Story 1.4).
  3. CLI consumer-facing (`migrate plan`) é dry-run by default; `migrate execute` exige confirmação (microcopy `migration.confirm`).
- [x] **Catálogo atualizado na mesma transação lógica?** **SIM.** Após `verify()` passar, runner atualiza:
  1. `partitions.schema_version` (`_update_catalog_schema_version` — `_runner.py:598-605`) dentro de `with self.catalog._transaction()`.
  2. `partitions.checksum_sha256` + `file_size_bytes` (`_update_catalog_checksum` — `_runner.py:607-631`) — recalcula SHA256 do arquivo migrado.
  3. `_checksum_cache` invalidado via DELETE explícito (`_runner.py:628-630`) porque CASCADE FK não dispara em UPDATE.
  4. `_migration_log` recebe `mark_migrated` (`_record_log` / `_update_log` — `_runner.py:513-557`) também dentro de `_transaction()`.
  Two-phase commit espelhado do path de write da Story 1.5.
- [x] **`schema_version` escrito no metadata Parquet?** **SIM.** `ParquetMigration._build_metadata` (`_base.py:227-252`) força `b"schema_version": self.to_version.encode("utf-8")` + `b"migrated_from": self.from_version.encode("utf-8")` (auditoria de qual migration produziu o arquivo). Verify default checa metadata `schema_version` (`_base.py:185-195`); V100ToV110 estende verify para confirmar campo `liquidity_classification` (`v1_0_0_to_v1_1_0.py:71-99`).
- [x] **Checksum calculado e armazenado?** **SIM.** `_update_catalog_checksum` recalcula via `_sha256_file` (importado de `parquet_writer` — reuso do mesmo wrapper já validado em Story 2.2). SHA256 é armazenado em `partitions.checksum_sha256` na mesma transação.
- [x] **Testes cobrem dedup, gap detection, schema versioning?** **SIM.**
  - Schema versioning: `tests/unit/test_migration_base.py` (16 tests — ABC contract + registry + dry-run) + `tests/unit/test_migration_runner.py` (9 tests — plan/execute/rollback/cleanup/resume).
  - Round-trip + idempotência aditiva: `tests/property/test_migration_aditive.py` (Hypothesis 100 examples — INV-9 preservação byte-a-byte de campos comuns + novo campo NULL).
  - CLI integration: `tests/integration/test_migrate_cli.py` (5 tests — Typer CliRunner cobrindo plan, execute, rollback, cleanup).
  - Dedup/gap-detection: **N/A para migration** (responsabilidade do writer/validator — cobertos pelos suites de Stories 1.4/2.1, intactos sem regressão).
- [x] **`fsync(parent_dir)` pós-replace (durabilidade Linux/Windows)?** **PARCIAL.** `ParquetMigration.write_new` faz `fsync(file)` no tmp antes de `os.replace` (`_base.py:288-292`) e `_make_backup` faz `fsync(backup)` best-effort (`_runner.py:485-491`). **Não há `fsync(parent_dir)` pós-replace explícito** dentro de `write_new` do mixin — diferença vs `ParquetWriter.write` (Story 1.4) que faz `_fsync_directory` antes e depois. Decisão Sol: aceito como **F-S-1 LOW** (tracking) porque (a) backup `.bak` ainda existe na mesma pasta e o catálogo é fonte de verdade transacional, (b) caller normal é a CLI em modo single-shot (não batch concorrente), (c) próxima iteração do framework pode importar `_fsync_directory` do `parquet_writer` se operação real exigir. Não bloqueia gate.
- [x] **Threshold de rewrite vs new file aplicado (evita O(n²) append)?** **N/A.** Migration não faz append — sobrescreve a partição inteira. Threshold `_PARTITION_ROW_LIMIT` permanece efetivo no path de write normal (Story 2.2 preservado).

**Justificativa:**

A Story 2.3 entrega o framework executável que MIGRATIONS.md v1.0.0 SCAFFOLD prometia,
elevando o esqueleto à condição de **código real + CLI + checkpoint resumível + rollback
documentado** sem violar **NENHUMA** das garantias storage que Sol custodia:

1. **Schema canônico Parquet v1.0.0 INTACTO.** `SCHEMA_VERSION` e `pyarrow_schema()`
   preservados. V100ToV110 só age em arquivos de disco; o schema canônico em código
   permanece v1.0.0 (decisão COUNCIL-14 §"Decisões formalizadas" 2 — bump efetivo
   quando v1.1.0 for "the new normal", futuro Epic 4).
2. **R4 (schema contrato perpétuo) preservada.** Migration aditiva (bump minor) NÃO
   quebra leitores antigos. Property test 100 examples Hypothesis confirma:
   `migrate(read_v1_table)` preserva todos os 17 campos canônicos byte-a-byte; novo
   campo sempre NULL.
3. **R5 (idempotência) preservada** em tripla defesa: (a) checkpoint via
   `_migration_log` skipa `status='migrated'`, (b) `applies_to(current_version)`
   curto-circuita no nível de plan, (c) `transform` é no-op se coluna nova já existe.
4. **INV-3 (atomicity) preservada** com nuance: `write_new` reusa pipeline
   `tmp + fsync(file) + os.replace` (pequena lacuna de `fsync(parent_dir)` registrada
   como F-S-1 LOW tracking — não bloqueia).
5. **Two-phase commit reusado** (Story 1.5 semântica): UPDATE de
   `partitions.schema_version` + `checksum_sha256` + `file_size_bytes` acontece
   **dentro** de `with catalog._transaction()` e **somente após** `verify()` passar.
   Falha em qualquer fase → restauração automática via `_restore_backup_if_exists`.
6. **Catálogo SQLite bumpado de forma controlada.** `CATALOG_VERSION 1.0.0 → 1.1.0`
   é aditivo puro (nova tabela `_migration_log` + index); `_apply_migrations` aplica o
   delta no boot de catálogos pré-existentes (idempotente). Tabelas existentes
   (`partitions`, `downloads`, `gaps`, `contracts`, `_checksum_cache`,
   `_pending_commits`) **NÃO modificadas**.
7. **Backup `.parquet.bak` por partição preservado** (AC7). Criado **antes** de
   overwrite via cópia byte-a-byte + fsync best-effort. Rollback restaura via
   `os.replace` atômico. CLI `migrate cleanup --older-than 30` retention policy.
8. **Fronteira `public_api/` intocada.** Aria sign-off em COUNCIL-14 §"Sign-off Aria"
   confirma. Diff `public_api/` = vazio. Migration framework é interno ao `storage/`
   (módulos com prefixo `_`), exposto apenas via CLI `data-downloader migrate ...`.
9. **+31 novos tests, 0 regressões.** Suite completa 622 PASS + 1 SKIP (vs 591 PASS
   + 1 SKIP em HEAD anterior). Cobertura de ABC, Registry, Runner, CLI integration e
   property test Hypothesis (100 examples) cumprem AC9 (5 dos 6 testes mandatados;
   pre-conditions test rastreado como F-S-2 LOW por decisão COUNCIL-14 §4).

A Story 2.3 é o tipo de mudança que Sol normalmente examina com **ceticismo elevado**
— framework de migração é onde bugs silenciosos podem corromper datasets de 10+ GB e
inviabilizar backtests. O design conservador (backup obrigatório + checkpoint
resumível + rollback transacional + verify pós-write + property test 100 examples +
schema canônico em código intacto) entrega **defesa em profundidade** suficiente para
Sol assinar com tranquilidade.
