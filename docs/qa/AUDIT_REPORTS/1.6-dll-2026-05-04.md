## 4. Achados (Findings)

> Cada finding numerado. Severity: CRITICAL | HIGH | MEDIUM | LOW | INFO.

### F-N-1 — LOW — `contracts validate` CLI não aceita flag `--exchange`

- **Arquivo:** `src/data_downloader/cli.py:248-322` (`contracts_validate`).
- **Descrição:** CLI sempre usa `exchange="F"` implicitamente (default
  do `probe_contract`). Para tickers Bovespa (PETR4, VALE3 — também
  presentes no seed), operador precisaria passar `exchange="B"`. Hoje
  isso é hardcoded como F. Funcionalmente OK pois equities NÃO são
  alvo primário do probe (não fazem rollover), mas API inconsistente
  com `contracts vigent` que aceita `--exchange / -e`.
- **Manual ref:** Manual §3.1 L1673 (`B`/`F` literal); Q05-V; R8.
- **Sugestão de fix:** Adicionar `exchange: str = typer.Option("F",
  "--exchange", "-e", help="Bolsa: F (BMF) ou B (Bovespa)")` em
  `contracts_validate` e propagar para `probe_contract(exchange=exchange)`.
  Validação `if exchange not in ("F", "B")` reusa lógica do
  `download_chunk`.
- **Reprodução mínima:**
  ```bash
  data-downloader contracts validate PETR4 PETR4
  # Hoje: usa exchange="F" → pode falhar com NL_EXCHANGE_UNKNOWN
  # Esperado: --exchange B
  ```
- **Severity rationale:** LOW — equities não são caso primário; probe
  para equities é cosmético (validação de existência). Não bloqueia.

### F-N-2 — INFO — `PROBE_TIMEOUT_SECONDS = 300` hardcoded (sem flag CLI)

- **Arquivo:** `src/data_downloader/orchestrator/contracts_probe.py:50` +
  `src/data_downloader/cli.py:248-322`.
- **Descrição:** Probe usa timeout fixo de 5 min. Em cenários
  patológicos (DLL com latência alta, primeiro probe pós-init com
  cold cache, pregão lento com pouco volume), 5 min pode dar
  falso-positivo `status=timeout`. Não há flag CLI para override.
- **Manual ref:** Q02-E (99% reconnect); 1.3 audit item 16.
- **Sugestão de fix:** Adicionar `--timeout` no CLI `contracts validate`
  (default `PROBE_TIMEOUT_SECONDS`) — útil para probe inicial após
  deploy ou em horários fora do regular trading.
- **Reprodução mínima:** N/A (precisa DLL real lenta).
- **Severity rationale:** INFO — aceito por design para probe rápido;
  mitigação: operador re-roda (`probe_contract` é idempotente em
  catálogo: re-marca `validated_at` se sucesso na 2ª tentativa).

### F-N-3 — INFO — `_resolve_sample_date` fallback (`today - 7d`) ignora calendário B3

- **Arquivo:** `src/data_downloader/orchestrator/contracts_probe.py:199-218`.
- **Descrição:** Fallback usa `date.today() - timedelta(days=7)` —
  pode cair em fim-de-semana ou feriado. `download_chunk` retorna
  0 trades → probe falha com `reason="no_trades"`. Operador é
  avisado, mas perde ciclo. Cross-ref com Sol audit F-S-6 (mesma
  observação no lado storage).
- **Manual ref:** N/A (DLL não distingue dia útil — apenas retorna
  vazio).
- **Sugestão de fix:** Quando Story 2.X (`bizdays-integration`)
  entregar `is_b3_business_day`, usar
  `last_b3_business_day_before(today - 7d)`. Enquanto isso, operador
  pode passar `--sample-date` explicitamente.
- **Reprodução mínima:**
  ```bash
  # Probe num sábado/domingo:
  data-downloader contracts validate WDO WDOJ26
  # Resultado: ProbeResult(success=False, reason="no_trades")
  ```
- **Severity rationale:** INFO — não bloqueia; UX. Mitigado por
  `--sample-date` opcional.

### F-N-4 — INFO — Janela 09:00..18:00 hardcoded (não cobre after-market WDO)

- **Arquivo:** `src/data_downloader/orchestrator/contracts_probe.py:123-124`.
- **Descrição:** Probe pede janela 09:00..18:00 BRT (cobre pregão
  regular B3). After-market do WDO (16:30-18:00 ext) está na faixa,
  mas se B3 mudar horário (e.g. extensão para 19:00 — historicamente
  já aconteceu), probe perderá ~1h de trades que poderiam validar
  o contrato. Pequeno risco — pregão regular tem >> 0 trades para
  WDO/WIN, então `trades > 0` quase certamente vai disparar.
- **Manual ref:** N/A (B3 calendário externo).
- **Sugestão de fix:** Constantes nomeadas `_PROBE_WINDOW_START_HOUR
  = 9` e `_PROBE_WINDOW_END_HOUR = 18` (não bloqueia, mas facilita
  ajuste futuro se B3 mudar horário). Considerar 09:00..19:00 para
  margem extra (custo: ~1h a mais em sleep da DLL — irrelevante).
- **Reprodução mínima:** N/A (cenário hipotético).
- **Severity rationale:** INFO — janela atual cobre 99% dos casos.
  Tracking opcional.

### F-N-5 — INFO — Probe consome quota de download da Nelogica

- **Arquivo:** `src/data_downloader/orchestrator/contracts_probe.py` (todo).
- **Descrição:** Cada probe = 1 chamada de `GetHistoryTrades` por
  `contract_code` × `sample_date`. Nelogica pode ter rate-limit ou
  quota daily (manual ProfitDLL pt_br não documenta limites
  explícitos, mas `Q08-E` registra que reconexões automáticas podem
  trigger throttling — convenção da casa). Probe em massa (e.g.
  loop sobre 100 contratos) pode entrar em rate-limit. Não há
  back-off neste módulo.
- **Manual ref:** Q08-E (reconexão throttling — convenção observada);
  Manual §3.1 L1750 (rate-limit não documentado).
- **Sugestão de fix:** Story 2.X (probe em massa via
  `data-downloader contracts validate --root WDO --year 2026`)
  deve adicionar back-off exponencial e quota tracker. V1 atual
  (1 contrato por invocação CLI) é seguro.
- **Reprodução mínima:** N/A (precisa rate-limit real Nelogica).
- **Severity rationale:** INFO — V1 single-shot OK; Story 2.X
  precisa addressar.

**Nenhum finding HIGH ou CRITICAL.**

correta** sobre o primitive `download_chunk` (Story 1.3 — auditado
APPROVED, R3/INV-1 verificada em teste). Story 1.6 NÃO modifica
`dll/wrapper.py`, NÃO registra novos callbacks, NÃO adiciona código
em hot-path da ConnectorThread.

- **Lei R3 / INV-1 preservada:** `_mark_validated` UPDATE em catalog
  acontece em OrchestratorThread APÓS `download_chunk` retornar
  (todas as threads finalizadas) — NUNCA dentro de callback DLL.
  `download_chunk` (1.3) já garante INV-1 com test
  `test_history_callback_does_not_invoke_translate_trade_inv1`.

  letra. Smoke test usa `WDOJ26` explícito (linha 62
  `tests/smoke/test_probe.py`). Esta é a razão da Story 1.6
  existir — confirmado.
- **Timeout reduzido para 300s (5 min)** vs. `DEFAULT_TIMEOUT_SECONDS
  = 1800` (30 min) — APROVADO para probe: janela é 1 dia útil
  (≪ meses), DLL responde em segundos com warm cache. Q02-E
  (99% reconnect) ainda tolerada via `_ProgressMonitor` (1.3).
  F-N-2 (INFO) sugere flag CLI para casos extremos.
- **Janela 09:00..18:00 hardcoded** cobre pregão regular B3 — sem
  necessidade de calendário/feriado (cf. F-N-4 INFO se B3 mudar
  horário no futuro).
- **Bolsa default `"F"`** alinha com WDO/WIN (uso primário). F-N-1
  (LOW) sugere flag CLI para coerência com `contracts vigent`.
- **Atualização do catálogo SOMENTE em sucesso:** `_mark_validated`
  chamado APÓS guards `chunk.status != "completed"` (linhas 144-161)
  E `trades_count == 0` (linhas 163-175) — falhas geram
  `ProbeResult(success=False, reason=...)` sem tocar catálogo.
  Defesa graciosa: `probe.mark_validated_no_row` log warn se UPDATE
  não afetar linha (probe ad-hoc fora de seed).
- **`_cb_refs` não tocado:** Story 1.6 não cria callbacks novos.
  Reuso 100% do pipeline 1.3.
- **WINFUNCTYPE preservada delegada:** `THistoryTradeCallbackV2`
  permanece WINFUNCTYPE (1.3 audit item 2).
- **CLI wiring (`contracts_validate`)** usa `with ProfitDLL() as dll`
  (context manager) → `finalize()` automático no exit (Q-AMB-03
  resolvido pela Story 1.2). `wait_market_connected(timeout=60)`
  antes de probe — gracioso para cold-start.
- **CLI exit codes consistentes:** 0 OK, 1 probe falhou,
  2 input error, 3 missing creds, 4 DLL not connected — alinha
  INTEGRITY.md §8.

**5 findings, todos LOW/INFO** (F-N-1..F-N-5), sem bloqueio. F-N-1
