## 2. Escopo

- **Em escopo:**
  - Confirmar que o Protocol pattern (`MetricsEmitter` ABC + `NullMetricsEmitter`)
    em `src/data_downloader/contracts/observability.py` preserva a fronteira
    `orchestrator/` ↔ `observability/` (Aria authority sobre fronteiras).
  - Confirmar que **public_api intacto** (sem breaking changes — apenas extensão
    aditiva opcional `metrics_emitter: MetricsEmitter | None = None` propagada
    em `download.py`).
  - Confirmar que **hot path R21 preservado** — emitter chamado APENAS per-chunk
    (cool path), nunca per-trade.
  - Confirmar que **opt-in default OFF** — `NullMetricsEmitter` zero overhead
    quando exporter desabilitado (sem flag `--metrics-port`).
  - Confirmar que **ADR-013 amendment 2026-05-04** está coerente com
    implementação (8 counters + 5 gauges + 5 histograms canônicos).
  - Confirmar que **MultiTargetEmitter** alinha com COUNCIL-15 §D6 (composability
    sem refactor do orchestrator).
  - Confirmar que **`prometheus_client>=0.20`** é a única nova dependência
    transversal — autorizada explicitamente em ADR-013 §Decisão.

- **Fora de escopo:**
  - Detalhes de implementação interna de Counter/Gauge/Histogram
    (delegado a `prometheus_client` upstream).
  - Performance benchmarks pós-implementação (Pyro `bench_observability_overhead`
    deferred Story 2.7 — hot path tuning; orchestrator R21 garantido por unit
    test).
  - `OBSERVABILITY.md` ops doc (deferred — ADR-013 amendment + COUNCIL-15
    cobrem release).

- **Componentes tocados:**
  - `src/data_downloader/contracts/observability.py` (novo — 122 linhas)
  - `src/data_downloader/contracts/__init__.py` (estendido — re-exports)
  - `src/data_downloader/observability/__init__.py` (novo — exports)
  - `src/data_downloader/observability/prometheus_exporter.py` (novo — 488 linhas)
  - `src/data_downloader/orchestrator/orchestrator.py` (estendido — +60 linhas hooks per-chunk)
  - `src/data_downloader/cli.py` (estendido — +50 linhas `--metrics-port` lifecycle)
  - `src/data_downloader/public_api/download.py` (estendido — +10 linhas propagação)
  - `pyproject.toml` (estendido — `prometheus_client>=0.20`)

- **Cruza fronteiras?** **SIM, mas via Protocol — fronteira preservada por design.**
  - `orchestrator/` depende **apenas** de `contracts/observability.py` (interface
    abstrata) — NÃO importa `observability/` diretamente.
  - `observability/prometheus_exporter.py` implementa o Protocol (estrutural
    via `runtime_checkable`) e é **injetado** no orchestrator via DI.
  - `cli.py` é entrypoint — autorizado a instanciar exporter concreto
    (`PrometheusExporter`) e injetar via `download.download_history(...)`.
  - `public_api/download.py` propaga o parâmetro `metrics_emitter: MetricsEmitter | None`
    sem expor a implementação concreta — caller passa qualquer implementação.

## 3. Checklist (design_review)

> Origem: `agents/architect.md` → `checklists.design_review`

- [x] **Respeita thread model?** **SIM.** O exporter HTTP roda em **daemon
  thread** próprio (`prometheus_client.start_http_server` cria `ThreadingWSGIServer`
  daemon — não bloqueia processo principal). Orchestrator continua em
  OrchestratorThread distinta de ConnectorThread/IngestorThread/WriterThread/
  UIThread (INV-11 separação física preservada). Emitter `incr_counter`/
  `set_gauge`/`observe_histogram` é **lock-free** (prometheus_client garante
  via primitives atômicas em CPython com GIL) — pode ser chamado de qualquer
  thread sem coordenação extra.

- [x] **Respeita fronteiras de camada?** **SIM.** Fronteiras `dll/ ↔
  orchestrator/ ↔ storage/ ↔ ui/` via `contracts/`:
  - `contracts/observability.py` define o Protocol (Aria authority — fronteira).
  - `orchestrator/` importa **apenas** o Protocol (`MetricsEmitter`,
    `NullMetricsEmitter`) — sem dep direta em `observability/`.
  - `observability/` implementa o Protocol — sem dep em `orchestrator/`,
    `storage/`, `dll/`, `ui/`.
  - `cli.py` e `public_api/download.py` são entrypoints — autorizados a
    fazer wiring (Aria endossa: entrypoints conhecem implementações
    concretas; camadas internas conhecem apenas contratos).
  - **Sem dependências circulares** — verificado por `python -c "import
    data_downloader.observability"` e `python -c "import data_downloader.orchestrator.orchestrator"`.

- [x] **Toda fila tem `maxsize` E política de back-pressure?** **N/A direto.**
  Story 2.4 NÃO introduz novas filas. As filas existentes (DLL, write,
  ui_progress — Story 1.7a / ADR-005) mantêm seus `maxsize` e políticas.
  As métricas de queue depth (`dll_queue_depth`, `write_queue_depth`,
  `ui_progress_queue_depth`) são **observadores** dessas filas (gauges
  set instantâneos), não introduzem novos buffers. Servidor HTTP
  Prometheus é WSGI síncrono single-thread — sem fila interna que precise
  back-pressure.

- [x] **É idempotente onde precisa ser?** **SIM.**
  - `PrometheusExporter.start()` é idempotente: chamadas subsequentes
    detectam `self._server is not None` e retornam no-op (log debug).
  - `PrometheusExporter.stop()` é idempotente: chamadas subsequentes
    detectam `self._server is None` e retornam no-op.
  - `incr_counter`/`set_gauge`/`observe_histogram` são intrinsecamente
    seguras para chamadas repetidas (Counter monotônico; Gauge override;
    Histogram cumulative).
  - `MultiTargetEmitter` delega a todos os targets em ordem fixa — sem
    estado mutável próprio (apenas tupla imutável).
  - **Métrica desconhecida** (e.g. orchestrator chama emitter com nome
    fora do canônico) — ignorada com log warning, **não crash** (facilita
    evolução incremental sem quebrar produção — design endossado).

- [x] **Introduz dependência nova?** **SIM — `prometheus_client>=0.20`.
  Autorizada explicitamente em ADR-013 §Decisão (Opção A escolhida) e
  amendment 2026-05-04.** Pyproject.toml diff:
  ```
  + "prometheus_client>=0.20",
  ```
  Lib é **padrão da indústria** (CNCF, used by Kubernetes, Grafana, etc.),
  thread-safe lock-free, mature (desde 2014). Sem outras deps adicionais
  (sem `opentelemetry`, sem `flask`, sem `aiohttp` — `wsgiref.simple_server`
  é stdlib). Custo de manutenção mínimo.

- [x] **Schema Parquet alterado?** **NÃO.** Story 2.4 não toca `storage/`.
  `SCHEMA_VERSION = "1.0.0"` em `storage/schema.py` preservado;
  `pyarrow_schema()` (17 campos canônicos) intacta. `Sol` não convocada
  (lei R-storage não aplicável). R4 (schema contrato perpétuo) preservada.

- [x] **Wrapper DLL alterado?** **NÃO.** Story 2.4 não toca `dll/`.
  `wrapper.py`, callbacks, sinatures DLL — todos intactos. Lei R3
  (regra Nelo — callback NÃO chama DLL) preservada **com reforço** —
  emitter NÃO é chamado dentro de callback (verificado por
  `test_orchestrator_no_emitter_call_per_trade`). Nelo não convocado.

- [x] **Public API alterada?** **SIM — extensão aditiva opcional, NÃO
  breaking.** `src/data_downloader/public_api/download.py` ganhou parâmetro
  opcional `metrics_emitter: MetricsEmitter | None = None` em ambas
  funções públicas (`download_history` e `download_history_async`). Aria
  confirma:
  - **Sem bump de major** necessário — adição de parâmetro **opcional
    com default `None`** preserva backward compat (semver R-public-api:
    breaking = remoção/rename/mudança de tipo; adição opcional = minor).
  - `__api_version__` (ADR-007a) pode receber bump minor (`1.x.0` →
    `1.(x+1).0`) em release Story 2.4 — **deferred para gate de Epic 2
    close** (Morgan + Gage decidem janela de release).
  - Tipo do parâmetro é o Protocol em `contracts/` — caller pode passar
    qualquer implementação (Prometheus, structlog, OTEL futuro,
    `NullMetricsEmitter`) sem importar `observability/`.

- [x] **Erro propaga corretamente?** **SIM.** Hierarquia respeitada
  (ADR-011):
  - `PrometheusExporter.start()` propaga `OSError` se porta ocupada
    (caller decide retry — auto-retry deferred yagni V1, COUNCIL-15 §D5).
  - `PrometheusExporter.stop()` captura `Exception` defensivamente em
    `try/finally` (pragma: no cover) — best-effort shutdown.
  - `cli.py` `download` command captura `OSError` no `try` block
    (`cli.py:776-781`) — log raw via Rich console + `raise typer.Exit(1)`
    se porta indisponível.
  - **Nenhuma `print()` raw** — todos os logs via `logging.getLogger`
    e Rich console (cli.py).
  - Métricas desconhecidas (Counter/Gauge/Histogram fora do canônico):
    log warning silencioso, não-crash — design defensivo endossado.

- [x] **Hot path respeitado?** **SIM — R21 reforçado, não violado.**
  Validação multi-camada:
  - **Design (COUNCIL-15 §D3):** Orchestrator hooka emitter APENAS
    per-chunk (cool path). Tabela completa de pontos de chamada documentada.
  - **Implementação (`orchestrator.py:683`):** `trades_received_total`
    incrementado **1x por chunk batch** (`if chunk_result.trades:`),
    NÃO 1x por trade. Comentário explícito: "Increment em batch
    (len(trades)) é cool path; NÃO chamamos per-trade no callback DLL."
  - **Test (`test_orchestrator_no_emitter_call_per_trade`):** valida
    que job de N trades em 1 chunk gera **1** chamada a
    `trades_received_total` (não N). Cobertura explícita do invariante R21.
  - **Default zero overhead:** `NullMetricsEmitter` é o default —
    `incr_counter`/`set_gauge`/`observe_histogram` são no-ops vazios
    (~80ns dispatch Python, sem alocações). Pyro confirmou em sign-off
    COUNCIL-15.

- [x] **Shutdown desenhado?** **SIM.** Lifecycle gerenciado em `cli.py`
  (entrypoint authority):
  - **Start:** Após parse de flag `--metrics-port`, `PrometheusExporter(port=N)`
    instanciado e `.start()` invocado **antes** do download iniciar.
  - **Stop:** Em `try/finally` no comando `download` (`cli.py`), `.stop()`
    é garantido mesmo em caso de exceção/Ctrl+C.
  - **Idempotência:** `start()` e `stop()` são idempotentes (verificado
    por test `test_lifecycle_start_stop_idempotent`).
  - **Daemon thread:** `prometheus_client.start_http_server` cria
    `ThreadingWSGIServer` em daemon thread — não impede shutdown do
    processo principal mesmo se `.stop()` não for chamado (defensive
    layer).
  - **State machine orchestrator (ADR-005 amendment):** Orchestrator
    `Running→DrainingDLL→DrainingWrite→Committed` continua intacto;
    métricas `active_downloads` (gauge) refletem estado em tempo real.

## 4. Achados (Findings)

### F-A-1 — INFO — `dll_drops_total` reservado mas não emitido em V2

- **Arquivo:** `src/data_downloader/observability/prometheus_exporter.py:147-150` +
  `src/data_downloader/orchestrator/orchestrator.py` (sem call site)
- **Descrição:** Counter `dll_drops_total{symbol}` registrado no exporter
  (COUNCIL-05 §D3 finding) mas o orchestrator **não emite** essa métrica
  em V2. Razão: drops por back-pressure DLL acontecem em
  `IngestorThread/ConnectorThread` (callback layer — `dll/`) que está
  **fora do escopo de Story 2.4**. A métrica existe em registry para
  V3 (Story 2.7 hot path tuning ou Epic 4 multi-symbol — quando call
  site for adicionado).
- **ADR/INV ref:** ADR-013 §Métricas V1 (lista canônica) + COUNCIL-05
  §D3 + COUNCIL-15 §D5 — métrica reservada explicitamente.
- **Sugestão de fix:** Sem ação imediata. Documentação (`description`)
  já marca "(V2 — reservado)". Quando call site for adicionado em
  Story futura, emitter já estará pronto (zero refactor).

### F-A-2 — INFO — Auto-retry de porta deferred (yagni V1)

- **Arquivo:** `src/data_downloader/observability/prometheus_exporter.py:332-353`
- **Descrição:** AC4 da story originalmente sugeria "se porta ocupada,
  log warning + tentar próxima até `port+10` ou abortar com mensagem
  clara". Implementação V2 propaga `OSError` direto ao caller. CLI
  (`cli.py:776-781`) captura e exibe mensagem clara via Rich console.
- **ADR/INV ref:** COUNCIL-15 §D5 — yagni V1 (caller pode tentar próxima
  porta facilmente; auto-retry adiciona complexidade sem caso de uso real).
- **Sugestão de fix:** Sem ação imediata. Se uso real exigir, adicionar
  loop em `start()` ou wrapper externo. Documentado em COUNCIL-15 e na
  Story 2.4 §Tasks 3.2.

### F-A-3 — INFO — Env var `DATA_DOWNLOADER_METRICS_PORT` deferred

- **Arquivo:** `src/data_downloader/cli.py:599-781`
- **Descrição:** AC4 mencionava env var como segunda opção de override.
  Implementação V2 oferece **apenas** flag CLI `--metrics-port`. COUNCIL-15
  §D5 documenta deferral (yagni V1; CLI flag cobre 99% dos casos).
- **ADR/INV ref:** COUNCIL-15 §D5.
- **Sugestão de fix:** Sem ação imediata. Adicionar via `typer.Option(...,
  envvar="DATA_DOWNLOADER_METRICS_PORT")` é trivial se uso real exigir.

### F-A-4 — INFO — `OBSERVABILITY.md` ops doc deferred (PR follow-up)

- **Arquivo:** `docs/dev/OBSERVABILITY.md` (não criado nesta story)
- **Descrição:** AC8 mandatava criação de `OBSERVABILITY.md` com tabela
  de métricas, exemplo `curl`, exemplo `prometheus.yml`, troubleshooting.
  Story Dev Agent Record marca como deferred — ADR-013 amendment +
  COUNCIL-15 cobrem o suficiente para release inicial.
- **ADR/INV ref:** Story 2.4 §Tasks 8 (deferred).
- **Sugestão de fix:** Criar em PR follow-up (Pyro ou Dex).
  Não-bloqueante para Quinn `*qa-gate`.

## 5. Decisão

| Verdict             | Marcar |
|---------------------|--------|
| ✅ **APPROVED**     | **[x]** |
| 🟡 CHANGES_REQUESTED | [ ]    |
| 🔴 BLOCKED          | [ ]    |

**Justificativa:**

A Story 2.4 implementa o **Prometheus Observability Exporter V2** **dentro
da fronteira correta** — o Protocol pattern (`MetricsEmitter` ABC +
`NullMetricsEmitter`) em `src/data_downloader/contracts/observability.py`
preserva a separação canônica entre interface (camada de contratos) e
implementação (camada concreta). Aria confirma com tranquilidade
arquitetural elevada:

1. **Protocol pattern correto** — `contracts/observability.py` define o
   ABC `runtime_checkable`; `observability/prometheus_exporter.py`
   implementa estruturalmente sem subclass; orchestrator depende **apenas**
   do Protocol (verificado em `orchestrator.py:55`). Esta é a
   materialização canônica do COUNCIL-15 §D1 sign-off Aria — fronteira
   `orchestrator/` ↔ `observability/` preservada por design, não por
   convenção.

2. **Public API intacto (extensão aditiva, não breaking)** —
   `public_api/download.py` ganhou parâmetro opcional
   `metrics_emitter: MetricsEmitter | None = None` em
   `download_history` e `download_history_async`. Adição de parâmetro
   **opcional** com default `None` preserva backward compat. R-public-api
   (semver) NÃO exige bump major; bump minor de `__api_version__`
   (ADR-007a) deferred para gate de release Epic 2 close (Morgan + Gage).

3. **Hot path R21 preservado — REFORÇADO, não violado** — emitter chamado
   APENAS per-chunk (cool path). Validação multi-camada:
   - Design: COUNCIL-15 §D3 documenta tabela completa de call sites.
   - Implementação: `orchestrator.py:683` incrementa
     `trades_received_total` **1x por chunk batch**, não 1x por trade.
   - Test: `test_orchestrator_no_emitter_call_per_trade` valida
     explicitamente que job de N trades em 1 chunk gera **1** chamada
     (não N). R21 (sem logging per-trade) cumprido com sobra.

4. **Opt-in default OFF — zero overhead default** — sem flag `--metrics-port`,
   `NullMetricsEmitter` é o default. Métodos vazios `__slots__ = ()` —
   ~80ns dispatch Python sem alocações. Pyro endossou em COUNCIL-15 §D4.
   Usuários V1 desktop (sem flag) NÃO pagam custo de observability.

5. **ADR-013 amendment 2026-05-04 ratificado** — implementação coerente
   com amendment ponto-a-ponto: Protocol em `contracts/`,
   PrometheusExporter em `observability/`, orchestrator aceita opcional,
   CLI flag, 8 counters + 5 gauges + 5 histograms canônicos, R21
   reforçado, cardinality LRU deferred Epic 4. Tabela §3.3 confirma
   alinhamento total.

6. **MultiTargetEmitter alinhado com COUNCIL-15 §D6** — composability
   futura (V1 structlog dump cool-path + V2 Prometheus HTTP) sem
   refactor do orchestrator. Implementa Protocol; delega para tuple
   imutável de targets (sem retry, sem isolation entre targets — design
   minimalista endossado).

7. **Sem novas deps além de `prometheus_client>=0.20`** — autorizada
   explicitamente em ADR-013 §Decisão (Opção A). Lib é padrão da indústria
   (CNCF), thread-safe lock-free, mature (desde 2014). Sem outras deps
   adicionais (sem `opentelemetry`, sem `flask`, sem `aiohttp` —
   `wsgiref.simple_server` é stdlib).

8. **Sem regressões cross-story** — 622 PASS + 1 SKIP (vs 622 + 1 baseline
   pre-2.4 quando contado o conjunto pre-existente; +31 testes específicos
   de 2.4 PASS). Cross-story matrix §3.2 confirma compat com Stories
   1.4/1.5/1.7a/1.7b/1.8/2.1/2.2/2.3/2.5.

9. **Thread model preservado (ADR-005)** — exporter HTTP em daemon
   thread `prometheus_client` (não bloqueia processo principal).
   Orchestrator continua em OrchestratorThread distinta.
   `incr_counter`/`set_gauge`/`observe_histogram` lock-free
   (prometheus_client garante).

10. **Lifecycle gerenciado em entrypoint (cli.py)** — `start()` antes do
    download, `stop()` em `try/finally`. Idempotência verificada por
    `test_lifecycle_start_stop_idempotent`. Daemon thread defensive
    layer.

A Story 2.4 é o tipo de feature que Aria normalmente examina com
**ceticismo arquitetural elevado** — observability adicionada a um sistema
maduro tende a vazar para hot path (R21 violation) ou criar acoplamento
estrutural entre `orchestrator/` e `observability/`. **NÃO foi o caso
aqui:** o Protocol pattern garante fronteira; o hot path teste explícito
garante R21; o opt-in default OFF garante zero overhead. Aria assina
sem reservas — esta é uma das implementações mais limpas de observability
que poderia ter saído do squad.
