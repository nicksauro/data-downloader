## 2. Escopo

- **Em escopo:**
  - Confirmar **implementação fiel ADR-015 Opção A** (broker no master +
    workers via `multiprocessing.Queue` + ACK), sem reabertura de
    decisão arquitetural.
  - Verificar **pool persistente** (não spawn-per-job — finding H20
    Pyro): workers aquecidos reusados entre jobs.
  - Confirmar **ACK protocol AC4**: worker recebe ACK síncrono via
    response_queue; UUID `request_id` correlaciona; timeout configurável
    levanta `BrokerTimeoutError`.
  - Verificar **SQLite thread-bound conn**: broker abre **própria
    conexão R/W na sua thread** (mesmo db_path), preserva WAL semantics
    sem violar `check_same_thread`.
  - Validar **`BrokerCatalogClient` espelha contrato `Catalog`**
    (worker-side stub) — Orchestrator usa transparente.
  - Confirmar **Q17-OPEN registrado** em `docs/dll/QUIRKS.md` para
    licença Nelogica multi-instância (P2 COUNCIL-13 escalado a
    humano + Nelo).
  - Verificar **sem mudança de fronteira public_api** (multi-symbol
    fica apenas na CLI em V1.0 — confirmação D6 COUNCIL-25 + D9
    COUNCIL-13).
  - Verificar **WAIVER smoke real estende COUNCIL-09 corretamente**
    (mesmo padrão 1.7b/1.8 — sign-offs Aria+Pyro+Morgan implícitos).

- **Fora de escopo:**
  - Implementação detalhada de cada classe (testada em 38 unit/integ
    tests separados — auditoria Quinn).
  - Bench mock vs target (auditoria Pyro registrada em COUNCIL-25 D4 —
    speedup 2.72x FAIL vs 3.2x mas justificado por payload mínimo;
    re-validação fica para 4.1-followup com smoke real).
  - Smoke real multi-symbol (deferred via WAIVER `4.1-real-smoke-deferred-2026-05-04.md`).

- **Mudou public_api?** **NÃO.** Sub-pacote `orchestrator/broker/` é
  totalmente interno; CLI é a única superfície que ganha multi-symbol
  em V1 (D6 COUNCIL-25). `public_api.download(...)` continua single-symbol.
  **SemVer impact: NONE.**

- **Mudou fronteira interna?** **SIM (NEW).** Sub-pacote
  `src/data_downloader/orchestrator/broker/` cria nova fronteira de
  processo (master ↔ workers via mp.Queue + ACK). Encapsulada dentro
  do orchestrator — não exposta publicamente.

### 3.1 design_review (Aria checklist)

| Item                                                | Status | Evidência |
|-----------------------------------------------------|--------|-----------|
| Respeita thread model?                              | ✅     | Broker é **thread no master** (não subprocess) — `catalog_broker.py:_run` cria conexão SQLite **na própria thread** (linha 222-239) preservando `check_same_thread=True` default. Workers são **processos separados** com 1 DLL cada (R20). DLL nunca chamada em callback (INV-1) — broker no master só vê Queue messages, nunca toca DLL. |
| Respeita fronteiras de camada?                      | ✅     | `broker/` é sub-pacote em `orchestrator/` — fronteira interna, NÃO em `public_api/`. Workers usam `BrokerCatalogClient` (worker-side stub) que espelha contrato `Catalog` (subset usado pelo Orchestrator). Broker mestre usa instância real de `Catalog` (Story 1.5) thread-local. |
| Toda fila tem maxsize e política?                   | ⚠️ INFO | `mutation_queue` (mp.Queue) e `response_queue` (mp.Queue) **NÃO têm maxsize explícito** (default `mp.Queue()` é unbounded — apenas limitado por OS pipe buffer). | É idempotente onde precisa ser?                     | ✅     | `register_partition` no broker usa `Catalog.register_partition` que é UPSERT (INV-6 + R5 — Story 1.5). Re-enviar mesma payload = no-op idempotente. Test `test_multi_symbol_mock.py::test_lifecycle_idempotent` valida. |
| Introduz dep nova? Tem ADR?                         | ✅     | **Zero deps novas.** Apenas stdlib (`multiprocessing`, `threading`, `queue`, `uuid`, `dataclasses`, `enum.StrEnum`). ADR-015 já cobria o desenho; Story 4.1 é **implementação fiel**, não decisão nova. |
| Schema Parquet alterado?                            | ✅ N/A | Story 4.1 NÃO toca `storage/`. Workers escrevem Parquet usando `ParquetWriter` existente (Story 1.4) sem alteração. Sol audit não requerido. |

## 5. Findings (severity matrix)

### CRITICAL
*(nenhum)*

### HIGH
*(nenhum)*

### MEDIUM
*(nenhum)*

### LOW
- **F-A-1:** Bench mock N=4 speedup 2.72x **FAIL vs target 3.2x** (Pyro
  D4 COUNCIL-25). Justificado: bench mock executa apenas 1 mutação
  (`register_partition`) por job — IPC broker overhead (~100-500µs por
  mutação) NÃO é amortizado em payload mínimo. Real downloads
  (multi-chunk × meses → 50-200 mutações por job) terão IPC diluído.
  Re-validação fica para 4.1-followup com smoke real (P3 COUNCIL-25).
  Mitigação: WAIVER documenta + 4.1-followup story-debt agendada.
  Acceptable porque AC7 explicitamente deixa "Pyro define exato após
  bench piloto" — bench mock é piloto.

### INFO
- **F-A-2:** `mutation_queue` e `response_queue` (mp.Queue) **sem
  maxsize explícito** — default unbounded apenas limitado por OS pipe
  buffer. Aceitável para V1 porque ACK síncrono provê backpressure
  natural (≤ N mutações em flight). Tracking informal para 4.1-followup
  (smoke real revelará se backpressure agressiva é necessária sob
  carga real).
- **F-A-3:** Worker crash recovery (AC8 §"Worker crash simulado")
  diferido para 4.1-followup. `stop_pool` apenas detecta workers mortos
  via `is_alive()` + log warning (linha 322-340). Restart automático
  fica para futuro quando smoke real revelar comportamento esperado.
  Documentado em `pool.py:24-36` e Tasks/Subtasks 2.3 da story.
- **F-A-4:** `BrokerCatalogClient` é **duck-typed** (não declara
  `Protocol`) — Aria deliberadamente decidiu não extrair `CatalogProtocol`
  formal em V1 (Subtask 1.4 da story). Aceitável: subset de métodos é
  pequeno, interface estável. Extração formal fica para V1.x se mais
  worker types surgirem. Documentado em Story 4.1 Dev Notes "Protocol
  Pattern".
