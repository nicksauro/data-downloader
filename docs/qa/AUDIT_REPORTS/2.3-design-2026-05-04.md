## 2. Escopo

- **Em escopo:**
  - Confirmar que o framework é **interno ao módulo `storage/`** — fronteira
    `public_api/` permanece intocada (R-public-api estável).
  - Confirmar que **nenhum ADR amendment é necessário** — o framework é a
    realização (concrete) do esqueleto já endossado em ADR-002 §"Migrações" /
    MIGRATIONS.md SCAFFOLD.

    pattern já estabelecido em Stories 1.4–2.1).

  - Detalhes de implementação interna (já cobertos pelo Sol audit em
    `2.3-storage-2026-05-04.md`).
  - Performance benchmarks (Story 2.2 endereçou throughput; migration é
    operação one-shot, não hot path).
  - Pre-conditions duras no runner (delegado ao caller — decisão COUNCIL-14 §4).

- **Componentes tocados:**
  - `storage/migrations/` (novo subpacote — 8 arquivos)
  - `storage/catalog.py` (estendido — +`_DDL_V1_1_0_DELTAS`, `CATALOG_VERSION` bump)

  - `ui/microcopy_loader.py` (estendido — 9 IDs `migration.*`)

  `data_downloader.storage` + extensão controlada de `cli.py` (Typer sub-app
  isolado) + microcopy (UI layer já estabilizada).

## 3. Checklist (design_review)

> Origem: `agents/architect.md` → `checklists.design_review`

- [x] **Respeita thread model?** **SIM.** Migration framework roda no thread
  do CLI (single-process, single-thread, não-callback, não-orchestrator).
  Não cruza com ConnectorThread/IngestorThread/WriterThread/UIThread/
  OrchestratorThread (INV-11 separação física preservada — irrelevante
  porque migration NÃO opera em pipeline live).
- [x] **Respeita fronteiras de camada?** **SIM.** Framework respeita
  `dll/ ↔ orchestrator/ ↔ storage/ ↔ ui/ via contracts/`:
  - Não toca `dll/` (não chama nenhuma função da DLL).
  - Não toca `orchestrator/` (operação one-shot, não live).
  - Vive **dentro** de `storage/migrations/` (extensão natural do módulo
    `storage/` que já contém `catalog.py`, `parquet_writer.py`,
    `_vectorized.py`, etc.).
  - CLI consome via `import` direto (`from data_downloader.storage.migrations
    import MigrationRunner`) — Aria autoriza porque CLI é entrypoint e já
    consome `storage/` diretamente em outros sub-apps (ex: `audit`,
    `integrity`).
- [x] **Toda fila tem `maxsize` E política de back-pressure?** **N/A.**
  Migration é operação síncrona partition-by-partition; não há filas
  inter-thread. Loop de execução em `_runner.py:241-316` é determinístico.
- [x] **É idempotente onde precisa ser?** **SIM.** Tripla defesa
  (documentada em Sol audit §3.2):
  1. `_migration_log` checkpoint resumível (run_id + partition_path PK).
  2. `Migration.applies_to(current_version)` curto-circuita.
  3. `transform` no-op se coluna nova já existe.
- [x] **Introduz dependência nova?** **NÃO.** Usa apenas stdlib (`uuid`,
  `time`, `os`, `re`, `dataclasses`, `pathlib`, `importlib`, `sqlite3`
  via `Catalog`) + dependências já presentes (`pyarrow`, `typer`, `rich`,
  `structlog`). Sem ADR de nova lib.
- [x] **Schema Parquet alterado?** **NÃO.** `SCHEMA_VERSION = "1.0.0"` em
  `storage/schema.py` preservado; `pyarrow_schema()` (17 campos canônicos)
  intacta. V100ToV110 só age sobre arquivos de disco; bump efetivo de
  `SCHEMA_VERSION` em código fica para quando v1.1.0 for "the new
  normal" (decisão COUNCIL-14 §"Decisões formalizadas" 2; futuro Epic 4
  multi-asset). **Sol foi consultada e auditou** (verdict APPROVED em
  `docs/qa/AUDIT_REPORTS/2.3-storage-2026-05-04.md`).
- [x] **Wrapper DLL alterado?** **NÃO.** Story 2.3 não toca `dll/`. Lei
  R3 não aplicável. Nelo não convocada.
- [x] **Public API alterada?** **NÃO.** `git diff c6d61ae^ c6d61ae --
  src/data_downloader/public_api/` retorna vazio. `__api_version__`
  (ADR-007a) preservada. `data_downloader.storage.migrations` é módulo
  novo INTERNO; consumidores externos continuam usando `public_api/` sem
  precisar conhecer o framework.
- [x] **Erro propaga corretamente?** **SIM.** Hierarquia respeitada
  (ADR-011):
  - `MigrationRunner.execute` captura `Exception` por partição,
    registra em `_migration_log` (status='failed' + error),
    dispara rollback automático (`_restore_backup_if_exists`), e
    propaga via `MigrationRunResult` (`partitions_failed`).
  - CLI `migrate execute` traduz para exit codes (0 clean, 1 parse,
    2 pre-condition, 3 migration error, 4 rollback error) — alinhado
    com `_format_microcopy("migration.error.partition_failed", ...)`.
  - Não usa `print()` raw; usa Rich console + microcopy loader.
- [x] **Hot path respeitado?** **N/A.** Migration NÃO é hot path
  (operação one-shot). Logging por partição é aceitável e desejável
  (auditoria). R21 (sem logging per-trade) não aplicável.
- [x] **Shutdown desenhado?** **SIM por design.** Migration é operação
  síncrona; não há state machine `Running→Draining→Committed` (ADR-005
  amendment endereça orchestrator live). Em caso de Ctrl+C ou crash:
  - Backups `.parquet.bak` permanecem em disco (cleanup é manual ou
    via `migrate cleanup`).
  - `_migration_log` registra o último estado (`pending` ou `failed`).
  - Resume via `migrate execute --run-id X` retoma do checkpoint.
  - CLI captura `KeyboardInterrupt` no Typer (default).

## 4. Achados (Findings)

### F-A-1 — INFO — DDL `_migration_log` duplicada (referência documental + executável)

- **Arquivo:** `src/data_downloader/storage/catalog.py:189-205` + `src/data_downloader/storage/migrations/catalog/v1_0_0_to_v1_1_0.sql`
- **Descrição:** A DDL de `_migration_log` existe em dois lugares: `_DDL_V1_1_0_DELTAS` em Python (fonte de verdade executável) e `migrations/catalog/v1_0_0_to_v1_1_0.sql` (referência documental + alinha com convenção AC1). Ambas idênticas hoje, mas drift futuro é risco.
- **ADR/INV ref:** ADR-004 (catálogo two-phase commit) — não viola; é cosmético.
- **Sugestão de fix:** Adicionar test em `tests/unit/test_migration_runner.py` que carrega ambos DDLs e compara byte-a-byte (deferred). Comentários em ambos os arquivos já apontam para o outro. Sem ação imediata.

### F-A-2 — INFO — Pre-conditions duras delegadas ao caller (CLI)

- **Arquivo:** `src/data_downloader/storage/migrations/_runner.py` (sem implementação)
- **Descrição:** Pre-conditions (lock file, disk space ≥ 2× dataset, `_pending_commits` empty) são responsabilidade do CLI/caller, não do `MigrationRunner`. Decisão COUNCIL-14 §"Decisões formalizadas" 4 — runner é puro execução.
- **ADR/INV ref:** R-pre-conditions — formalizado em COUNCIL-14.
- **Sugestão de fix:** Endurecer no futuro se operação real exigir (ex: Story 4.X que use migrate em produção com dataset 10+ GB). Por hora, CLI single-shot dev/test é OK sem pre-conditions automáticas. Sem ação.

**Total findings: 0 CRITICAL, 0 HIGH, 0 MEDIUM, 0 LOW, 2 INFO.**

## 5. Decisão

| Verdict             | Marcar |
|---------------------|--------|
| ✅ **APPROVED**     | **[x]** |
| 🟡 CHANGES_REQUESTED | [ ]    |
| 🔴 BLOCKED          | [ ]    |

**Justificativa:**

A Story 2.3 implementa o framework de migração de schema **dentro da fronteira correta** —
`storage/migrations/` é extensão natural do módulo `storage/` que já contém
`catalog.py`, `parquet_writer.py`, `_vectorized.py`. Aria confirma:

1. **Fronteira `public_api/` intocada.** `git diff c6d61ae^ c6d61ae --
   src/data_downloader/public_api/` = vazio. `__api_version__` (ADR-007a)
   preservada. Consumidores externos não precisam conhecer o framework.
2. **Sem ADR amendment necessário.** Framework é a realização (concrete)
   do esqueleto já endossado em ADR-002 §"Migrações" / MIGRATIONS.md
   SCAFFOLD. Catalog version bump 1.0.0 → 1.1.0 é aditivo puro (nova
   tabela `_migration_log`), não exige ADR.
3. **CLI `migrate plan|execute|rollback|cleanup` segue patterns
   estabelecidos.** Typer sub-app pattern espelha `audit_app`,
   `integrity_app`, `contracts_app`. 9 microcopy IDs em PT-BR via
   MICROCOPY_CATALOG (R17). Exit codes documentados (AC3).
4. **Sem regressões em outras stories.** Cross-story matrix (§3.2)
   confirma compat com Stories 1.4 (writer), 1.5 (catalog two-phase
   commit), 1.6 (contratos), 1.7a/b (orchestrator), 1.8 (CLI), 2.1
   (integrity), 2.2 (vectorized writer). Suite 622 PASS + 1 SKIP, 0
   regressões (vs 591 PASS pre-2.3).
5. **Thread model preservado.** Migration roda no thread do CLI
   (single-process, não cruza com pipeline live). INV-11 (separação
   física) irrelevante porque framework não opera durante orchestrator.
6. **Erro propaga via hierarquia ADR-011.** `MigrationRunner.execute`
   captura `Exception` por partição, registra em `_migration_log`,
   dispara rollback automático, propaga via `MigrationRunResult`. CLI
   traduz para exit codes apropriados.

A Story 2.3 é o tipo de feature que Aria normalmente examina com
**ceticismo arquitetural** — frameworks adicionados após o sistema
estar maduro tendem a vazar para `public_api/` ou criar cross-cutting
dependencies. **NÃO foi o caso aqui:** o framework é hermético
(prefixo `_` em `_base`, `_registry`, `_runner`), exposto apenas via
CLI controlado, e respeita a fronteira que ADR-007a definiu. Aria
assina sem reservas.

já APPROVED (`2.3-storage-2026-05-04.md`); este artefato fecha a
