## Task #10 — Deadlock da suite de integração + triagem dos FAILs pré-existentes (Quinn @qa, 2026-05-11)

**Status:** DONE (com 6 FAILs/ERRORs remanescentes documentados como fora-de-escopo — ver `docs/qa/KNOWN-TEST-FAILURES-v1.1.0.md`).

### O que era o deadlock

Duas causas-raiz combinadas, ambas só visíveis na **suite completa**:

1. **`catalog_broker.py` — `extra={"name": ...}` no logging stdlib.** `name` é
   atributo reservado de `logging.LogRecord` → `KeyError("Attempt to overwrite 'name'")`
   quando algum teste anterior (UI/logging) configura um handler stdlib. Isolado, o
   root logger não tem handler → o `extra` nunca é validado → sem erro. **Fix:** renomeado
   para `broker_name` (`catalog_broker.py`, produção, ~6 linhas).
2. **Signature drift dos mocks `_FakeProfitDLL`.** Os mocks implementavam a API V1
   antiga `translate_trade(handle, struct) -> int` + progress callback de 4 args;
   produção migrou (Story 1.7b-followup / Q-DRIFT-05) para `translate_trade(handle) -> TradeFields|None`
   + progress callback de 2 args `(TAssetID, c_int)`. Consequência: `download_chunk`
   chamava `translate_trade(handle)` → `TypeError` engolido → 0 trades, `last_packet_seen=False`,
   0 progress → `download.timeout` (10–1800s) → retry exponencial → suite "pendura"
   (com `WorkerPool.submit_jobs` esperando até `n_jobs × 3600s`). **Fix:** mocks atualizados
   para API V2 em `test_orchestrator.py`, `test_orchestrator_with_metrics.py`,
   `test_orchestrator_with_retry.py`, `test_multi_asset_mock.py`,
   `test_multi_symbol_mock.py` (via `_mock_worker_factory.py`),
   `tests/property/test_orchestrator_idempotent.py`. Bônus: mock sem `rounds` agora emite
   chunk vazio mas **completo** (progress=100) em vez de `return 0` mudo — evita
   timeout/retry quando o chunker ADR-023 (1d) gera mais chunks que rounds.

## Task #11 — QTimer leak → cluster UI flaky (Dex @dev, 2026-05-11)

### Causa-raiz
`settings_screen.py:1242` `QTimer.singleShot(3000, lambda: self._set_state(STATE_NORMAL))` órfão (lambda captura `self`). Teste destrói a `SettingsScreen` antes dos 3s → timer dispara no teste seguinte sobre objeto C++ deletado → `RuntimeError: Signal source has been deleted` / `QThread: Destroyed while thread '' is still running`, poluindo testes de UI de forma não-determinística (8-10 fail/err variando por run). Wave A adicionou cobertura de `settings_screen` (Dex B-1/B-2), expondo o leak preexistente.

### Fix aplicado (mínimo invasivo — só QTimer leak)
| Arquivo | Antes | Depois |
|---------|-------|--------|
| `settings_screen.py` `__init__` | — | `self._state_restore_timer = QTimer(self); setSingleShot(True); timeout.connect(lambda: self._set_state(STATE_NORMAL))` |
| `settings_screen.py:1242` `_on_save_clicked` | `QTimer.singleShot(3000, lambda: self._set_state(STATE_NORMAL))` | `self._state_restore_timer.start(3000)` |
| `settings_screen.py` `_show_toast` | `QTimer.singleShot(duration_ms, self._toast.hide)` | timer membro `self._toast_hide_timer` (parented) |
| `settings_screen.py` `_load_initial_values` (auto-detect toast, 250ms) | `QTimer.singleShot(250, lambda: self._show_toast(...))` | timer membro `self._auto_detect_toast_timer` (parented) |
| `catalog_screen.py` `_show_toast` | `QTimer.singleShot(duration_ms, self._toast.hide)` | timer membro `self._toast_hide_timer` (parented) |
| `download_screen.py` `_show_toast` | `QTimer.singleShot(duration_ms, self._toast.hide)` | timer membro `self._toast_hide_timer` (parented) |

`QTimer` parented em `self` é destruído junto com a screen → não vaza. Não cancelados (deixados como OK, delay ≤50ms ou tick 0): `settings_screen.py` `QTimer.singleShot(50, self._do_check_updates)`, `catalog_screen.py` `QTimer.singleShot(0, self.refresh)` — delays curtos/tick imediato, risco desprezível em teardown de teste.

## Task #14 — ruído de teardown Qt no Windows → `pytest tests/integration` exit code 9 (Felix @ui, 2026-05-11)

### Sintoma
`pytest tests/integration` (e `-k ui`) com **100% dos testes passando** abortava no shutdown do
interpretador: `Windows fatal exception: access violation` (×N), `QObject::~QObject: Timers cannot
be stopped from another thread` (×~18), `QObject::killTimer: …`, `QThread: Destroyed while thread
'' is still running` → **exit code 9** (0xC0000409 = `STATUS_STACK_BUFFER_OVERRUN`, o "fail fast" do
CRT no abort). Não afeta pass/fail dos testes, mas quebra o gating de CI por exit-code.

### Causas-raiz (3 leaks de QThread/QTimer no teardown)
1. **`MetricsAdapter` (`ui/widgets/metrics_panel.py`)** — o `QTimer` interno é criado em
   `_on_thread_started` e "pertence" à worker thread (parent = `self`, que vive lá após
   `moveToThread`). `shutdown()` (a) parava o timer **a partir do MainThread** (operação cross-thread
   ilegal) e (b) deixava o adapter com afinidade à worker thread morta — destruição posterior (GC no
   `atexit`) disparava `Timers cannot be stopped from another thread` em loop. **Bônus-bug:** o
   `Qt.ConnectionType.…` em `shutdown()` lançava `NameError` silencioso (engolido por `except`) porque
   `Qt` não estava importado em `metrics_panel.py` — então a tentativa anterior de limpeza nunca rodava.
2. **`MainWindow.closeEvent` (`ui/main_window.py`)** — só dava `shutdown()` no `_metrics_adapter`,
   não nos adapters dos screens (`DownloadAdapter`/`CatalogAdapter`). Screens-filhos não recebem
   `closeEvent` quando a janela principal fecha → essas `QThread` ficavam vivas no teardown
   (`QThread: Destroyed while thread 'download-adapter' is still running`).
3. **`test_ui_download_screen.py::test_clicking_download_with_valid_inputs_transitions_to_loading`** —
   disparava um download REAL via `adapter.start` (sem DLL/.env → loop de retry/timeout dentro da worker
   thread) que não respeita `cancel` a tempo; ao destruir a screen no teardown a `QThread 'download-adapter'`
   ainda estava ocupada → leak. (Mesma família do deadlock que Quinn tratou na task #10, mas no caminho REAL.)

### Fix aplicado (~70 linhas; nenhum lifecycle de adapter refatorado de forma ampla)
| Arquivo | Mudança |
|---------|---------|
| `ui/widgets/metrics_panel.py` | `import Qt`; `contextlib`. `shutdown()` agora pede via `QMetaObject.invokeMethod(self, "_teardown_in_worker", BlockingQueuedConnection)` que a **worker thread** pare o timer **e** chame `self.moveToThread(QCoreApplication.instance().thread())` — mover só funciona da thread que possui o objeto, logo só de dentro da worker. Depois adapter+timer têm afinidade com o MainThread → destruição limpa. Novos slots `@Slot() _on_thread_started` / `@Slot() _teardown_in_worker`. `QThread.setObjectName("metrics-adapter")`. |
| `ui/main_window.py` | `closeEvent` itera `self._screens` e dá `shutdown()` em cada `screen._adapter` além do `_metrics_adapter`. |
| `ui/adapters/download_adapter.py` | `QThread.setObjectName("download-adapter")`. `shutdown()` pede `handle.cancel(timeout=0.0)` antes do `quit()` + `wait(5000)`. **`terminate()` NÃO usado** — abortar a thread no meio de uma chamada nativa gera `access violation` (testado e revertido). |
| `ui/adapters/catalog_adapter.py` | `QThread.setObjectName("catalog-adapter")`. |
| `ui/screens/settings_screen.py` | `setObjectName` nos 3 worker threads lazy (`settings-test-connection` / `settings-integrity` / `settings-reconcile`) — só debug. |
| `tests/integration/test_ui_download_screen.py` | o teste de transição mocka `public_api.download` (handle vazio que "termina" imediato) — não é o download que valida, e o real travava o worker; fixture `download_screen` chama `_adapter.cancel()` + `qtbot.wait(150)` antes do `shutdown()`. |

### Validação
- `pytest tests/integration -k ui --timeout=120 -q` rodado **2×** → **exit 0**; zero `access violation` / zero `Timers cannot` / zero `QThread: Destroyed` (antes: ~18 `Timers cannot` + 1 `QThread: Destroyed` + N `access violation`, exit 9). Evidência: `docs/qa/task14-kui-final.txt`, `docs/qa/task14-diag.txt`.
- `pytest tests/integration --timeout=120 -q` (suite completa) → **451 passed, 2 skipped, exit code 0, zero ruído Qt** (~186s). Evidência: `docs/qa/task14-full-final.txt`.
- `ruff check` limpo nos 6 arquivos tocados.

### Verdict: DONE (fix aplicado, exit 0 limpo na suite de integração)

→ Detalhes e classificação: `docs/qa/KNOWN-TEST-FAILURES-v1.1.0.md` §4.

## Task #18 — `data_dir` relativo escreve dentro do bundle (descoberto no smoke real) (Dex @dev, 2026-05-12)

### Sintoma
Smoke real `run_smoke_real.ps1 -Symbol WDOFUT -Days 5` contra o bundle frozen: o download
**funcionou** (6 chunks 1d, 2.866.352 trades, status=completed — ADR-023 OK), **mas os parquets
foram escritos DENTRO do bundle**: `dist\data_downloader\_internal\data\history\F\WDOFUT\2026\05.parquet`
(~64 MB) em vez de em `./data/` (cwd) ou `user_data_dir()`. Após instalar via `Setup.exe` em
`C:\Program Files\...` (read-only) isso falharia para todo usuário; em AppData polui o bundle (some num update).

### Root cause confirmada
- `cli.py` comando `download`: `_DOWNLOAD_DATA_DIR_OPT = typer.Option(None, "--data-dir", …)`
  (default `None`) → na função `download()` (linha ~1238): `resolved_data_dir = Path(data_dir) if data_dir is not None else Path("data")`.
  Esse `Path("data")` é **relativo** e resolvido **lazy** (quando o parquet writer faz `data_dir / "history" / …` + `mkdir`/`open`).
- A ProfitDLL, ao carregar (`DLLInitialize`), faz `chdir()` para o diretório dela — em frozen mode é
  `dist\data_downloader\_internal\` (quirk **Q-DRIFT-10**). O app só **restaura** o cwd no FIM do download
  (`dll.cwd_restored quirk=Q-DRIFT-10`). Durante a escrita dos parquets o cwd ainda é `_internal\` →
  `Path("data")` resolve para `_internal\data\`.
- Confirmado que **nada carrega a DLL antes da linha 1238** — `api_download()` (linha ~1289) é a
  primeira coisa que dispara o load. Logo `.resolve()` aplicado na linha 1238 captura o cwd ORIGINAL do shell.

### Fix aplicado
- **`src/data_downloader/cli.py`** (linha ~1238):
  - antes: `resolved_data_dir = Path(data_dir) if data_dir is not None else Path("data")`
  - depois: `resolved_data_dir = (Path(data_dir).expanduser() if data_dir is not None else Path.cwd() / "data").resolve()`
  - `.resolve()` torna o path **absoluto AGORA**, pinando-o ao cwd original antes da DLL mexer nele. Comentário explicativo inline (cita Q-DRIFT-10).
- **Outros comandos** (`integrity`/`reconcile`/`validate`/`doctor`/`migrate*`) usam `typer.Option(Path("data"), …)` —
  também relativo lazy, **mas NÃO carregam a DLL** (são offline; o cwd nunca é trocado). Risco menor → **não alterados**
  nesta task (mudar o default a nível de módulo capturaria o cwd no import, que pode não ser o do shell). Documentado aqui como risco residual baixo.
- **`tests/smoke/run_smoke_real.ps1`**: passa `--data-dir $smokeDataDir` explícito (`$repoRoot\data\smoke-real`)
  no comando `download`; validação `[3/4]`/`[3b/4]` (G-2) agora procura parquets em `$smokeDataDir\history`. Smoke determinístico, não polui `$repoRoot\data\`.
- **Cleanup**: `dist\data_downloader\_internal\data\` deletado inteiro (parquet órfão de 64 MB escrito pelo smoke bugado; bundles corretos não levam `data/`).

### Validação
- `python -c "from data_downloader.cli import app"` → import OK.
- `pytest tests/unit -k "cli or download" --timeout=120 -q` → **112 passed, 1088 deselected** (~25s).
- `ruff check src/data_downloader/cli.py` → limpo. `ruff check .` → só 2 erros pré-existentes (`UP047` em `exception_adapter.py` e `retry.py`), nenhum relacionado.
- Teste de lógica em dev mode: simulado `os.chdir()` após `.resolve()` → `resolved_data_dir` permanece absoluto e pinado ao cwd original. ✅
- **NÃO rebuilt o .exe** (Orion faz o rebuild depois — fix é puramente lógico, não muda o spec).

## Task #21 — app GUI crasha ao baixar (ProfitDLL crash: duplo init + stdio inválido) (Dex @dev, 2026-05-12)

### Sintoma
App GUI instalado (`...\AppData\Local\Programs\data-downloader\data_downloader.exe`) crasha ao clicar "Baixar".
- `_internal/Erro.log` da DLL: `CreateDataLoader Failed | Access violation ... Read of address FFFFFFFFFFFFFFFF` em `System.@IntfCopy` ← `BaseDataLoaderU.CreateBaseDataLoader` ← `InitializationManagerU.TInitializationManager.InitializeAfterLogin` ← `ConnectorThreadU.TConnectorThread.LoginResult`.
- `_internal/PostMessageError.log`: `WM_VALID_LICENSE`/`WM_REPLAY_SYNC_CLOCK`/`WM_LOGIN_RESULT` perdidos com `SysLastError=1444` (ERROR_INVALID_THREAD_ID) / `1400` (ERROR_INVALID_WINDOW_HANDLE).
- Windows Event Log: `data_downloader.exe` crash em `Qt6Core.dll 0xc0000409` (`__fastfail`).
- O **CLI** (`data_downloader-cli.exe download ...`) FUNCIONA (testado: 791k trades, `dll.finalized code=0`). Só a UI crasha.

### Root causes (2 RCAs do squad, ambos válidos e complementares)
- **RCA-A (Nelo @profitdll-specialist, ALTA) — Q08-E: duplo init da ProfitDLL no mesmo processo.** "Testar Conexão" (`SettingsScreen._TestConnectionWorker.run` fazia `with ProfitDLL() as dll: ...` → `__exit__` chama `finalize()`) seguido de "Baixar" (`public_api.download._build_real_dll` criava OUTRA `ProfitDLL()` e re-inicializava) → 2º init crasha: a DLL Classic NÃO é re-inicializável; `Finalize` não zera estado interno; a `ConnectorThread` da 1ª sessão morreu mas o 2º init posta msgs pra ela (1444/1400) e `CreateDataLoader` pega ponteiro de interface morto. **Não é regressão da v1.1.0** — existe desde v1.0.5 (quando "Test Connection" foi adicionado). Docs: `docs/dll/QUIRKS.md` Q08-E + `docs/adr/ADR-022-single-session-sequential-policy.md`.
- **RCA-B (Felix/Dex, MÉDIA) — frozen windowed `console=False` → stdio inválido.** O UI EXE é `console=False` (`build/data_downloader.spec.template` ~linha 374; o CLI EXE é `console=True` ~linha 397). Em modo windowed o processo não tem console: `sys.stdout`/`sys.stderr`/`sys.stdin` são `None` e os fds 0/1/2 OS-level ficam inválidos. A ProfitDLL Delphi, na `ConnectorThread` durante `InitializeAfterLogin`/`CreateDataLoader`, toca esses handles → access violation. CLI (`console=True`) tem fds válidos → funciona. Docs: `docs/dll/QUIRKS.md` Q-DRIFT-39.

### Fix aplicado (os DOIS — compatíveis)
**Fix 1 — DLL singleton process-global (cobre RCA-A):**
- **Novo** `src/data_downloader/dll/session.py`: `get_dll(*, market_only=True, **init_kwargs) -> ProfitDLL` mantém uma instância process-global (module-level `_DLL_INSTANCE` + `threading.Lock`). 1ª chamada: cria `ProfitDLL()`, `initialize_market_only(**kwargs)`, guarda, registra `atexit.register(shutdown_dll)`. Chamadas seguintes: devolve a MESMA instância (NÃO re-inicializa, NÃO finaliza). Init que falha NÃO é guardado. `shutdown_dll()` finaliza 1x (idempotente). `has_active_dll()` para o caller saber se reusar.
- `src/data_downloader/ui/screens/settings_screen.py` `_TestConnectionWorker.run`: `with ProfitDLL() as dll: dll.initialize_market_only(...)` → `dll = get_dll(market_only=True, key=..., user=..., password=..., minimal_handshake=True)`. Mantém `wait_market_connected(timeout=30, retry_attempts=1)` e lê `dll_version`. SEM `finalize()` / `__exit__`.
- `src/data_downloader/public_api/download.py` `_build_real_dll`: `ProfitDLL()` + init → `get_dll(...)`. Se `has_active_dll()` (instância já criada pelo Test Connection), reusa e PULA `wait_market_connected` (1ª sessão já drenou o estado `MARKET_CONNECTED` do queue) → emite `INF_DLL_READY` e retorna. Se fresh: init + wait normalmente. `_run_download_worker` `finally` NÃO finaliza mais a DLL (singleton); `own_dll` removido.
- `src/data_downloader/ui/main_window.py` `closeEvent`: chama `shutdown_dll()` (best-effort) — finalize 1x no encerramento da UI. CLI: `atexit` cobre o teardown do processo CLI.
- CLI `download` herda o singleton (já passava por `_build_real_dll` → `get_dll`; 1 init/processo lá, que já era o caso).

**Fix 2 — `_ensure_valid_stdio()` no startup (cobre RCA-B):**
- `src/data_downloader/ui/app.py`: nova função `_ensure_valid_stdio()` chamada como **PRIMEIRA** linha de `main()` (antes de `bootstrap_env`, `setup_logging`, `QApplication`, qualquer uso da DLL). Para cada fd 0/1/2: `os.fstat(fd)` → se `OSError`, `os.open(os.devnull, ...)` + `os.dup2`. Também garante `sys.stdout/stderr/stdin` não-`None`. No-op se já válidos.

### Tests adicionados
- `tests/unit/test_dll_session.py` (7 testes): `get_dll` 2x → mesma instância (1 init só); init kwargs repassados; `has_active_dll` tracking; `shutdown_dll` finaliza (idempotente); novo `get_dll` após shutdown → nova instância; init falho não guardado; rejeita `market_only=False`.
- `tests/unit/test_app_stdio.py` (3 testes): `_ensure_valid_stdio` substitui `sys.stdout=None` por não-None; no-op quando fds válidos (não troca streams); idempotente.

### Validação
