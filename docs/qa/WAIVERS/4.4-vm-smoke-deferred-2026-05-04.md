# WAIVER — `4.4` — `2026-05-04` (VM smoke deferred)

## Justificativa

A AC6 da Story 4.4 inclui smoke gated em VM Windows limpa
(`tests/smoke/test_release_install.md`) exigindo:

- VM Windows 10/11 limpa, sem Python instalado, sem ProfitDLL prévia.
- Download da release V1.0 da página GitHub (release ainda não publicada
  — circular dependency até pipeline ser executado pela primeira vez).
- Verify SHA256 contra `build-manifest-v1.0.0.json`.
- Click-through manual no SmartScreen warning (Caminho B / ADR-016).
- Setup `.env` com credenciais Nelogica reais + `PROFITDLL_KEY`.
- Smoke download de 1 dia útil real (ex.: WDOJ26 2026-04-15).
- Simulação de update: publicar `v1.0.1-rc1` com bump trivial → verificar
  que UpdaterStub detecta + propõe download manual + usuário aplica.
- Evidência sanitizada salva em `docs/qa/SMOKE_EVIDENCE/4.4-{ISO}.md`.

Conforme `docs/qa/SMOKE_PROTOCOL.md` §2 e
`docs/decisions/COUNCIL-09-mvp-gate-without-real-smoke.md`, este teste
**só pode ser executado pelo humano**:

1. Quinn (agente) não possui máquina Windows VM disponível +
   ProfitDLL.dll instalada + licença Nelogica.
2. A release V1.0.0 ainda não foi publicada no GitHub (humano executa
   `scripts/github_release.py` após este WAIVER ser arquivado e Story 4.4
   ser aprovada).
3. SmartScreen click-through não é automatizável sem ferramenta de
   automação UI Windows que extrapola escopo agente.
4. Setup `.env` com `PROFITDLL_KEY` real (secret) é fora do squad-de-1
   agente — humano controla credenciais.

A escalação a humano é a **saída legítima prevista no modo autônomo do squad**
para tarefas estruturalmente impossíveis ao agente (política COUNCIL-09).
Em vez de bloquear Story 4.4 indefinidamente, este WAIVER formaliza a
exceção e cria story-debt rastreável (`docs/stories/4.4-followup.story.md`).

Este WAIVER **não diminui o rigor do gate de release V1.0**: smoke real
em VM limpa continua **bloqueante** para publicação real do `.exe` da
V1.0 pelo @devops humano (campo `bloqueia_release` abaixo). Apenas
desbloqueia o squad para fechar Epic 4 + entregar artefatos de build
(scripts, spec, INSTALL.md, updater stub, council, tests) prontos para
o humano executar a publicação.

## Risco aceito

**O que pode dar errado em produção / dataset por aceitar este WAIVER:**

1. **Pipeline build_release.py + PyInstaller real** não foi exercitado E2E.
   Mocks dry-run cobrem orquestração + manifest schema, mas:
   - PyInstaller pode falhar com hidden import faltante (ex.: nova dep
     em Story 4.x não declarada em `hiddenimports` do spec).
   - Spec template pode emitir `dist/data_downloader/` com tamanho fora
     do range esperado (50-200 MB) — validate_output reporta mas humano
     precisa investigar.
   - Output pode faltar Qt platform plugin (`platforms/qwindows.dll`) —
     spec já tem defesa (`collect_all('PySide6')` + explicit append),
     mas nunca foi smoke-validated em build real desde amendment ADR-003.

2. **github_release.py + gh CLI real** não foi exercitado.
   Mock garante schema do manifest + body composition, mas:
   - `gh` CLI authentication pode estar misconfigured.
   - GitHub API rate limit pode bloquear release em janelas de muito
     uso (sem retry implementado no script — falha com mensagem clara
     mas humano precisa re-rodar).

3. **UpdaterStub real-world GitHub API** não foi exercitado contra repo
   real — apenas payload sintético. Possíveis surpresas:
   - GitHub rate-limit (60 req/h sem auth) — mitigação: stub usa headers
     polidos + check é manual user-triggered (1 req por click).
   - Mudança de schema GitHub Releases API (rara; hoje versão `2022-11-28`).

4. **SmartScreen UX em VM limpa** — sem reputation building, primeira
   execução pode mostrar dialog mais agressivo do que documentado em
   INSTALL.md §5. Mitigação: humano valida + atualiza screenshots em
   `docs/release/INSTALL.md` se UX divergir.

**Mitigação:** O smoke real V1.0 em VM limpa é **bloqueante de
publicação V1.0.0** real. Nenhum binário V1.0 será publicado em
GitHub Releases por @devops humano sem este smoke PASS arquivado em
`docs/qa/SMOKE_EVIDENCE/`. Risco residual = zero para usuário final;
risco operacional = re-runs do humano se algo falhar
(probabilidade média porque é a primeira vez que o pipeline roda real).
