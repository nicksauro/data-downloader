# WAIVER — `1.8` — `2026-05-04`

## Justificativa

A Story 1.8 entrega baselines de performance em modo **mock-based v1.1.0-mock**
(pipeline E2E real `Orchestrator + ParquetWriter + Catalog` + mock DLL inline
no padrão `_FakeProfitDLL` de `tests/integration/test_orchestrator.py`). O
plano original previa baselines `v1.0.0-real` derivados de smoke real contra
ProfitDLL ao vivo (Story 1.7b smoke), mas Story 1.7b está em WAIVER
(`docs/qa/WAIVERS/1.7b-real-smoke-deferred-2026-05-04.md`) aguardando humano
disponível para rodar o smoke (estruturalmente impossível ao agente per
`docs/qa/SMOKE_PROTOCOL.md` §2).

A escalação a humano é a **saída legítima prevista no modo autônomo do squad**
para tarefas estruturalmente impossíveis ao agente. Em vez de bloquear a
Story 1.8 indefinidamente aguardando o humano (o que paralisaria o close de
Epic 1 e o início de Epic 2), este WAIVER formaliza a exceção e cria
story-debt rastreável (`docs/stories/1.8-followup.story.md`) que fecha quando
o humano rodar o smoke real e Pyro re-baseline com `v1.0.0-real` (substituindo
`v1.1.0-mock` em `BASELINES.md`).

Este WAIVER **não diminui o rigor do gate de Epic 1**: re-baseline real
continua sendo bloqueante para release V1 (campo `bloqueia_release` abaixo).
Apenas desbloqueia o squad para prosseguir com Epic 2 (incluindo Story 2.2
perf-write-optimization que vectoriza o gargalo identificado em Story 1.8) em
paralelo enquanto o humano agenda o smoke real.

**Política idêntica à da Story 1.7b** (COUNCIL-09 ratificada): mocks são
sufficient evidence para gate atual; smoke real é gate de release V1, não
gate individual de story.

## Risco aceito

**O que pode dar errado em produção / dataset por aceitar este WAIVER:**

1. **Latência DLL real + reconnect 99% (Q11-99) NÃO MEDIDOS.** Mock DLL =
   0 latência rede. Smoke real provavelmente revelará:
   - Tempo total `bench_chunking` 1 mês maior que extrapolação linear
     (~40 min mock vs ? min real — depende da rede, latência API Nelogica,
     frequência de reconnect 99%).
   - p99 `bench_callback_to_disk` provavelmente similar (writer é gargalo
     dominante, mas reconnect pode adicionar tail).
   - RSS / CPU steady state — proxies parciais existem (mock pipeline RSS
     observado < 500MB), mas não medidos formalmente.

2. **Speedup `bench_multi_symbol` N=4 com IO real pode ser melhor OU pior**
   que 2.88x mock (mock todos workers escrevem em mesmo SSD; real adiciona
   contenção de rede que pode mascarar contenção de IO local).

3. **Hot path `bench_callback_to_disk` chunk-mode p99 1_510ms é mock-only**
   — números reais provavelmente similares (writer = 85% do custo per
   COUNCIL-02), mas há margem de erro de ~20% sem real.

4. **Flame graph (AC5 deferred)** — pyspy/cprofile úteis especialmente em
   smoke real onde DLL latency + reconnect podem aparecer no top frames.
   Mock pipeline já bem caracterizado (gargalo = ParquetWriter validate +
   dedup + sort + sha256), mas flame graph real validaria em produção.

**Mitigação:**
- Re-baseline real é **bloqueante de release V1**. Nenhum binário V1
  será publicado por @devops sem `v1.0.0-real` registrado em
  `BASELINES.md`.
- Story 2.2 (perf-write-optimization) deve re-baseline contra `v1.0.0-real`
  ANTES de merge final (per COUNCIL-10 §6 — Aria endorsement).
- Risco residual = zero para release; risco operacional = um eventual
  descoberta-tarde de gap real maior que gap mock antes de V1 (mas com Story
  1.4.5 + 1.8 + 2.2 cobrindo escopo, probabilidade baixa).
